## 1. 목표

**산출물은 "리팩토링 계획서"다 — 리팩토링 실행이 아니다.** 마지막 리팩토링(4차 리뷰,

이슈들을 분리 등록한다(실행은 본 타스크 범위 밖).

## 2. 선행 완료 사항

- 거버넌스 2원칙 확정(SOLID·복잡도) + 시각 검증 거버넌스 명문화
  (`mydocs/manual/visual_verification_governance.md`) — 계획서의 회귀 게이트 전략이 참조.
- #1582 방향 합의: 1단계 = `SourceProvenance` + `LayoutCompatibilityProfile` 소유권 분리
  (shim 유지, behavior 불변), baseline/feature freeze 선행.
- 선행 리팩토링 3회 이력(`mydocs/feedback/r-code-review-1st~5th.md`) — 방법론 전례.

## 3. 수행 단계 (3단계)

### 1단계 — 현황 재진단 (정량)

거버넌스 2원칙 축으로 마지막 리팩토링 이후 상태를 실측:

- **복잡도**: 파일/함수 크기 핫스팟 스캔(src/ 전수) — 4차 리뷰 시점 지표와 대조
  (예: `layout_column_item` 827줄이 지금은?; typeset.rs 등 신규 비대 지점).
- **SOLID**: #1582 감사가 지적한 9개 도메인 축 혼재 지점을 현 코드에서 확인
  (`Document` 필드 축 분류, `DocumentCore` 표면, `lib.rs` public surface 등).
- 산출: `mydocs/tech/task_m100_1883_diagnosis.md` (수치 표 + 4차 리뷰 대비 변화).

### 2단계 — 리팩토링 계획서 초안

이슈 #1883 의 7개 항목을 담은 계획서:
현황 재진단 요약 / 범위·단계(위험도 순, #1582 1단계 포함) / baseline freeze 항목 /
feature freeze 시점·기간 / 회귀 게이트 전략(거버넌스 문서 참조) / 성공 기준(SOLID 점수·
복잡도 지표 정량 목표 + 6차 리뷰 계획) / 0.8·v1.0 일정 결합.

- 산출: `mydocs/plans/refactoring_plan_2026.md` (리팩토링 마스터 플랜 — task 절차 문서와 구분).

### 3단계 — 컬래버레이터 리뷰 게시 + 마무리

- 계획서 초안을 devel 반영 후 이슈 #1883 에 요약 게시, @jangster77 @postmelee 리뷰 요청.
- 최종 보고서(`mydocs/report/task_m100_1883_report.md`) + 오늘할일 갱신.
- 피드백 수렴·계획 확정·실행 이슈 분리는 리뷰 회신 후 후속(본 타스크는 "게시"까지).
