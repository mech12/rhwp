# Task M100 #1547 수행계획서 — PR 리뷰 문서 전용 커밋 heavy CI fast-pass gate

- 이슈: #1547 "외부 contributor PR 리뷰 문서 전용 커밋의 heavy CI fast-pass gate 도입 검토"
- 마일스톤: M100 (v1.0.0)
- 브랜치: `local/task1547`
- 작성일: 2026-06-26

## 1. 배경

외부 contributor PR 을 repository collaborator 가 검토하는 경우, 현재 운영 규칙은 review 문서와
오늘할일 갱신을 PR head 에 포함할 수 있도록 한다. 이 방식은 collaborator 가 merge 후 별도 문서 정리 PR 을
만들지 않아도 되게 하는 예외 경로다.

문제는 코드 변경이 이미 GitHub Actions 를 통과한 뒤 `mydocs/pr/**` 리뷰 문서만 추가해도 PR head 가
갱신되어 heavy CI 가 다시 실행된다는 점이다. PR 처리 자체는 끝났는데 CI 대기만 추가되어 merge 시간이 늘어난다.

단순한 `paths-ignore` 또는 `[skip ci]` 는 required check 를 pending 으로 남길 수 있어 안전하지 않다. 따라서
workflow 는 실행하되, 조건이 엄격히 맞는 경우에만 heavy job 을 job-level 로 skip 하고 최종 gate 를 통과시키는
방식을 검토한다.

## 2. 현황

- `CLAUDE.md` 는 외부 PR 검토 기록을 `mydocs/pr/` 아래에 보관하도록 정한다.
- `mydocs/manual/pr_review_workflow.md` 9장은 collaborator-mediated 외부 PR 경로에서 review 문서와
  오늘할일을 PR head 에 포함할 수 있음을 명시한다.
- `.github/workflows/ci.yml` 과 `.github/workflows/codeql.yml` 은 `mydocs/**` 를 `paths-ignore` 에 포함한다.
- 하지만 PR 전체 diff 에 코드 변경이 남아 있으면, 리뷰 문서만 후속 커밋해도 PR workflow 가 다시 실행될 수 있다.
- `.github/workflows/render-diff.yml` 도 renderer 관련 PR 에서 같은 문제가 발생할 수 있다.

## 3. 목표

1. 리뷰 문서 전용 후속 커밋에서는 기존 heavy CI 를 다시 기다리지 않아도 되게 한다.
2. 직전 코드 검증 대상 SHA 가 이미 green 인 경우에만 fast-pass 한다.
3. 실패한 코드 SHA 위에 문서 커밋만 추가해도 통과하지 않게 한다.
4. 외부 fork PR 에서 untrusted code 를 privileged token 으로 실행하지 않는다.
5. 조건 판정이 불확실하면 fast-pass 하지 않고 기존 heavy CI 를 실행한다.

## 4. 비목표

- 코드 변경, 테스트 변경, workflow 변경, 샘플/baseline 변경이 포함된 PR 업데이트의 CI 생략.
- Commit Status API 로 기존 check 를 위조하듯 success status 를 생성하는 방식.
- GitHub branch protection 설정을 이 브랜치에서 직접 변경하는 것.
- PR review 문서 작성 규칙 자체를 폐기하는 것.

## 5. 조사·설계 방향

1. **허용 경로 확정**
   - 최소 허용 경로는 `mydocs/pr/**` 로 둔다.
   - 현재 collaborator-mediated 경로가 `mydocs/orders/{yyyymmdd}.md` 도 함께 포함할 수 있으므로,
     오늘할일 문서까지 fast-pass 허용 범위에 넣을지 별도 판단한다.

2. **trailing docs-only 구간 판정**
   - PR head 에서 뒤쪽 커밋들을 거슬러 올라가며, 각 커밋의 변경 파일이 허용 경로에만 속하는지 확인한다.
   - 허용 경로 밖 변경이 처음 나타나는 부모/커밋을 직전 코드 검증 대상 후보로 삼는다.
   - 단일 마지막 커밋만 보는 방식은 피한다. 문서 전용 커밋이 여러 개 쌓인 경우도 고려한다.

3. **직전 SHA green 판정**
   - GitHub Checks/Statuses API 로 후보 SHA 의 required check 결론을 조회한다.
   - `success`, `skipped`, `neutral` 만 통과로 인정한다.
   - check run 이 없거나 SHA 매칭이 불명확하면 fast-pass 하지 않는다.

4. **workflow 구조**
   - `preflight` job 은 항상 실행한다.
   - `preflight` 가 `fast_pass=true` 를 출력하면 heavy job 은 job-level `if` 로 skip 한다.
   - 최종 `*-gate` job 은 `always()` 로 실행해 fast-pass 또는 heavy job 결과를 종합한다.
   - branch protection required check 가 개별 heavy job 을 요구하는 경우, skipped 결론이 통과로 인정되는지
     실제 저장소 설정과 GitHub UI 에서 확인해야 한다.

5. **보안 원칙**
   - `pull_request_target` 을 쓰더라도 외부 PR head 코드를 checkout 하거나 실행하지 않는다.
   - 가능한 한 기존 `pull_request` workflow 안에서 read-only token 으로 판정한다.
   - API 조회와 git diff 판정은 untrusted code 실행 없이 수행한다.

## 6. 수정 범위 후보

- `.github/workflows/ci.yml`
  - `Build & Test` heavy job 앞에 preflight/gate 추가 검토
- `.github/workflows/codeql.yml`
  - CodeQL matrix job 의 fast-pass 조건 적용 검토
- `.github/workflows/render-diff.yml`
  - renderer 영향 PR 의 문서 전용 후속 커밋에서 visual diff 재실행 생략 검토
- `mydocs/manual/pr_review_workflow.md`
  - collaborator-mediated 경로에 "최종 CI 이후 리뷰 문서 전용 커밋 대기 문제와 fast-pass 조건" 보강 여부 검토

## 7. 완료 조건

1. fast-pass 조건이 엄격하고 보수적으로 정의된다.
2. 리뷰 문서 전용 후속 커밋에서는 heavy job 이 skip 되거나 최종 gate 가 빠르게 통과할 수 있다.
3. 코드 변경이 포함된 PR 업데이트에서는 기존 CI 가 정상 실행된다.
4. 이전 검증 대상 SHA 가 실패/미확인 상태이면 fast-pass 하지 않는다.
5. workflow 변경이 외부 fork PR 보안 모델을 약화하지 않는다.
6. 문서에 collaborator-mediated 외부 PR 처리 규칙과의 관계가 명확히 남는다.

## 8. 위험과 대응

| 위험 | 대응 |
|------|------|
| GitHub check 가 head SHA 와 merge SHA 중 어디에 붙는지 오판 | 구현 전 실제 PR check metadata 를 확인하고, 불명확하면 fast-pass 비활성 |
| required check 설정과 gate job 구조 불일치 | heavy job 이름 보존 또는 skipped 결론 통과 여부 확인, 필요한 admin 설정은 별도 요청 |
| 문서 커밋 뒤 실패한 코드 SHA 가 통과 | 직전 후보 SHA 의 check 결론을 필수 확인하고 없으면 heavy CI 실행 |
| 허용 경로가 넓어져 실질 변경을 놓침 | 기본은 `mydocs/pr/**` 로 제한, `orders` 포함은 구현계획서에서 별도 승인 |
| fork PR 보안 약화 | privileged context 에서 PR head checkout/실행 금지 |

## 9. 검증 방법

- workflow YAML 변경 후 `git diff --check`.
- `gh` 또는 GitHub API 로 실제 PR check run/status 구조 샘플 확인.
- trailing docs-only 판정 로직은 synthetic commit range 또는 기존 PR commit range 로 로컬 dry-run.
- fast-pass 조건 false 인 경우 기존 heavy job 이 실행되는지 GitHub Actions 에서 확인.
- fast-pass 조건 true 인 경우 heavy job skip 과 최종 gate success 를 GitHub Actions 에서 확인.
