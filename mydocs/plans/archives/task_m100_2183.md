## 1. 목적

현재 `.github/workflows/ci.yml`의 required check인 `Build & Test`는 Rust default-feature와 native Skia
검증만 집계한다. `rhwp-studio`, Chrome/Firefox extension, VS Code extension, `@rhwp/editor`의
package-lock 또는 source가 바뀌어도 해당 package를 설치·빌드·테스트하지 않아 green으로 끝날 수 있다.

#2183의 목적은 frontend 영향 경로를 식별해 #2124에서 Local PASS로 고정한 package gate를 GitHub Actions에
편입하고, 그 결과를 기존 `Build & Test` required check가 빠짐없이 집계하게 만드는 것이다.

## 2. 현행 조사 결과

### 2.1 CI 구조

- `preflight`는 review-only trailing commit을 판정하고 `fast_pass`를 출력한다.
- `build-default-feature-tests`와 `native-skia-tests`는 `fast_pass != true`일 때 실행된다.
- `Build & Test`는 두 Rust worker 결과를 집계하는 branch protection 표면이다.
- `WASM Build`는 tag 또는 `workflow_dispatch`에서만 실행되므로 PR package build에 `pkg/`를 제공하지 않는다.
- `render-diff.yml`은 별도 workflow에서 `wasm-pack build --target web --dev`를 수행하지만 workflow 간
  workspace/artifact를 공유하지 않는다.

### 2.2 package별 실행 전제

| 표면 | lockfile | 현재 명령 | 추가 전제 |
|------|----------|-----------|-----------|
| Studio | 있음 | `npm ci`, `npm run test`, `npm run build` | `@wasm` alias가 clean checkout에 없는 `pkg/`를 요구 |
| Chrome | 있음 | `npm ci`, `npm run build` | Studio Vite와 `pkg/`, `web/fonts`, shared SW symlink를 소비 |
| Firefox | 있음 | `npm ci`, `npm run build` | Studio Vite와 `pkg/`, `web/fonts`, shared SW symlink를 소비 |
| VS Code | 있음 | `npm ci`, `npm run compile` | CopyPlugin이 `pkg/rhwp_bg.wasm`을 요구 |
| `npm/editor` | 없음 | baseline root contract test | 현재 dependency와 package-local test script 없음 |
| Safari | npm package 없음 | manifest contract | Linux에서 native extension build를 수행하지 않음 |

`npm/editor`의 후속 PR #2187은 `scripts.test`와 `tests/transport.test.mjs`를 추가한다. #2183은 현재
baseline contract를 항상 실행하면서 package-local test가 선언된 branch에서는 해당 test도 자동으로 실행해야
#2187의 실제 검증 gate가 된다.

### 2.3 clean checkout WASM 제약

`pkg/`는 generated·ignored output이며 source of truth가 아니다. tracked `web/rhwp.*` 또는
`rhwp-studio/public/rhwp.*`를 `pkg/` 대용으로 복사하면 stale declaration/binary를 green 근거로 사용할 수
있어 #2124 authority 원칙과 충돌한다.

따라서 package build 전에 fresh binding이 필요하다. 다만 #2183은 release artifact를 검증하는 이슈가
아니므로 repository release WASM gate를 새로 만들지 않고, frontend worker 내부에서 기존 Render Diff와 같은
`wasm-pack build --target web --dev`를 사용해 CI 검증용 `pkg/`만 생성한다.

## 3. 목표

1. frontend 영향 경로가 포함된 PR/push에서만 frontend worker를 실행한다.
2. clean checkout에서 fresh WASM binding을 만든 뒤 모든 package 명령을 실행한다.
3. package install은 각 lockfile의 `npm ci`로 재현한다.
4. Chrome/Firefox build 뒤 extension dist contract를 실행한다.
5. current `@rhwp/editor` contract와 #2187 이후 package-local tests를 모두 수용한다.
6. frontend worker 실패가 기존 required check `CI / Build & Test`를 실패시키게 한다.
7. review-only fast-pass와 비-frontend PR의 기존 동작을 보존한다.

## 4. 포함 범위

### 4.1 workflow

- `.github/workflows/ci.yml`의 `preflight`에 frontend 영향 판정 output 추가
- frontend package worker job 추가
- 기존 `Build & Test` aggregate에 frontend worker 결과 연결
- Node/npm download cache와 frontend 전용 WASM cargo cache 정책 정의

### 4.2 frontend 영향 경로

기본 trigger 후보는 다음과 같다.

- `rhwp-studio/**`
- `rhwp-chrome/**`
- `rhwp-firefox/**`
- `rhwp-safari/**`
- `rhwp-vscode/**`
- `rhwp-shared/**`
- `npm/editor/**`
- `scripts/frontend-*.mjs`
- `web/**`
- `Cargo.lock`
- `src/wasm_api.rs`
- `.github/workflows/ci.yml`

이 목록은 #2183 이슈의 package 경로에 #2124 contract/smoke 의존 경로를 보강한 것이다. workflow 자체를
포함해 #2183 PR에서 새 worker가 실제 실행되게 한다.

현재 workflow-level `paths-ignore`의 `assets/**`는 이번 이슈에서 제거하지 않는다. 아직 존재하지 않는
`assets/fonts/**` canonical 경로는 #2125가 실제 경로와 copy 계약을 추가할 때 workflow trigger에도 함께
편입한다. #2183에서 모든 기존 screenshot/logo asset까지 full CI 대상으로 넓히지 않는다.

### 4.3 package gate

- fresh `pkg/` 생성 후 `scripts/frontend-wasm-bindings.test.mjs`
- `scripts/frontend-editor-embed.test.mjs`
- `npm --prefix npm/editor test --if-present`
- Studio `npm ci`, unit test, production build
- Chrome/Firefox 각각 `npm ci`, production build
- Chrome/Firefox shared SW tests
- extension dist contract 3종
- VS Code `npm ci`, production compile

test 개수는 branch에 따라 증가할 수 있으므로 CI 조건에 숫자를 하드코딩하지 않고 command exit status를
판정 기준으로 사용한다.

## 6. 확정 설계 방향

### 6.1 frontend 영향 판정

기존 review-only fast-pass script와 책임을 섞지 않고 `preflight` 안에 별도 read-only
`actions/github-script` step을 둔다.

- PR: Pull Request Files API의 전체 변경 파일로 판정
- push: `before...after` compare의 변경 파일로 판정
- tag/manual 또는 파일 목록이 없거나 잘린 경우: frontend 실행
- API 오류/판정 불명확: frontend 실행

즉 false negative보다 추가 실행을 선택하는 fail-open-to-validation 정책을 적용한다. step 자체에는
`continue-on-error`를 둘 수 있지만 output 기본값은 `true`로 두어 판정 실패가 gate 생략으로 이어지지 않게
한다.

### 6.2 frontend worker

별도 Ubuntu worker를 Rust workers와 병렬 실행한다.

- `actions/checkout@v5`
- Rust `1.93.1`, `wasm32-unknown-unknown`
- 기존 workflow 선례와 같은 wasm-pack 설치 방식
- `wasm-pack build --target web --dev`
- Node.js 22와 package-local `npm ci`
- npm cache는 네 lockfile을 입력으로 사용하고 `node_modules` 자체는 cache하지 않음
- frontend WASM cache는 Rust worker와 다른 key namespace를 사용
- PR은 restore-only, trusted `devel`/`main` push만 exact miss cache save

Chrome/Firefox build는 Studio Vite를 호출하므로 Studio install을 먼저 수행한다. generated `pkg/`와 각
`dist/`는 runner 임시 산출물이며 artifact 업로드나 commit을 하지 않는다.

### 6.3 `@rhwp/editor` 과도기

현재 branch에서는 root baseline contract를 필수 실행한다. `npm/editor/package.json`에 `scripts.test`가
존재하면 `npm test --if-present`가 #2187의 package-local transport tests를 추가로 실행한다.

`dependencies` zero 계약 검사는 기존 baseline test로 유지한다. #2183에서 package.json에 임시 test script나
lockfile을 추가해 #2187과 충돌시키지 않는다.

### 6.4 required check 보존

새 worker job 이름은 별도 관측용으로 노출하되 branch protection 표면은 기존 `Build & Test`로 유지한다.

- `Build & Test.needs`에 frontend worker 추가
- `frontend_required=true`이면 worker `success`만 허용
- `frontend_required=false`이면 정상 `skipped`를 허용
- review-only `fast_pass=true`이면 기존 candidate `Build & Test` 재사용 판정을 유지
- preflight 또는 필요한 worker가 실패/cancel되면 aggregate 실패

이 방식은 새 required check를 admin UI에 추가하지 않아도 기존 gate가 frontend 실패를 차단한다.

## 9. 완료 조건

1. frontend 영향 PR의 clean checkout에서 package gate가 실제 실행된다.
2. `pkg/` 부재를 stale tracked 파일 복사로 우회하지 않고 fresh dev binding으로 해결한다.
3. Studio test/build, Chrome/Firefox build, VS Code compile, editor/shared/dist contract가 통과한다.
4. #2187이 rebase되면 package-local `npm/editor` tests가 별도 workflow 수정 없이 실행된다.
5. frontend worker 실패 시 `CI / Build & Test`가 실패한다.
6. 비-frontend PR과 review-only fast-pass의 기존 의미가 유지된다.
7. `Build & Test`, Rust profile, release WASM, security/public contract 표면이 바뀌지 않는다.
8. added runner time과 cache 동작이 보고서에 기록된다.
