- 브랜치: `local/task1880`

## 1. 배경

이슈 본문은 PR #1886 이전 상태다. #1886(HWPX-origin 마커, `is_hwpx_variant`)으로
`is_hwpx_source` 플래그는 HWPX/convert-HWP 양 경로에서 이미 통일되었으나, 코멘트 3
기록대로 PI_MOVED 4건(3075729, 2780073, 2959953, 3171755)이 잔존한다. 코멘트 3의
진입점(2780073 pi=4 `format_table` `host_spacing.before` HWPX=6.7px/conv=0.0px)을
따라 조사를 재개했다.

## 2. 원인 (소스 조사 완료 — 실측 재현 포함)

**재현 확정** (현 devel bf5228df, 릴리즈 빌드):

- 3075729: HWPX 렌더 p13(oracle 정답) / convert-HWP 렌더 p12 — 회귀 재현.
- 2780073 `RHWP_TABLE_DRIFT=1` 트레이스: 분기는 pi=4 가 아니라 **pi=2부터**.
  양 경로 진입 flow 가 동일(cur_h=81.3)한데도 `host_before` HWPX=6.7 / conv=0.0.
  이후 모든 visible-host 자리차지 표(pi=2/4/6)에서 host_sp 가 정확히 6.7px씩 차이
  (29.1/27.2/27.2 vs 22.4/20.5/20.5) → pi=4 defer 가드 플립(잔여 27.0 ≥ row0 21.1
  vs 20.3 < 21.1) → 쪽 경계 이동.

**구조 비대칭의 정체** (ir-diff 사각지대):

1. `src/renderer/typeset.rs:10617` — `format_table` 이 **원시 비트필드**를 읽는다:
   `let table_text_wrap = (table.attr >> 21) & 0x07;`
   `typeset.rs:10711` — `!is_tac && table_text_wrap == 1`(자리차지)이면
   `host_spacing.before` 에서 `sb`(spacing_before)를 제외한다.
2. **HWP5 파서**(`src/parser/control.rs:153`): `table.attr = table.common.attr`
   — 파일의 원시 attr 전체 (text_wrap bits 21-23 포함).
3. **HWPX 파서**(`src/parser/hwpx/section.rs:1831`): `table.attr` 에 bit0(TAC
   호환)만 미러링, 나머지 0 — 의미 배치는 `common.text_wrap` 열거형에만 보관
   (의도된 설계, 주석 명시).

→ 같은 자리차지 표가 HWPX 파스에선 branch 미진입(`sb` 포함, before=6.7px),
convert-HWP 재파스에선 branch 진입(`sb` 제외, before=0.0px). `ir-diff` 는 의미
필드(wrap 열거형)만 비교하므로 원시 `table.attr` 차이가 0건으로 보였다.

**의미 동치 확인**: `src/parser/control/shape.rs:394` — HWP5 의
`(attr>>21)&7 == 1` ⇔ `common.text_wrap == TextWrap::TopAndBottom` (전단사).
원시 비트의 `>> 21` 소비처는 렌더러 전체에서 `typeset.rs:10617` 단 한 곳.

## 3. 목표

convert-HWP(is_hwpx_variant) 렌더의 `host_spacing.before` 산출을 HWPX 렌더와
일치시켜(인코딩 안정화) PI_MOVED 잔존 4건 클래스를 해소한다. native HWP5/HWP3
와 순수 HWPX 의 렌더 동작은 **비트 단위로 불변**이어야 한다.

## 4. 수정 방향 (#1886 origin 전달 방향의 연장)

`typeset.rs:10617/10711` 의 원시 비트 판정을 의미 필드 + 소스 게이트로 교체:

```rust
// 종전: (table.attr >> 21) & 0x07 == 1   ← HWPX 파스는 attr 미채움(항상 false)
// 신규: !self.is_hwpx_source.get()
//        && matches!(table.common.text_wrap, TextWrap::TopAndBottom)
```

경로별 효과:
- **native HWP5/HWP3** (`is_hwpx_source=false`): 원시 비트 ⇔ 열거형 동치이므로 불변.
- **순수 HWPX** (`is_hwpx_source=true`): 종전에도 attr=0 으로 branch 미진입 → 불변.
- **convert-HWP** (`is_hwpx_variant` → `is_hwpx_source=true`): branch 미진입으로
  전환 — HWPX 렌더와 동일 경로 (이번 수정의 유일한 행동 변화).

이슈 본문의 "빈-앵커 억제 조정"과 별개 층위임에 유의: #1836 트레이드오프
(phantom/real)는 건드리지 않는다. oracle 정합 방향과도 일치 — 3075729 는 HWPX
렌더(p13)가 정답이고 convert 렌더를 그쪽으로 당긴다 (코멘트 2 수정 방향 후보 1
"convert-HWP 렌더를 HWPX 렌더와 일치(인코딩 안정화)" 에 해당, 2776741 불변 기대).

**관찰 사항 (범위 외 기록)**: `table.attr & 0x01`(TAC bit) 도 HWPX 미러
(`treat_as_char && flow_with_text`) vs HWP5 원시(bit0)가 미묘히 다를 수 있으나,
본 4건 트레이스는 전부 tac=false 로 무관. 별도 이슈 후보로 보고서에 기록만 한다.
