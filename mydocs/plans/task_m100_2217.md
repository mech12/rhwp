# 수행계획서 - Task M100 #2217: 로컬 글꼴 다국어 이름 매칭 및 CanvasKit Typeface 등록

- 이슈: [#2217](https://github.com/edwardkim/rhwp/issues/2217)
- 브랜치: `codex/task2217-local-font-alias-20260711`
- 재현 파일: `samples/issue2217/20200830.hwp`
- 기준 환경: macOS Chrome의 Local Font Access API, `08서울한강체 M` 설치 상태

## 문제

HWP 문서는 `08서울한강체 M`을 지정하지만, macOS Chrome의 로컬 글꼴 감지는
`08SeoulHangang`처럼 영문 family만 반환할 수 있다. 현재 Studio는 `FontData.family`만 저장하고
정확 문자열로 비교하므로, 같은 글꼴의 한글 family/full name/PostScript name을 서로 다른 글꼴로
판정한다. CanvasKit은 번들 Noto Sans KR 하나만 등록하므로 감지한 로컬 글꼴을 실제로 그리지도 못한다.

## 목표

1. 로컬 글꼴의 다국어 이름을 안정된 하나의 레코드로 보존하고 HWP 이름에서 해석한다.
2. CSS/Canvas2D와 CanvasKit이 같은 해석 결과를 사용한다.
3. 로컬 글꼴 바이트는 사용자가 허용한 현재 세션의 렌더링에만 두고 영속 저장하지 않는다.
4. `20200830.hwp`에서 08서울한강체 계열의 선택과 표시를 재현 가능하게 검증한다.

## 단계

### Stage 1 - 로컬 메타데이터와 별칭 해석

- `FontData`의 family, full name, PostScript name, style을 보존하는 snapshot v2를 설계한다.
- 기존 snapshot v1을 안전하게 읽고, 한글/영문 이름 및 style 조합을 하나의 local record로 해석한다.
- 문서 글꼴 상태와 CSS font-family chain이 resolver의 canonical CSS family를 사용하게 한다.
- 재현 HWP와 별칭 단위 테스트를 추가한다.

### Stage 2 - CanvasKit 세션 글꼴 등록

- Chrome Local Font Access API의 `FontData.blob()`로 얻은 SFNT 바이트를 메모리 전용 registry에 둔다.
- 문서 글꼴명 resolver 결과에 따라 CanvasKit Typeface/FontMgr를 선택한다.
- Local Font Access API 미지원 환경은 현재의 웹폰트/대체 글꼴 경로를 유지한다.

### Stage 3 - 통합 검증과 시각 판정

- Studio 단위 테스트, WASM build, CanvasKit focused 검증을 수행한다.
- `20200830.hwp`의 브라우저 CanvasKit/Canvas2D 표시와 HWP 2020 PDF를 시각 대조한다.
- 메트릭 차이는 [#2206](https://github.com/edwardkim/rhwp/issues/2206) 결과와 구분해 기록한다.

## 제외

- 특정 폰트명만 하드코딩해 일반 이름 해석 문제를 우회하지 않는다.
- 동의 없이 전체 로컬 글꼴의 바이트를 읽거나 chrome.storage/localStorage에 저장하지 않는다.
- Firefox 등 Local Font Access API 미지원 브라우저에 설치 글꼴 바이트 접근을 강제하지 않는다.
