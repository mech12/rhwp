## 2. 문제 정의와 현재 근거

### 2.1 확정된 사용자 관찰

HWP 재현 문서의 첫 페이지에서 `1.1.1` 문단 끝에 `1`을 반복 입력하면, 기존 줄 경계를
지난 뒤 캐럿과 새 문자가 즉시 보이지 않거나 잠시 뒤 사라진 것처럼 보인다. 같은 위치에서
Enter를 입력하면 페이지가 다시 그려지면서 앞서 입력한 문자가 나타난다.

Stage 1과 최신 교차검증에서 HWP/HWPX 모두 같은 44번째 입력에서 모델은 5줄·174자까지
최신이지만 warm page tree는 129에서 끝나는 RED를 세 번씩 재현했다. 입력 문자열은 모델에
보존되며, 문제는 두 형식이 공통으로 사용하는 deferred layout-cache 경계에 있다.

재현 대상의 native 경로는 다음과 같다.

```text
section=0 / parent paragraph=0 / table control=2 / cell=2
cell paragraph=5 / initial char offset=130
```

### 2.2 코드에서 확인된 상태 전이

```text
작은 같은 셀 입력
→ InsertTextCommand
→ insertTextInCellDeferredPagination
→ 모델 text·LINE_SEG·후속 셀 문단 vpos 갱신
→ section dirty + page-tree cache 무효화, pagination은 보류
→ 현재 cursor page 한 장만 document-page-invalidated(text-edit)
→ rAF 단위 Canvas 갱신 + 조건부 static layer 재사용
→ 조건 충족 시 800ms 뒤 static layer 검증 렌더
```

현재 page-local 판정은 같은 셀·같은 before/after page, 8자 이하, 줄바꿈·탭 없음만
확인한다. 입력 전후 줄 수, 세로 범위, pagination fragment 또는 실제 영향 페이지 범위는
확인하지 않는다.

Rust deferred 입력은 새 모델과 새 `LINE_SEG`를 만든 뒤 section, page-tree와 layer JSON
cache만 무효화하고 즉시 pagination하지 않는다. full invalidation과 달리
`LayoutEngine::clear_layout_caches()`를 호출하지 않으므로 이전 4줄 기준
`cell_units_cache`가 남을 수 있다. 따라서 새 문단 줄 구조, warm layout cache와 이전
pagination fragment가 공존할 수 있다. Studio는 cursor rect가 가리키는 한 페이지만 다시 그리며,
115쪽 문서는 30쪽 자동 flush 제한을 넘으므로 저장·인쇄·구조 편집 전까지 deferred
pagination이 계속 pending일 수 있다.

800ms static-layer 검증은 두 기준 SHA의 재현에서 예약되지 않았고, full-layer Canvas
재렌더도 복구하지 못했다. 따라서 timer, static reuse와 Canvas 합성은 직접 원인 후보에서
제외한다.

Enter는 문단 분할, pending pagination flush, 즉시 pagination 및 full
`document-changed`를 함께 일으킬 수 있다. 따라서 “Enter 후 복구”만으로 pagination과
Canvas refresh 중 어느 쪽이 원인인지 판정하지 않는다.

### 2.3 사실과 가설의 경계

| 분류 | 내용 |
|------|------|
| 확정 | HWP/HWPX 모델·`LINE_SEG`는 최신이고 표시 데이터만 stale하다. |
| 확정 | 같은 최종 모델과 `end_cut=[37]`에서도 cold tree는 174/180까지 정확하지만 warm tree는 129에서 끝난다. |
| 확정 | pagination 없이 layout cache를 비우면 tree와 cursor가 full-flush 좌표로 복구되며 cut은 `[37]`로 유지된다. |
| 확정 | cache-only 통제 뒤 `cellBounds`는 945.9로 남고 full flush 뒤 971.5, cut은 `[38]`이 된다. |
| 확정 | 약 2초 지연은 새 offset을 찾지 못한 cursor exact-search가 115쪽을 순회한 비용이다. |
| 기각 | Canvas static reuse, 800ms verification, 입력 batch/sequential 및 cursor API 차이가 직접 원인이다. |
| 미확정 | scoped cache invalidation의 전 115쪽 연속성·중첩 표 전파 범위·매-key 비용 |

## 3. 목표

1. 줄 수가 4줄에서 5줄로 바뀌는 경계를 포함한 연속 입력을 브라우저에서 결정적으로
   재현하고 시간축 관측 계약을 고정한다.
2. 모델, `LINE_SEG`, pagination fragment/page tree, cursor 및 합성 Canvas 중 최초로
   stale 상태가 발생하는 경계를 계측으로 확정한다.
3. 확정된 단일 경계를 가장 작은 범위에서 보완해, Enter나 수동 refresh 없이 새 입력과
   캐럿이 즉시 표시되도록 한다.
4. 안정적인 줄 내부 입력에는 전체 pagination 0회를 유지하고, 실제 cell-flow advance가
   바뀌는 경계에서만 cursor 조회 전에 1회 flush해 bounds와 cut까지 확정한다.
5. HWP/HWPX 저장·재로드와 #2185 한 글자 기준선, #1918 fast path 성능 안전성을 유지한다.

## 4. 수행 원칙

1. 데이터 보존 판정은 모델 text와 원본 형식 저장·재로드 결과를 최우선 권위로 삼는다.
2. 표시 판정은 page layer tree와 브라우저가 합성한 페이지 영역을 함께 사용한다. main
   flow canvas 하나만 캡처해 static/overlay layer를 누락하지 않는다.
3. 반복 입력 뒤 모델을 바꾸지 않는 `flushDeferredPagination()`과 full refresh 결과를
   정답 대조로 사용한다. Enter는 사용자 재현 증적으로만 사용한다.
4. global layout-cache clear는 원인 격리 통제로만 사용하며, 115쪽 동등성과 매-key 비용을
   통과하기 전 production 해법으로 확정하지 않는다.
5. production 계약은 두 층으로 고정한다. 모든 deferred 셀 편집은 편집 셀·소유 표의
   cache만 제거하고, 실제 cell-flow advance가 바뀐 입력만 pre-cursor 1회 flush한다.
6. targeted cache generation 또는 전체 renderer/paginator 재설계가 필요하면 구현을 중단하고
   후속 이슈 범위를 다시 승인받는다.

## 5. 수행 범위

### Stage 1. 결정적 재현과 시간축 상태 계측 고정

1. 최신 WASM과 Studio를 준비하고 HWP를 우선 재현한 뒤 HWPX에 같은 절차를 적용한다.
2. 고정 문서 경로와 offset 130에 캐럿을 놓고, 고정 입력 횟수 대신 실제
   `lineCount`가 4에서 5로 바뀔 때까지 `1`을 한 글자씩 입력한 뒤 2~3자를 더 입력한다.
3. 입력 전과 각 경계에서 다음 체크포인트를 기록한다.
   - 입력 동기 처리 직후
   - 2 rAF 뒤
   - 100ms 뒤
   - 850ms 뒤
   - 1.6초 뒤
   - 명시적 deferred pagination flush와 full refresh 뒤
4. 각 체크포인트에서 다음 상태를 하나의 레코드로 수집한다.
   - 셀 모델 text와 누적 suffix
   - cursor position, char offset, page, rect, `cellOverflowed`
   - 대상 문단 `LINE_SEG`의 줄 수·`text_start`·vpos·높이
   - deferred pagination pending 여부와 page count
   - 대상 문단이 나타나는 page fragment/page layer tree의 TextRun·좌표
   - invalidated page와 render context, filtered layer 호출
   - caret DOM과 합성된 페이지 영역 screenshot/pixel 비교
5. 입력 중 `flushDeferredPagination()` 호출 횟수를 기록해 page-local 경로가 유지되는지
   확인한다.

### Stage 2. warm layout-cache 회귀 계약과 전체 영향 범위 확정

1. HWP/HWPX를 fresh reload하고 cold/prewarm/mid-warm을 분리한다.
   - batch와 sequential 44자·50자
   - raw tree, direct cursor와 path-near cursor를 각각 첫 관찰로 격리
   - 같은 모델·line starts·`PartialTable` cut을 단언
2. pagination 없는 global-clear 진단 통제와 explicit full flush oracle을 비교한다.
   - page 0 tree 최대 char, cursor page/x/y/height와 fallback 여부
   - `cellBounds`, `cellOverflowed`, 다음 문단 vpos
   - 115쪽 전체 target range와 `PartialTable` cut fingerprint
3. 전 페이지 target range가 0..model length를 누락·중복 없이 잇는지 확인한다. page 0만
   GREEN이면 cache-only 후보는 완료안으로 채택하지 않는다.
4. 편집 셀·소유 표만 제거하는 scoped invalidation의 중첩 표 ancestor 범위와, flush 0회인
   일반 입력의 unrelated-cell cache hit 보존 계약을 실행 가능한 RED로 고정한다.
5. scoped invalidation과 boundary flush의 역할을 분리해 검증한다.
   - 일반 입력: edited cell/table eviction, flush 0회
   - 상대 flow advance 변화: 같은 eviction 뒤 cursor 조회 전 full flush 정확히 1회
   - ancestor generation 설계가 필요하면 #2214에서 확대하지 않고 후속 이슈로 분리

### Stage 3. 회귀 핀과 검증 계약 설계

   - 한 글자 입력 후 줄 시작 `[0, 44, 84, 122]`
   - 다음 문단 `vpos = 17160`
   - 전체 115쪽과 저장·재로드 동일성
2. 별도 native #2214 회귀는 다음을 검증하도록 설계한다.
   - HWP/HWPX에서 4줄→5줄 전환까지 한 글자씩 deferred 입력
   - 매 입력 후 모델 text가 정확히 누적됨
   - 기존 네 줄 시작점 `[0, 44, 84, 122]`가 prefix로 보존됨
   - 반복 입력 후 줄 구조와 다음 문단 vpos가 명시적 flush 및 재로드 결과와 동일함
   - 새 줄이 생긴 뒤에는 다음 문단 `vpos=17160`을 고정하지 않음
   - 115쪽과 원본 형식 저장·재로드 결과 보존
   - cold/warm 최종 모델 동일성, warm RED와 선택 수정 후 exact tree/cursor GREEN

3. 신규 Studio E2E는 실제 페이지 합성 화면을 기준으로 다음을 검증한다.
   - 입력마다 모델 text와 cursor offset 증가

   - 선택 수정 뒤 visible tree/cursor/crop이 explicit flush oracle과 동등함
   - 안정 입력 flush 0회, 실제 cell-flow 경계 flush 정확히 1회
4. page-local 판정 또는 Canvas fast path를 수정하면 관련 TypeScript 단위 계약을

5. #1918의 정적 레이어 재사용과 빠른 연속 입력 성능이 퇴행하지 않는 대표 probe를
   함께 고정한다.

## 8. 완료 조건

- HWP/HWPX에서 줄 수가 4줄에서 5줄로 바뀌는 연속 입력이 결정적으로 재현되고 자동화된다.
- 모든 입력 문자가 1~2 rAF 안에 layer tree와 합성 화면에 나타나며 caret이 마지막 문자 뒤를
  가리킨다.
- 850ms와 1.6초 뒤에도 입력 문자, 줄 배치와 caret이 사라지거나 이전 상태로 되돌아가지 않는다.
- 모델 text, explicit flush, 원본 형식 저장·재로드 결과의 입력 문자열이 정확히 일치한다.
- 선택 수정 뒤 visible page tree, cursor와 합성 화면이 explicit flush oracle과 동등하다.
- 115쪽 전체 target range가 누락·중복 없이 이어지고 flow 경계 뒤 `cellBounds`/cut이
  1회 flush oracle과 일치한다.
- 기존 #2185 한 글자 `[0, 44, 84, 122]`, `vpos=17160`, 115쪽 기준선을 유지한다.
- 반복 입력 후에는 기존 네 줄 시작점을 prefix로 보존하고, 새 줄 및 다음 문단 vpos는 explicit
  flush와 재로드 결과에 일치한다.
- HWP/HWPX 모두 최종 115쪽을 유지한다.
- 안정 입력은 전체 pagination 0회, 실제 cell-flow 경계는 정확히 1회이며 50자 재현 누계도
  1회다.
- #1918 page-local fast path의 대표 입력 성능과 정적 레이어 표시가 퇴행하지 않는다.
