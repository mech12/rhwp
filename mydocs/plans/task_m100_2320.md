# 수행계획서 — #2320 문단 중간 vpos 되감김(쪽 경계) 미인식

- 이슈: [#2320](https://github.com/edwardkim/rhwp/issues/2320)
- 마일스톤: M100 (v1.0.0)
- 브랜치: `local/task2320`
- 유형: 버그 정정형 — **구현계획서 생략** (본 계획서에 단계 포함, #2292/#2293/#2318 선례)

## 1. 증상

`treatise sample.hwp` p2 오른쪽 단(단 1)에서 다음 페이지 넘김 계산 실패.
문단 29(5줄) 전체 + 문단 30 첫 줄이 단 1에 배치되어 **하단 106.8px 잘림 +
내용 화면 소실** (`LAYOUT_OVERFLOW: page=1, col=1, para=30, overflow=106.8px`).
네이티브 공통(skia/svg/studio). 한컴 2022 PDF: 문단 29 **첫 줄에서 끊고
p3 로 넘김**.

## 2. 근인 (착수 정찰 실증)

1. 문단 29 저장 LINE_SEG: line 0 vpos=67050(단 하단) → line 1 **vpos=4926
   으로 되감김** — 한컴의 "문단 중간 쪽 경계" 인코딩.
2. 되감김 감지 함수는 존재: `detect_column_breaks_in_paragraph`
   ([pagination.rs:762](../../src/renderer/pagination.rs#L762) — vpos 감소 =
   단 경계). 그러나 호출 게이트가
   **`st.col_count > 1 && st.current_column == 0`**
   ([engine.rs:1233](../../src/renderer/pagination/engine.rs#L1233)) —
   문단이 **단 0 에서 시작할 때만** 감지한다.
3. 문단 29 는 단 1(마지막 단)에서 시작 → 되감김 미감지 → 문단 통째 배치.
   마지막 단의 되감김은 "다음 페이지 단 0 으로 계속"을 의미하는데 이
   진행 경로가 없다.
4. 부수 관찰: 단 1 배치 계측(used=906.7px ≤ body 914.6px)은 적합 판정이지만
   실제 배치 y 는 1153.7(bottom 1046.9)로 넘침 — 계측과 배치 y 의 불일치.
   1차 정정(분할)으로 본 케이스는 해소 예상이며, 잔존 시 별도 관찰 기록.

## 3. 정정 방향

**마지막 단(비-0 단 포함)에서 시작하는 문단의 문단 내 되감김을 단/쪽 진행
분할점으로 처리한다.**

- 단 0 시작: 기존 `paginate_multicolumn_paragraph` 경로 (불변)
- 단 k>0 시작: 되감김 감지 시 해당 줄에서 분할 — 다음 단으로 진행하되,
  마지막 단이면 **쪽 넘김 + 다음 페이지 단 0** 에서 잔여 줄 계속
- 케이스 구조 가드: 다단(col_count>1) 구역 한정. 1단 구역의 문단 내
  되감김(쪽 경계 인코딩 가능성)은 이번 범위에서 제외하고 관찰만 기록
  (일반화 남용 금지 — 대비 샘플 확보 시 별도 이슈)
- 저장 지오메트리 신뢰 계보(#2112, #2299/PR #2314)와 방향 정합: 저장
  신호(되감김)를 재계산으로 상쇄하지 않고 경계로 승격
