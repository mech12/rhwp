- 성격: Studio(TS) + wasm(Rust) 성능 — 렌더 결과 무변경(순서/시점만 변경)

## 1. 문제와 가설

이슈 계측: 입력 1회에 `getCursorRectByPath` **626.9ms** (115쪽 거대 표,
`samples/issue1949_giant_cell_nested_tables_perf.hwp`).

코드 정찰 결과 구조적 원인 가설:
1. 텍스트 편집 → `page_tree_cache` 무효화 (정상 설계).
2. `get_cursor_rect_by_path_native`는 `find_pages_for_paragraph`로 **호스트 문단이 걸친
   모든 페이지**를 얻는데, 거대 표는 PartialTable로 수십~115쪽에 걸침.
3. 페이지 목록을 **앞에서부터** `build_page_tree_cached`로 재빌드하며 캐럿을 탐색 —
   무효화 직후라 매 페이지가 캐시 미스 → 거대 표 페이지 레이아웃을 연쇄 재계산.

즉 병목은 "경로 API가 무겁다"가 아니라 **탐색 시작점이 캐럿 위치와 무관**한 것.

## 2. 수행 단계 (관문형)

### 1단계 — 계측 재현 + 가설 검증 + 구현계획서
- Rust 단위 계측: 거대 샘플에서 `find_pages_for_paragraph` 반환 페이지 수, 캐럿 발견까지
  빌드된 페이지 수, 총 시간. 가설(다수 페이지 연쇄 빌드) 확정 → `task_m100_2021_impl.md`.

### 2단계 — 수정 A (Rust): 페이지 힌트 우선 탐색
- `getCursorRectByPath`에 **선택적 `hint_page` 파라미터** 추가(기존 시그니처 호환 유지 —
  wasm 신규 진입점 또는 Option 인자): 힌트 페이지(및 ±1)를 먼저 검사, 실패 시 종전 전체
  탐색 fallback. Studio `CursorState.updateRect()`가 현재 캐럿 페이지를 전달.
- 렌더 결과·좌표 값 완전 동일(탐색 순서만) — 동등성 회귀 테스트로 고정.

### 3단계 — 수정 B (Studio): 입력 직후 낙관적 caret + idle 정밀 재계산
- 입력 fast path에서 이전 rect + 삽입 글자폭으로 caret x 임시 전진, 정확한
  `updateRect()`는 rAF/idle로 이연 — input handler에서 동기 장시간 호출 제거.
- 이슈 수용 기준의 회귀 테스트: fast path가 input 내 동기 rect 재계산으로 퇴행하지
  않도록 node --test 가드(호출 시점 계측 wrapper).

### 4단계 — 검증 + 보고
- 계측 전/후 대비(입력 1회 long task), 전체 게이트, 최종 보고서.
