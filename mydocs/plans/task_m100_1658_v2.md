## 3. 근인 가설 (조사 결과)
`total_content_height = max(stored last_seg_end, calc_composed_paras_content_height,
calc_nested_controls_bottom_height)`. over-count 케이스는 computed 항이 `inner_height` 근접.
- `calc_nested_controls_bottom_height` = max_para(`para_top(stored vpos) + Σ nested_table_height`).
  중첩 표 높이/para_top 합산이 stored 실제 바닥보다 과대 가능.
- `max()` 라 한 항만 과대해도 total 과대 → 상단정렬. 그러나 단순 stored 권위화는 #44(under-count)
  재회귀(가드 fixture 가 이를 잠금).
→ **stored vpos 를 1차 권위로 하되, 중첩 컨트롤이 stored 바닥을 초과하는 경우만 보정**하는 회계 필요.

## 4. 단계 설계 (구현계획서에서 3~6단계로 구체화)
1. **측정 정합 진단**: over-count 3종에서 3개 항(stored/composed/nested_bottom) 각 값과 한컴 기대
   content-height 를 정량 비교 → 어느 항이 왜 과대인지 확정(디버그 계측, 소스 무변경).
2. **total_content_height 회계 재설계**: stored last_seg_end 권위 + 중첩 표가 stored 바닥 초과 시에만
   초과분 보정(naive max 제거). #44 가드/over 3종 동시 정합.
3. **(A) 적용·검증**: valign 게이트 over 3종 FIX + 가드 불변, 전 통제 게이트 무회귀.
4. **(B) 연계 검토**: 동일 회계 원리를 block-continuation(별표4) 선행 행 측정에 적용 가능한지 분석,
   가능하면 적용(클리핑↓) — 불가/고위험이면 분리 보고(안전 우선).
5. **최종 검증·보고**: 전 게이트 + 합성 fixture 회귀 게이트 편입, 보고서.

각 단계 완료 후 단계별 완료보고서 + 승인 요청(소스 커밋은 단계별 타스크 브랜치 커밋).
