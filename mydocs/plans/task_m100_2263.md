# 수행계획서 — M100 #2263: BinData 지연 로딩

## 1. 배경

`Document` IR이 문서의 모든 내장 바이너리(BinData)를 **파싱 시점에 압축 해제한 상태로 상주**시킨다.

- `src/model/document.rs:43` — `bin_data_content: Vec<BinDataContent>`
- `src/model/bin_data.rs:70` — `BinDataContent.data: Vec<u8>`

결과적으로 **1페이지만 렌더해도 문서 전체의 이미지가 메모리에 올라온다.**

## 2. 측정 근거

native release 빌드, `/usr/bin/time -l` 로 최대 RSS 측정. 렌더링을 수행하지 않는 **텍스트 추출 경로(파싱 + 조판만)** 기준:

| 문서 특성 | 압축 해제 내장 이미지 총량 | 페이지 수 | 최대 RSS |
|---|---|---|---|
| 무손실 비트맵 다수 내장 | 약 173 MB | 20 | **244 MB** |
| 이미지 중간 규모 | 약 37 MB | 74 | 78 MB |
| JPEG 다수 내장 | 약 29 MB | 141 | 97 MB |

RSS가 페이지 수가 아니라 **압축 해제 이미지 총량**을 따라간다.

경험적 모델: `RSS ≈ 프로세스 바닥(약 5MB) + 압축해제 이미지 전량 + 페이지당 약 0.4MB 조판 구조`

재현 조건: 수 MB 규모 문서에 무손실 비트맵을 다수 내장하면 원본 파일 크기의 20~30배 RSS가 발생한다. 원본이 작은 이유는 컨테이너(ZIP deflate / CFB zlib)가 비트맵을 강하게 압축하기 때문이며, 파서가 이를 전량 미리 풀어 IR에 상주시킨다.

## 3. 목표

내장 바이너리를 IR에 압축 해제 상태로 상주시키지 않고, **실제로 참조되는 이미지만 필요 시 압축 해제**하는 구조로 전환한다.

핵심 관찰: 압축된 원본 컨테이너는 작다(위 사례에서 7.9MB). **원본 바이트를 보관하고 id 단위로 지연 압축 해제**하면 지배항이 사라진다.

## 4. 영향 범위

| 대상 | 영향 |
|---|---|
| WASM / 브라우저 뷰어 | 동일 비용을 현재도 지불 중. 탭 메모리 압박·대용량 문서 로드 실패 완화 |
| macOS Quick Look 확장 (후속) | 확장 프로세스는 약 80MB 경고 / 120MB 강제 종료. 본 이슈 해결이 진입 전제조건 |
| CLI 내보내기 | RSS 감소 |
| HWPX 저장 / roundtrip | **주의 대상** — 저장 시 BinData 재출력이 필요하므로 원본 접근이 유지되어야 함 |

## 5. 접근 방안

### 5.1 교체 지점

`BinDataContent`는 레이아웃 전반에 `&[BinDataContent]` 슬라이스로 전달되고, 조회는 `find_bin_data(bin_data_content, id)` 한 곳으로 수렴한다 (`src/renderer/layout.rs:2509` 등). **이 조회 경계가 지연 로딩의 유일한 교체 포인트**이므로 호출부 확산 없이 치환 가능하다.

### 5.2 설계 골자

- 압축 해제된 `Vec<u8>` 상주를 폐기하고, **원본 컨테이너 바이트를 보관**한 뒤 id 단위로 압축 해제하는 저장소를 도입한다.
  - HWPX: ZIP 엔트리(deflate) 지연 inflate
  - HWP5: CFB 내 BinData 스트림(zlib) 지연 inflate
- 조회 API는 기존 `find_bin_data` 시맨틱을 유지하되 소유권 대신 빌림/캐시 핸들을 반환한다.
- 압축 해제 결과에 **상한이 있는 캐시**를 두고, 페이지 렌더 종료 시 해제 가능하게 한다.

### 5.3 병행 확인 필요 항목

- `page_tree_cache` (`src/document_core/commands/document.rs:150`) 가 페이지별 렌더 트리를 캐시한다. 이 캐시가 이미지 바이트를 품고 있다면 지연 로딩 효과가 상쇄되므로, **캐시가 무엇을 보유하는지 먼저 확인하고 필요 시 축출 정책을 함께 다룬다.**
- 저장 경로(HWPX serialize)가 `bin_data_content`를 소비하는 방식. 원본 접근이 끊기면 이미지 유실 회귀가 발생하므로 **roundtrip 게이트가 1차 방어선**이다.

## 6. 검증 계획

| 항목 | 방법 | 통과 기준 |
|---|---|---|
| 메모리 회귀 | 이미지 다수 내장 문서 1페이지 렌더 RSS를 `/usr/bin/time -l` 로 측정 | 압축 해제 이미지 총량이 RSS를 지배하지 않음 |
| 시각 회귀 | SVG 회귀 게이트 | 기존 대비 차이 없음 |
| 저장 무결성 | `cargo test --test hwpx_roundtrip_baseline` | 기존 통과 유지 (이미지 유실 없음) |
| WASM | WASM 빌드 후 뷰어 로드 | 동일 적용, 렌더 정상 |
| 일반 | `cargo test`, `cargo clippy` | 통과 |

## 7. 위험 요소

- **저장 시 이미지 유실**: 지연 로딩 도입 시 원본 접근이 끊기면 HWPX 저장에서 BinData가 비게 된다. roundtrip baseline 이 이를 잡아야 한다.
- **캐시 상쇄**: 렌더 트리 캐시가 이미지를 붙들고 있으면 메모리 이득이 사라진다. 5.3에서 선확인한다.
- **WASM 제약**: WASM에서는 원본 바이트가 이미 메모리에 있으므로 이득 구조가 native와 다르다. 그래도 "압축 해제본 전량 상주" 제거 이득은 동일하다.
- **소유권 구조 변경 파급**: `&[BinDataContent]` 를 받는 레이아웃 함수 시그니처가 다수다. 조회 경계가 좁아 파급은 제한적일 것으로 보이나, 구현 계획서 단계에서 실제 범위를 확정한다.
