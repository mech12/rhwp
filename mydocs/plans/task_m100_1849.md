# Task M100 #1849 수행 계획서

## 개요

- 이슈: #1849 `[CI] devel push profile 배치 재조정: release-test 회귀 + release smoke`
- 부모 이슈: #1668 `[Tracking/RFC] CI Build & Test 실행 시간 단계별 단축`
- 근거 이슈: #1664, #1666
- 마일스톤: M100 / v1.0.0 조판 엔진 체계화
- 작업 브랜치: `task-1849-ci-profile-policy`
- 작업 성격: CI profile 배치 정책 재검토 및 workflow 변경 후보

## 착수 판단

#1849는 #1666의 구현 오류 수정이 아니다. #1666은 메인테이너 결정사항을 엄격히 반영해 PR 경로를
`release-test` 중심으로 전환했고, PR `Build & Test` 단축 효과를 확인했다.

다만 #1666 merge 후 `devel` push에서 full `release --tests` 비용이 실제 수치로 드러났다. 따라서 #1849는
#1666 측정 결과를 근거로 PR / `devel` / `main` / tag / release workflow 사이의 profile 배치 정책을
재조정할지 판단하는 후속 sub-issue다.

이 작업은 완료된 #1666의 하위 이슈가 아니라, 부모 tracking 이슈 #1668의 하위 이슈로 관리한다.

## 문제 정의

현재 `.github/workflows/ci.yml`의 `Build & Test` job은 #1666 이후 다음처럼 동작한다.

| 이벤트 | Build | Native Skia tests | Run lib tests | Run integration tests |
|--------|-------|-------------------|---------------|-----------------------|
| `pull_request` | `release-test` | `release-test` | `release-test` | `release-test` |
| 그 외 push/tag/manual | `release` | `release` | `release` | `release` |

이 분기 덕분에 PR은 빨라졌지만, `devel` push도 tag/release 수준의 full `release --tests`를 수행하게 되어
merge 후 피드백 루프가 50분대가 됐다.

## 정책 후보

| 단계 | 제안 profile | 의도 |
|------|--------------|------|
| PR CI | `release-test` 전체 회귀 검증 | 빠른 피드백과 회귀 가드 1:1 추적성 유지 |
| `devel` push CI | `release-test` 전체 회귀 검증 + 제한적 `release` smoke | merge 후 통합 확인은 빠르게 유지하되 release profile 자체의 파손은 조기 감지 |
| `main` push / tag / release workflow | full `release` build/test/integration | 실제 릴리스 산출물에 가까운 최종 검증 |
| nightly/scheduled | full `release --tests` | release-only failure를 너무 늦게 발견하지 않기 위한 보완 후보 |

일반적인 Rust CI 운영에서는 PR과 merge queue 성격의 경로는 빠른 테스트 profile로 회귀를 잡고, LTO 또는
release-grade 검증은 tag/release/scheduled 경로에 배치하는 방식이 더 흔하다. 이 프로젝트도 회귀 가드 수가 많고
외부 PR 처리량이 높은 편이므로, `devel` push를 50분대로 유지하는 것은 PR 처리와 후속 merge 확인 측면에서 비용이 크다.

## 구현 후보

구현 계획서 승인 전에는 아래 후보 중 하나를 확정하지 않는다.

### 후보 A: 보수적 전환

- PR: 현행 유지. 전체 회귀 검증 `release-test`.
- `devel` push:
  - Build: `cargo build --release --verbose`를 release smoke로 유지
  - Native Skia tests: `cargo test --profile release-test --features native-skia skia --lib --verbose`
  - Run lib tests: `cargo test --profile release-test --lib --verbose`
  - Run integration tests: `cargo test --profile release-test --tests --verbose`
- `main` push / tag / `workflow_dispatch`: full `release` 유지

장점:

- `devel` push에서 가장 큰 비용인 release integration test를 제거한다.
- release profile 자체의 compile/link 파손은 `cargo build --release` smoke로 계속 잡는다.
- 현재 job 이름과 required check 표면을 유지할 수 있다.

단점:

- `devel` push에서 release-only test failure를 놓칠 수 있다.
- release-only test failure는 `main` / tag / release / scheduled 경로에서 발견될 수 있다.

### 후보 B: smoke 강화

후보 A에 더해 `devel` push에서 제한적 release test smoke를 추가한다.

예:

- `cargo test --release --lib --verbose` 전체 또는
- 특정 smoke test subset

장점:

- release-only lib test failure를 더 빨리 잡을 수 있다.

단점:

- #1666 측정상 release lib test 자체는 3-4분대지만, feature/test target 조합에 따라 compile 비용이 다시 커질 수 있다.
- 구현 복잡도와 측정 해석 비용이 늘어난다.

### 후보 C: 현행 유지

- `devel` push full `release --tests` 유지
- #1665 병렬화로 40분대 integration 비용을 줄이는 방향만 검토

장점:

- release-grade 검증 의미가 가장 강하다.

단점:

- `devel` push `Build & Test` P50 56m55s, P90 59m14s 비용을 유지한다.
- 병렬화 전에는 merge 후 피드백이 지나치게 느리다.
- 병렬화 후에도 runner-minutes 증가 가능성이 크다.

## 1차 권장안

후보 A를 1차 구현안으로 검토한다.

이유:

- #1666 PR 경로 개선은 유지한다.
- #1666에서 새로 드러난 `devel` push release integration 비용을 직접 줄인다.
- `devel` push에 `cargo build --release` smoke를 남겨 release profile compile/link 파손을 조기 감지한다.
- full release 검증은 `main` push / tag / release workflow 쪽으로 남길 수 있다.
- #1665 병렬화 전에 profile 정책을 먼저 안정화해 병렬화 효과 산정 기준을 명확히 만들 수 있다.

## 범위

### 포함

- `.github/workflows/ci.yml`의 profile 선택 정책 검토

- 회귀 가드 파일 통합, 명명 규칙 변경, 자산 구조 변경은 하지 않는다.
- `Swatinem/rust-cache` 도입은 #1667 범위다.
- job 병렬화는 #1665 범위다.
- branch protection / required check 이름 변경은 하지 않는다.

## 검증 계획

### 구현 전

- `.github/workflows/ci.yml`의 현행 분기 확인
- #1666 measurement의 before/after 기준선 확인
- #1668 부모 추적 문서의 공통 측정 기준 확인

### 로컬 정적 검증

- `git diff --check`
- 가능하면 `actionlint .github/workflows/ci.yml`
- 변경 범위 확인: `.github/workflows/ci.yml` 중심
- `Cargo.toml`, `tests/**`, `tests/golden_svg/**` 변경 없음 확인

### PR CI 관측

- PR 경로가 현행 `release-test` 전체 회귀 검증을 유지하는지 확인
- `Build & Test` job 이름과 check 표면 유지 확인
- `Build`, `Native Skia tests`, `Run lib tests`, `Run integration tests` 모두 실행 확인
- cache restore hit/miss와 PR save skipped 확인
- 회귀 가드 1:1 추적성 확인

### merge 후 `devel` push 관측

- `devel` push에서 전체 회귀 검증이 `release-test`로 실행되는지 확인
- release smoke가 실행되는지 확인
- full `release --tests`가 `devel` push에서 제거되었는지 확인
- cache exact hit / save skipped 또는 save success 상태 확인
- `Run integration tests` 시간이 #1666 merge 후 P50 42m27s 대비 줄었는지 확인

### main/tag/release 경로 확인

- `main` push / tag / release workflow에서 full `release` 검증이 남아 있는지 확인
- 실제 tag run을 만들지 않는 경우 workflow 조건과 PR diff로 경로를 명시한다.
- full release 검증이 부족하다고 판단되면 nightly/scheduled 추가는 후속 정책 판단으로 분리한다.
