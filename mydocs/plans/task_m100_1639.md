# Task #1639 수행계획서 — 빈 host 문단 음수 offset co-anchored float 표가 페이지/배치 순서를 역전

## 1. 이슈

- GitHub: <https://github.com/edwardkim/rhwp/issues/1639>
- 제목: `[HWP] PR #1518 이후: 빈 문단 co-anchored float 표의 음수 vertical_offset 으로 페이지 순서 역전 (실문서, #1510 계열)`
- 상태: OPEN
- 관련: #1510(PR #1518 보정), #1535(PR #1548)·#1549(PR #1568)는 visible-host 계열 — 본 건은 **빈-host** 계열로 별개.

## 2. 재현 (확정)

작업지시자 제공 한글 양식(나린뜰 HACCP 검증점검표, 비공개)과 issue1549 구조 Clone and Narrow 합성 fixture 로 확정.

- 빈 host 문단(`text_len=0`)에 co-anchored TopAndBottom float 표가 2개 이상 있고 후행 표의 `vertical_offset` 이
  음수면, 표 배치/렌더/페이지 순서가 배열(문서) 순서와 역전된다.
- 실파일 dump(나린뜰): 문단0(빈), `[2]`표(설명) `vert=문단(+1154)`, `[3]`표(본문) `vert=문단(u32 4294962885
  = i32 -4411)`. export-pdf 결과 1p=본문, 2p=설명(역전). 한컴/LibreOffice 는 1p=설명, 2p=본문.
- 합성 fixture dump: `[2]`표 +200, `[3]`표 -4411 → `dump-pages` 배치 `[3,2,4]`(배열 `[2,3,4]` 역전).

## 3. 근본 원인 (소스 확정)

`src/renderer/typeset.rs`:

- `should_sort_para_float_tables = !para_has_non_whitespace_text(para)` — 빈 host 에서 para-relative float 표를
  정렬한다(#986/#1088: 빈 host 는 배열 순서 ≠ 시각 순서일 수 있어 `vertical_offset` 오름차순 안정정렬).
- 정렬 키 `t.common.vertical_offset as i32` — 음수 비트패턴이 음수 i32 로 해석된다.
- `ctrl_order.sort_by_key((voffset, tiebreak))` 오름차순 → 음수 offset(-4411)이 양수(+200) 앞으로 정렬되어
  `ctrl_order` 가 역전 → 배치/렌더/페이지 순서 역전.
- #1510(visible-host)·PR #1518 은 visible-host 케이스를 보정했고, 빈 host 음수 offset 은 정렬 대상으로 잔존.
- 파싱은 정상(offset 비트는 u32 로 보존, 정렬 시점에 i32 해석) — 데이터/파싱 버그가 아니라 정렬 휴리스틱 한계.

## 4. 수정 방향 (옵션 검토 → B 채택)

- **옵션 A**(정렬 키 `signed_hwpunit(..).max(0)` 클램프): 음수→0. 그러나 선행 표가 양수(나린뜰 +1154, fixture
  +200)면 `0 < 양수` 라 여전히 역전 → **본 케이스 미해결**(fixture dump 게이트로 선행 표 양수 확인).
- **옵션 B**(채택): 게이트에 "음수 para-topbottom float 존재 시 정렬 OFF" AND 조건 추가. 음수 혼재 빈 host 는
  배열(문서) 순서를 보존. 양수 전용 빈 host 의 #986/#1088 정렬과 visible-host 제외(#1510)는 불변.
- **옵션 C**(lane-aware 부분 억제): 정밀하나 구조 변경 大 → 범위 초과.

근거: #1510 "한글은 앵커/문서 순서대로 배치" 불변식을 빈 host 로 일관 적용. 최소 변경(게이트 1곳) + 회귀 안전.
