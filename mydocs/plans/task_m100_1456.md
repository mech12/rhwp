## 1. 배경

rhwp-studio 캔버스에서 **HWP 차트/OLE가 첫 로드 시 공백**으로 뜨고, 맨 처음 열린 차트 이미지가 이후 파일에도 고착되며, HWPX를 먼저 열고 같은 종류 HWP를 열면 그제서야 렌더되는 비대칭이 C1a(#1453) 시각검증 중 드러났다.

native `render_page_svg` + `export-svg`(SVG 경로) 산출물은 hwp/hwpx 모두 정상이다 — **studio 캔버스 경로 전용** 결함이며 C1a 차트 로직 회귀가 아니다. C1a가 차트를 실제 렌더하게 만들며 잠복 결함이 표면화됐다.

## 2. 사실 검증 (소스 수정 전 확인 완료)

| 항목 | 확인 결과 |
|---|---|
| studio 화면 렌더 경로 | SVG 아님 — 캔버스. `PageRenderer.renderPage`(`page-renderer.ts:24`), `canvas-view.ts:184`에서 호출 |
| 차트/OLE 래스터화 | `rawSvg` PaintOp → `web_canvas.rs:1016 render_raw_svg` → `draw_image`(`web_canvas.rs:2535`)로 **비동기 `HtmlImageElement`** |
| 캐시 미스 동작 | `draw_image` 캐시 미스 + 디코드 전이면 **아무것도 안 그림**, onload 재렌더 트리거 없음(`web_canvas.rs:2596`) |
| #1181 안전망 게이트 | `scheduleReRender`가 `imageCount > 0`일 때만 작동(`page-renderer.ts:305`) |
| imageCount 산정 주체 | **TS측** `collectLayerPlaneSummary`가 풀 트리 JSON(`getPageLayerTree`) 순회로 독립 계산 |
| 누락 지점 | `collectLayerPlaneSummary`가 `op.type==='image'`만 카운트, **`rawSvg` 누락**(`page-renderer.ts:449`) |
| 데이터 가용성 | 풀 트리 JSON은 RawSvg op에 `"type":"rawSvg"`를 정상 emit(`paint/json.rs:938`) → 데이터는 있는데 TS가 안 셈 |
| #1181이 "불완전"인 이유 | #1181(commit `b54ad826`)은 RawSvg를 **죽은 엔드포인트** `get_page_overlay_images`의 `image_count`(`rendering.rs:844`)에 더함. 그 TS 래퍼 `getPageOverlayImages`(`wasm-bridge.ts:482`)는 **호출처 없음** → 트리거 경로와 미연결 → 무효 |
| prefetch의 rawSvg 처리 | `prefetchLayerImages`는 이미 rawSvg 임베디드 data URL 추출(`page-renderer.ts:400-407`) — 단 게이트 bail로 도달 못 함 |
| 고착(stuck) 원인 | IMAGE_CACHE(thread_local, SVG바이트 FNV해시 키, 세션 지속) — 동일 내용 차트는 같은 해시 → 캐시 재사용(시각 동일이라 무해). **서로 다른 내용 차트 간 고착** 실재 여부는 런타임 확인 필요 |
| 샘플·정답지 | `samples/chart/`(hwp 27 + hwpx 27종) 존재. SVG 경로 정답지 `pdf/chart/` |

→ 이슈의 현상·원인·제안이 모두 사실로 확인됨. **근본 원인 = 재렌더 트리거 게이트(`imageCount`)가 rawSvg를 안 셈**.

## 3. 목표

rawSvg 단독(차트/OLE) 페이지가 비동기 디코드 완료 후 **결정적으로 재렌더되어 그려지도록** 만든다. `collectLayerPlaneSummary`가 `rawSvg` op도 재렌더 트리거 카운트에 반영하여 `scheduleReRender`(200/600/1500ms 재렌더 + `prefetchLayerImages`)가 발화되게 한다.

## 4. 범위

작업지시자 결정(2026-06-24): **최소·핵심(#1) 먼저 → 런타임 검증에서 고착이 재현되면 CanvasPool(#3)까지 확장.**

### 포함 — #1 (최소·핵심)
- `page-renderer.ts` `LayerPlaneSummary`/`collectLayerPlaneSummary`에 rawSvg 카운트 추가(`rawSvgCount` 별도 필드 권장, `imageCount`=래스터 이미지 op 의미 유지).
- 트리거 호출부 `renderPage`(`page-renderer.ts:42`)에서 `scheduleReRender(..., imageCount + rawSvgCount)` 전달.
- E2E 회귀 테스트 신규(`issue-1456-chart-rerender.test.mjs`).

### 조건부 포함 — #3 (CanvasPool)
- Stage 3 런타임 검증에서 **서로 다른 내용 차트 간 고착이 재현될 때만** `canvas-pool.ts` release/acquire 픽셀 클리어 + 오버레이 레이어 정리 추가.

### 제외
- **#2(견고화)**: `web_canvas.rs draw_image` 캐시 미스 시 `HtmlImageElement.onload`→재렌더 이벤트 콜백(Rust→JS plumbing). 복잡도·리스크로 본 이슈 범위 밖, 후속 이슈 후보.
- `get_page_overlay_images`의 죽은 #1181 코드 정리 — 기능 변경과 섞지 않음. 보고서에 관찰로만 기록.

### 변경 면적
- #1: rhwp-studio TS만(`page-renderer.ts`) + E2E 테스트 신규. Rust 무변경.
- #3(조건부): `canvas-pool.ts`.

## 5. 검증 기준

1. **타입/빌드**: `cd rhwp-studio && npm run build` 통과(타입 무회귀).
2. **E2E(신규)**: 새 세션(IMAGE_CACHE 초기화)에서 차트 HWP 로드 → 캔버스 차트 영역 **비공백**(픽셀 샘플링) 단언.
3. **비재사용 단언**: 서로 다른 두 차트 순차 로드 → 두 번째가 첫 번째 픽셀을 재사용하지 않음.
4. **수동 재현**: studio 새로고침 → 차트 HWP 첫 로드 → (수정 전 공백 / 수정 후 렌더) 확인.
5. **무회귀**: 기존 핵심 e2e(`canvas-render-diff` 등) 통과.
