## 2. 착수 전 조사 (코드 무수정)

### 2-1. #1937 fix 무관 확인
#1937(각주 continuation) 수정 포함 바이너리로도 수면비행선박 여전히 >400s. →
페이지 폭주(#1937)와 **별개**인 재측정 성능 축.

### 2-2. 수면비행선박 구조 — 거대 셀 + 대량 중첩 표
- `dump` 자체가 19.8s(구조 출력만). 파싱은 OK(info 즉시).
- 바깥 표: **3행×1열 RowBreak**(제목반복). 그런데 **셀[2] 문단수 = 2507**.
- 셀[2] 안에 **중첩 표 수십 개**(9×9=74셀, 14×2=28셀, 8×2, 6×2, 4×3 …)가 산재
  (p[81]/p[288]/p[322]/p[619]/p[1517]/p[1786]/p[2276]/p[2280] …).
- → 바깥 3×1 표가 RowBreak 로 여러 쪽에 분할될 때, **셀[2](2507문단+수십 중첩표)의
  콘텐츠·중첩표 높이를 페이지마다 재측정**하면 O(pages × 2507문단 × 중첩표) 로 폭발.

### 2-3. 가설
분할 루프(`typeset.rs` `typeset_block_table`/`advance_row_cut`/`row_block_content_height`
및 `height_measurer` 중첩표 측정)가 거대 셀 콘텐츠를 **쪽마다 재계산** → 병리적 성능.
`#1046 cut_row_h` 재측정 계열(세션 진단)과 정합 추정. **프로파일링으로 확정 필요.**

## 3. 수행 접근

프로파일링으로 핫스팟 함수를 1개로 특정한 뒤, **측정 결과 메모이제이션(캐시)**로
O(pages×content) → O(content) 로 낮춘다. 시각/페이지 결과 불변(순수 성능 최적화)을
회귀 게이트로 보증.

## 4. 단계 구성 (구현계획서에서 상세화)

- **Stage 1 — 핫스팟 특정**: 3개 파일에 대해 단계별 타이밍 계측(parse/measure/paginate/
  render) + 거대 셀·중첩표 재측정 호출 카운트. 지배 핫스팟 함수 1개 확정.
- **Stage 2 — 최적화 설계**: 핫스팟 재측정 결과를 표/셀 단위로 캐시(중첩표 MeasuredTable,
  셀 콘텐츠 높이). 캐시 무효화 조건·메모리 영향 평가. 승인 요청.
- **Stage 3 — 구현 + 게이트**: 캐시 적용, 3개 파일 render 시간 목표(수 초) 확인,
  **시각/페이지 결과 불변** 회귀(cargo test --lib + 표/중첩표 통합 테스트 +
  render-tree 좌표 동일성) 확인.

## 5. 리스크

- 캐시 무효화 오류 시 잘못된 높이 재사용 → 시각 회귀. → render-tree 좌표 before/after
