# 수행계획서 — Task #1616

**제목**: 발신명의 footer(Page+Bottom) 독립-쪽 배치 규칙 역공학
**마일스톤**: M100 · **이슈**: edwardkim/rhwp#1616 · **브랜치**: `local/task1616` (base: `local/task1612`)
**전사**: Task #1600 −1쪽 갭 잔여 8건(razor-thin). #1611(overflow) 후 남은 footer 케이스는
기하상 page 1 에 맞는데 한글이 page 2 에 footer 단독 배치.

## 1. 목표
통제셋(92건, 한글 정답 보유)의 footer 보유 문서에서 rhwp 특징을 추출해 "한글이 footer 를
독립 쪽으로 미는" 임계 규칙을 통계적으로 도출. 규칙 도출 시 후속 수정, 미도출 시 한계 확정.

## 2. 가설 후보 (측정 대상)
- **H1 headroom**: 한글은 footer 가 page1 하단에 너무 가까우면(`available − (footer_vpos+footer_h)`
  < 임계) 독립 쪽으로 민다. (−1 8건 headroom ~26px vs match 1쪽 케이스 ~113~183px 관측.)
- **H2 inner-height**: 한글이 footer 내부표 실제 높이로 렌더(rhwp 는 선언높이로 clamp).
  inner_height > 선언높이면 한글에서 overflow → page 2.
- **H3 body_fill**: 한글은 body_end / available 비율이 임계 초과 시 footer 를 민다.

## 3. 방법
1. **특징 추출**(rhwp, 한글 불필요): 각 통제셋 footer 문서의 footer_vpos_px·footer_declared_px·
   footer_inner_height_px·body_available_px·body_end_px. `dump`/`dump-pages` 파싱.
2. **라벨**: 통제셋 hangul_pages, rhwp_pages → "한글 독립쪽(push)" vs "유지(keep)".
3. **분리 분석**: 각 가설 특징이 push/keep 을 분리하는지(임계·오분류) 측정.
4. 깔끔히 분리되는 특징 발견 시 → 규칙화 + Stage 2 수정. 미분리 시 → 한계 확정·문서화.

## 4. 단계 (조사 우선)
- **Stage 1**: 특징 추출 하니스 + 분리 분석(코드 수정 없음). 규칙 도출 여부 판정 → 보고.
- **Stage 2+**(규칙 도출 시): RED + typeset 수정 + 통제셋 게이트(net>0). (#1611 의 Page+Bottom
  처리 확장.)
