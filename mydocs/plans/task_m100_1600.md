## 1. 배경

전수 3차원 검증(`mydocs/report/hwpx_lossless_3axis_20260627.md`)에서 rhwp 렌더링이
한글 대비 ~91% 페이지 정합, 불일치 9.3% 중 대부분 **−1쪽**. 한글이 2쪽으로 넘기는 내용을
rhwp가 1쪽에 담는다. = **rhwp 콘텐츠 누적 높이가 한글보다 미세하게 작다.**

## 3. 근본원인 가설 (Stage 1 확정 대상)

per-line/element ~2-5px 누적이 페이지 경계를 가른다. 후보:
- **줄간격(line spacing)** 계산이 한컴보다 작음 (ls=13.9px @ 줄간격 160% 등).
- **문단 간격(sb/sa)** 또는 빈 문단 높이.
- **표 높이**(셀 padding/행 높이) 과소.
- **페이지 채움 임계**(마지막 항목 오버플로 허용 폭).

(#1589 sub-line 누적 높이차 조사와 동류 — 이진탐색+한글 정답지 비교가 추측 압도.)

## 4. 방법론

권위 정답지 = 한글 편집기/PDF 페이지 분할 위치. 통제 비교:
1. −1쪽 케이스 N개 고정 셋 구성(한글 페이지수 기지).
2. 한글에서 페이지 브레이크가 **어느 문단/줄**에서 일어나는지 확인(visible/PDF).
3. rhwp 해당 지점 누적 높이 vs 한글 → 차이의 **단일 높이 컴포넌트** 역추적(이진탐색:
   줄간격만/표만/spacing만 조정해 페이지수 변화 측정).
4. 컴포넌트 확정 후 정합 수정 → 통제 비교(−1쪽 해소율, 악화 0).

## 5. 구현 단계 (5단계, Stage1 조사 포함)

- **Stage 1 — 근본 컴포넌트 확정**: 고정 −1쪽 셋 + 한글 브레이크 위치 비교 + 이진탐색으로
  과소 높이 컴포넌트 1개 역추적. (코드 수정 전 조사 — 보고 후 승인.)
- **Stage 2 — RED**: 해당 컴포넌트 높이 단위 테스트(현재 과소 → RED) + 대표 케이스 페이지수 가드.
- **Stage 3 — 정합 수정**: 컴포넌트 높이 계산 한컴 정합. Stage2 GREEN.
- **Stage 4 — 회귀**: 기존 레이아웃 골든(SVG/페이지수) 회귀 0 + lib/baseline.
- **Stage 5 — 통제 비교**: 무작위 표본 페이지 정합률 재측정(−1쪽 해소, +1쪽 회귀 0).

## 6. 회귀 위험 (높음)

레이아웃 높이 계산은 **전 문서 영향**. 한 컴포넌트 조정이 +1쪽 회귀를 유발할 수 있어,
통제 비교(개선−회귀>0)가 필수 게이트. 골든 SVG 광역 변동 점검.
