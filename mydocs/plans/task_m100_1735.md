# 수행계획서 — 방점(U+302E/U+302F) 렌더링 정합 (M100 #1735)

- **이슈**: [edwardkim/rhwp#1735](https://github.com/edwardkim/rhwp/issues/1735)
- **브랜치**: `local/task1735` (from `local/devel`)
- **마일스톤**: M100 (v1.0.0)
- **분류**: bug / rendering (유니코드 렌더 정합)

## 1. 개요 / 목표

`samples/unicode/각 항목에 명시되어 있는_유니코드.hwp` 를 rhwp에서 열면 한컴 정합과 다르게 표기된다. 맨 앞 **방점 U+302E**(HANGUL SINGLE DOT TONE MARK)가 한컴의 독립된 가운데 점(`·각 …`)과 달리, rhwp에서는 **점 + 빈 원(◌)이 붙은 결합문자 아티팩트**로 나타난다.

목표: rhwp가 방점 **U+302E/U+302F** 를 한컴처럼 **독립된 spacing 가운데 점**으로 렌더하도록 하여 dotted-circle 아티팩트를 제거하고 시각 정합을 맞춘다.

## 2. 근본 원인 (코드 근거)

1. **결합문자 dotted-circle (주 증상)**
   - U+302E는 유니코드 결합문자. 유효 base 없이(줄 시작/공백 뒤) 셰이핑되면 브라우저/엔진이 **dotted-circle(U+25CC)** placeholder를 삽입하고 톤 점을 그 위에 쌓는다.
   - 모든 렌더 표면이 IR 텍스트를 그대로 내보냄. 렌더 전용 확장 choke point `expand_pua_render_text → expand_pua_display_text`([src/renderer/composer.rs:1691-1713](../../src/renderer/composer.rs#L1691-L1713))에 톤마크 처리 없음.
   - 호출처(모두 render/paint): [svg.rs:2664](../../src/renderer/svg.rs#L2664), [html.rs:283](../../src/renderer/html.rs#L283), [canvas.rs:345](../../src/renderer/canvas.rs#L345), [web_canvas.rs:2060](../../src/renderer/web_canvas.rs#L2060), [skia/text_replay.rs:327](../../src/renderer/skia/text_replay.rs#L327), [paint/json.rs:1426](../../src/paint/json.rs#L1426)(스튜디오 계약). **텍스트 추출 경로와 공유하지 않음** → 치환해도 텍스트 시맨틱 보존.

2. **폭 오분류 (부 증상 — 간격/겹침)**
   - `get_width()`([font_metrics_data.rs:35-59](../../src/renderer/font_metrics_data.rs#L35-L59))는 저장폭 0이면 `None` 반환. U+302E는 결합 글리프라 advance=0 → `None`.
   - 폴백 휴리스틱([text_measurement.rs:326-334](../../src/renderer/layout/text_measurement.rs#L326-L334))에서 `is_cjk_char`·`is_fullwidth_symbol`·`is_narrow_punctuation` 모두 false(0x3000–0x303F 블록 미포함) → **반각(0.5em)**. 한컴 대비 폭 어긋남.

> 참고: dump 상 문단 0.1(cc=1, text_len=0)은 **정상 빈 마지막 문단**이며 렌더 누락 버그 아님(한컴도 미출력). 초기 "문단 통째 누락" 가설 폐기.

## 3. 범위

- **방점 U+302E / U+302F 2자로 한정** (작업지시자 확정).
- 0x3000–0x303F 블록 전체 확장은 하지 않음(회귀 위험 회피).
- 공통 렌더/측정 모듈만 수정. 파서·HWP3 전용 로직 무관.

## 4. 수행 방침

1. **렌더 치환(주)**: `composer.rs` 렌더 전용 확장에서 방점을 spacing 가운데 점 글리프로 치환. 기존 PUA 특수문자 치환(`pua_plain_text_display`)과 동일 패턴. IR 텍스트 불변.
   - 치환 글리프 후보(시각 비교로 확정): **U+302E → U+318D(ㆍ, ARAEA)** 권장 — 코드베이스가 이미 전각·spacing·가운데 점으로 인식([text_measurement.rs:1802](../../src/renderer/layout/text_measurement.rs#L1802)). 대안 U+00B7(·).
   - **U+302F(쌍방점)** → best-effort. 현재 샘플엔 없음. 마땅한 글리프 부재 시 후속 이슈 분리.
2. **폭 일관성(부)**: `text_measurement.rs`에서 U+302E/U+302F 측정 폭을 치환 글리프 advance와 일치(예: `is_fullwidth_symbol`에 2자 추가). 측정 원본은 IR의 U+302E이므로 **측정 폭 == 렌더 글리프 폭** 보장이 핵심.
3. **회귀 자산(선택)**: 톤마크 유닛 테스트 추가. `samples/unicode` 정식 편입 여부는 작업지시자 판단.

## 5. 예상 구현 단계 (상세는 구현계획서에서 확정)

- **Stage 1**: 렌더 치환(composer.rs) — dotted-circle 제거. SVG로 ◌ 미삽입 확인.
- **Stage 2**: 폭 분류(text_measurement.rs) 일관화 + 시각 정합(한컴 PDF 비교) 튜닝.
- **Stage 3**: 유닛 테스트 + fmt/test/clippy 게이트 + 최종 검증.

(3단계 기준. 구현계획서에서 3~6단계로 확정)

## 6. 검증

1. `cargo build` 후 `rhwp export-svg "samples/unicode/각 항목에 명시되어 있는_유니코드.hwp" -o output/svg/` → `<text>`에 U+25CC 미삽입, 방점 독립 점 렌더.
2. PDF/PNG 내보내기로 한컴 정답지 `samples/unicode/…_유니코드.pdf`와 시각 비교(점 위치·크기·`각`과의 간격).
3. `rhwp dump -s 0 -p 0` + SVG x좌표로 폭 정합(겹침/과대공백 없음) 확인.
4. `cargo fmt`(수정 파일 한정, 무관 diff 금지) → `cargo test` → `cargo clippy` 통과.
5. 작업지시자 시각 승인(정답지 = 한컴 PDF).
