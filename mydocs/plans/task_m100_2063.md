## 2. 근본 원인 (조사 완료, 코드 정독)

### (주) O(n²) 셀 측정 — `cell_units_uncached` (table_layout.rs:4627)
```rust
let has_visible_text_with_nested_table = table.cells.iter().any(|cell| {  // ← 전체 셀 스캔
    cell.paragraphs.iter().any(|p| !p.text.trim().is_empty()
        && p.controls.iter().any(|c| matches!(c, Control::Table(_)))) });
```
- 이 값은 **표 단위 불변량**(측정 대상 `cell`과 무관)이나 **셀별 함수 안**에서 계산된다.
- `cell_units`는 셀 포인터로 메모이즈되지만, 52,694셀 캐시를 채우는 과정에서 셀마다
  전체 셀(52,694) 스캔 → **52,694² ≈ 28억 회** (각 회 문단·컨트롤 중첩 순회). = 지배적 hot-path.

### (부) 프로파일로 확인할 2차 후보
- `row_has_prior_rowspan_cover` (L6207) `table.cells.iter().any()` — 행/페이지 루프서 호출 시 O(행×셀).
- `table_partial.rs:208` `table.cells.iter().any()` — 페이지 조각별 렌더 행마다 전체 셀 스캔.
- `resolve_row_heights` 페이지 조각별 재호출 — `measured_table` 전달 여부에 따라 O(페이지×셀).
  (`resolve_cell_padding`은 전체 셀 스캔 없음 — 확인 완료.)

### 과분할(+51)
- CellBreak 표의 셀별 세로 배치 누적이 한글보다 +32% 팽창. 행높이 드리프트(#1937/#1842 계열)와
  동근이나 **별개 correctness 사안**. 본 타스크에서는 **성능과 분리**한다(아래 범위 참조).

## 3. 범위

- **주 목표(본 타스크)**: 렌더 **성능** — O(n²) 제거로 초대형 표가 배치 타임아웃 내 완료.
  성능 개선은 **레이아웃 결과(페이지 수·좌표) 불변**이어야 한다(순수 최적화, 회귀 0).
- **부 목표(분리)**: +51 과분할(CellBreak 세로 팽창)은 correctness 사안 → 성능 픽스 머지 후
  별도 트랙/후속(또는 #1937 계열 통합)으로 다룬다. **같은 커밋에 섞지 않음**(CLAUDE.md 규칙).

## 4. 접근

1. **프로파일 확증**: 계측 훅(임시 eprintln 타이머 or 카운터)으로 4627 스캔이 지배적임을
   수치로 확인하고 2차 후보(6207/partial:208/resolve_row_heights) 기여도 측정.
2. **표-불변량 hoist**: `has_visible_text_with_nested_table`를 표 단위 1회 계산으로 이동.
   - 후보 A: 표 포인터 키 메모이즈 필드(`cell_units_cache`와 동형의 RefCell<HashMap>) 신설.
   - 후보 B: `cell_units_uncached` 시그니처에 미리 계산한 bool 전달(호출부 `cell_units`에서 1회 계산).
   - A/B 중 회귀·침습 최소안을 프로파일 후 확정.
3. 2차 후보가 유의하면 동일 원리(불변량 hoist / 페이지 간 재사용)로 제거.
4. 성능 목표: 21914299 전체 렌더가 배치 타임아웃(≤420s, 목표 ≪) 내 완료.
