## 1. 교정된 근본 원인

### 1.1 HWP5 바이너리는 indent 를 full HWPUNIT 로 저장한다 (절반 아님)

`parse_para_shape`(`doc_info.rs`) 에 raw 바이트 디버그를 넣어 확인:

- 3-11월 파일 paraPr(parse index 12) **raw 바이너리 indent = -3216** (HWPX `default` = full HWPUNIT 과 동일).
- HWP5 바이너리 전체에 **-1608 은 0건**. raw 값들(-3216, -9658, -15350, -4504, -1274 …)은 ir-diff 가 HWPX(A) 값으로 보고한 값과 정확히 일치.

→ HWP5 파서는 indent 를 **올바르게** full HWPUNIT 로 읽는다. 절반화는 파서가 아니다.

### 1.2 진짜 범인: `is_hwp3_variant` 후처리의 indent/spacing 절반화

`src/parser/mod.rs:327-330` (Task #1001 → #1037 → #1042):

```rust
if doc.is_hwp3_variant {  // summary_hwp3_era && total_paras>50 && ps_r<0.20 && cs_r<0.20
    for ps in &mut doc.doc_info.para_shapes {
        ps.indent /= 2;          // -3216 → -1608
        ps.spacing_before /= 2;
        ps.spacing_after /= 2;
    }
}
```

HWP3→HWP5 변환본의 HWP3 정합을 위해 도입된 보정. 3-11월 파일이 이 분기에 걸려 모던 HWP5 의 정상 indent(-3216)를 -1608 로 **손상**시킨다. (ir-diff 는 `parse_document` 경로라 이 보정이 적용되어 -1608, `dump`/raw 디버그는 파싱 직후라 -3216 — 두 경로의 차이가 혼선의 원인이었다.)

### 1.3 `is_hwp3_variant` 오탐 메커니즘

`cfb_reader::detect_hwp3_variant()` 는 **HwpSummaryInformation 텍스트에 "1990년~2003년" 문자열이 있으면** HWP3 시대로 판정하는 내용 기반 휴리스틱.

- 3-11월 summary: `"1997년 5월 25일 일요일, 3시 40분"` (작성 시각 메타, 1997 템플릿 유래 추정) + producer `Windows_10` (모던 재저장).
- ps_r=0.067, cs_r=0.044 (진짜 변환본 hwp3-sample16 의 0.174/0.154 보다 **더 낮아** 가드 통과).

→ 3-11월은 **모던 HWP5(정답 indent = full -3216, HWPX/PDF 와 일치)** 인데 summary 의 옛 연도 문자열 때문에 변환본으로 오탐된다.

### 1.4 시각 권위 확정 (정답 = full -3216)

upstream #1472 collaborator(jangster77): `.pdf`(한글2022) 와 `.hwpx` 가 정답지. 첨부 렌더 비교 — PDF / 한컴2024 hwp / 한컴2024 hwpx 모두 연속줄을 **깊은 내어쓰기**로 렌더(= full -3216). rhwp-hwp(절반화 -1608)만 얕음(오답). → **3-11월은 절반화하면 안 된다.**

## 2. 왜 v1(파서 ×2)이 틀렸나

블랭킷 ×2 는 변환본 보정(절반화)을 거치지 않는 **정상 HWP5 파일**(exam_science/exam_eng/21_언어)의 indent·spacing 을 2배로 부풀려 4개 PDF-ground-truth 통합 테스트(test_490/521/548/624)를 회귀시켰다. ×2 는 보편적이지 않다.

## 3. 수정 방향 — 결정 필요 (회귀 위험 영역)

이 수정은 민감한 HWP3 변환본 영역(#1001/#1042)과 직접 얽혀, 잘못 건드리면 진짜 변환본(hwp3-sample16 등) HWP3 정합을 회귀시킨다. **작업지시자 방향 결정 후 진행.**

- **옵션 A (탐지 강화 — 권장 후보)**: `detect_hwp3_variant` 또는 게이트를 강화해 3-11월 같은 모던 파일을 변환본에서 제외.
  - 후보 신호: producer/OS 문자열(`Windows_10` 등 모던 저장 마커), 모던 HWPX twin 존재, 파일 버전·생성기, summary 의 "옛연도"가 *생성일 필드*인지 본문 인용인지.
  - 장점: #1042 절반화 로직 불변(진짜 변환본 영향 0). 단점: 휴리스틱 추가 → 새 오탐/누락 가능, 케이스 검증 필요.
- **옵션 B (절반화 정당성 재검증)**: #1042 의 "변환본 indent = HWP3 ×2 → ÷2" 가정 자체를 진짜 변환본(sample16)의 HWP3 ground-truth 로 재검증. 절반화가 indent 에 대해 과적합이면 축소/제거.
  - 위험: HWP3 정답지 필요(민감 영역). 회귀 범위 큼.
- **옵션 C (보고만)**: upstream #1472 에 교정 진단을 코멘트로 남기고, 수정은 별도 합의.
