# 수행계획서 — task_m100_2048

- **이슈**: #2048 [vscode] macOS 웹뷰에서 WASM 로드 실패 — 메인 스레드 동기 컴파일(initSync) 차단
- **브랜치**: `local/task2048` (분기 기준: `upstream/devel`)
- **마일스톤**: M100 (v1.0.0)

## 1. 배경 / 문제

macOS의 VSCode 확장에서 HWP 문서를 열면 다음 오류로 렌더링이 중단된다.

> WASM 로드 실패: WebAssembly.Compile is disallowed on the main thread, if the buffer size is larger than 4KB. Use WebAssembly.compile, or compile on a worker thread...

Windows/Linux에서는 재현되지 않고 macOS 전용 증상이다.

## 2. 원인 (조사 완료)

`rhwp-vscode/src/webview/viewer.ts:44`

```ts
import { initSync, HwpDocument } from "@rhwp-wasm/rhwp.js";  // line 1
...
initSync({ module: buf });                                    // line 44
```

- `initSync`는 내부에서 `new WebAssembly.Module(bytes)`를 **메인 스레드에서 동기 실행**한다 (`pkg/rhwp.js:8400` 확인).
- `rhwp_bg.wasm`은 약 **5.9MB**로 4KB를 크게 초과 → 메인 스레드 동기 WASM 컴파일 금지 규칙에 위반.
- macOS 웹뷰는 규칙을 엄격히 강제하여 **차단**, Windows/Linux(Electron/V8)는 느슨하여 우연히 통과.

## 3. 해결 방향

동일 wasm-bindgen 패키지가 제공하는 **async 기본 export**(`__wbg_init`)로 교체한다. 이 함수는 내부 `__wbg_load`에서 `WebAssembly.instantiateStreaming` → 실패 시 `WebAssembly.instantiate`(모두 async)를 사용한다 (`pkg/rhwp.js:8420`, `__wbg_load` 확인 완료).

| 구분 | 현재 (동기) | 변경 (비동기) |
|------|------------|--------------|
| import | `import { initSync, HwpDocument }` | `import init, { HwpDocument }` |
| 초기화 | `initSync({ module: buf })` | `await init({ module_or_path: buf })` |

기존에도 `fetch(wasmUri).then(res => res.arrayBuffer())` 로 버퍼를 받고 있으므로 그 버퍼를 그대로 async init에 전달한다. Promise 체인 구조는 유지 가능.

## 4. 영향 범위

- **수정 파일**: `rhwp-vscode/src/webview/viewer.ts` (import 1줄 + 초기화 1줄)
- **재빌드**: webpack (`rhwp-vscode`), 산출물 `dist/` 갱신
- **다른 플랫폼**: async init은 Windows/Linux에서도 정상 동작 → 회귀 위험 낮음
- **WASM 바이너리(`pkg/`) 변경 없음**, Rust 코드 변경 없음

## 5. 검증 방법

- `rhwp-vscode` 빌드 성공 (webpack, 타입 체크)
- macOS VSCode에서 확장 실행 → HWP 문서 정상 렌더링 (오류 메시지 소멸)
- 기능 회귀 없음 (문서 로드 → 페이지 렌더 → 줌 동작)

## 6. 리스크

- 낮음. 변경은 동기 → 비동기 초기화 API 교체로 한정. 초기화가 이미 `.then()` 콜백 안이라 async 전환이 자연스럽다.
- 유일한 주의점: `init` 완료 이후에만 `wasmReady = true` 및 `HwpDocument` 생성이 일어나도록 순서 보장 (기존과 동일).
