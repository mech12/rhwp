# Task M100 #1664 수행 계획서

## 개요

- 이슈: #1664 `[CI] Actions cache quota/read-only 상태 해소 및 PR cache 저장 정책 정리`
- 부모 이슈: #1668 `[Tracking/RFC] CI Build & Test 실행 시간 단계별 단축`
- 마일스톤: M100 / v1.0.0 조판 엔진 체계화
- 작업 브랜치: `local/task1664`
- 기준 결정: #1668 메인테이너 코멘트 `2026-06-30T07:11:16Z`
- 문서 PR: #1701 (`mydocs/**` 정책/측정 기록 전용)
- 후속 코드 PR: #1702 (`.github/workflows/ci.yml` workflow 변경, draft 상태에서 관측값 수집)

## 목적

현재 `CI / Build & Test`는 `actions/cache@v5` 단일 step으로 cargo registry, cargo git, `target`을
restore/save한다. 이 구조에서는 PR merge ref에서도 cache save가 시도되어, cache quota 초과 또는
read-only 상태에서 매 PR마다 stale cache restore 뒤 재컴파일이 반복될 수 있다.

이번 작업의 목적은 CI 검증 의미를 바꾸지 않고, PR cache save를 차단해 cache quota/read-only 문제의 직접
원인을 먼저 해소하는 것이다.

## 현행 방식

- 대상 workflow: `.github/workflows/ci.yml`
- 대상 job: `Build & Test`
- 현재 cache step:
  - action: `actions/cache@v5`
  - path:
    - `~/.cargo/registry`
    - `~/.cargo/git`
    - `target`
  - key: `${{ runner.os }}-cargo-${{ hashFiles('**/Cargo.lock') }}`
  - restore key: `${{ runner.os }}-cargo-`
- 문제: `pull_request` run에서도 cache save 경로가 열려 있다.

## 메인테이너 결정 반영

### 도메인 전제

회귀 가드 1:1 추적성은 보존한다. CI 단축은 profile, cache, 병렬화 축에서만 추진하며 다음은 수용하지 않는다.

- 통합 테스트 파일 통합
- 회귀 가드 명명 규칙 변경
- `tests/golden_svg/issue-NNN/` 자산 구조 변경

따라서 #1664 변경은 cache restore/save 정책만 다루고, PR마다 162개 통합 테스트가 계속 실행되는 전제를
유지한다.

### 정책 판단 4건 답변

1. PR마다 `--release` + LTO 검증 유지 여부
   - 결정: PR은 `release-test` 중심으로 전환하고 release-grade 검증은 `devel` push / tag workflow로 이동한다.
   - #1664 적용: 본 작업에서는 profile을 바꾸지 않는다. 해당 결정은 #1666 범위로 분리한다.

2. PR cache restore-only, `devel` / `main`만 save
   - 결정: PR은 restore-only, trusted branch인 `devel` / `main`에서만 save한다.
   - #1664 적용: 이번 작업의 직접 구현 범위로 삼는다.

3. `Build & Test` 2-3개 job 병렬 분리
   - 결정: #1666 적용 후 효과를 재평가한 뒤 결정한다.
   - #1664 적용: 병렬 job 분리는 보류하며, workflow job 구조를 바꾸지 않는다.

4. `Swatinem/rust-cache` 검토
   - 결정: 검토는 허용하되 SHA pinning, 별도 PR, #1664 안정화 측정 선행이 필요하다.
   - #1664 적용: 현행 `actions/cache` 기반 정책을 먼저 정리한다. third-party action은 도입하지 않는다.

## 작업 범위

### 포함

- `.github/workflows/ci.yml`의 `Build & Test` cargo cache 정책 정리
- PR에서는 cache restore만 수행하도록 변경
- cache save는 `refs/heads/devel` 또는 `refs/heads/main`에서만 수행하도록 제한
- 기존 cache key와 restore key 의미 유지
- 변경 전후 측정 항목을 단계별 보고서와 최종 보고서에 기록

### 제외

- `Cargo.toml` profile 변경
- `cargo build` / `cargo test` 명령의 profile 변경
- `Build & Test` job 병렬 분리
- `Swatinem/rust-cache` 도입
- scheduled warm-cache workflow 신설
- 통합 테스트 파일 구조, 회귀 가드 이름, golden 자산 구조 변경

## 구현 방향

구체 구현 계획서는 별도 승인 단계에서 작성한다. 현재 수행계획 기준의 구현 방향은 다음과 같다.

1. 기존 `actions/cache@v5` 단일 step을 restore와 save step으로 분리한다.
2. restore step은 PR, push, manual run에서 모두 실행한다.
3. save step은 `github.ref == 'refs/heads/devel' || github.ref == 'refs/heads/main'` 조건에서만 실행한다.
4. save step은 exact cache hit가 아닐 때만 실행하는 조건을 검토해 동일 key 저장 충돌을 피한다.
5. wasm-build job은 PR에서 실행되지 않으므로 이번 #1664의 1차 변경 대상에서 제외한다.

## 측정 기준

각 sub-issue의 before/after 보고에 포함해야 하는 공통 측정 기준은 다음과 같다.

- PR checks 완료 시간 (P50, P90)
- `CI / Build & Test` job 시간
- 주요 step 시간
  - build
  - lib test
  - integration test
  - native-skia
- cache hit/miss/save 성공 여부
- cache 크기
- 실패 시 원인 가시성 (stderr line 위치 등)
- runner-minutes 변화
- branch protection / required check 변경 여부
- 회귀 가드 162개가 PR마다 모두 실행되는지 확인

마지막 항목은 단순 측정값이 아니라 도메인 제약이다. 1:1 회귀 가드가 PR마다 빠짐없이 실행된다는 것이 모든
변경의 전제다.

## #1664 추가 확인 항목

- PR run 로그에서 cache save 또는 reservation 시도가 사라졌는지 확인
- `devel` / `main` push run에서만 cache save가 시도되는지 확인
- `Cache reservation failed: You have reached your configured budget, your cache is now read only` 경고가
  사라졌는지 확인
- exact key restore 비율과 fallback restore 비율을 기록
- cache 총량 변화와 큰 PR ref cache 잔존 여부를 기록

## 리스크와 대응

- cache 삭제 또는 save 제한 직후 첫 run은 느려질 수 있다.
- PR별 두 번째 run에서 얻던 cache save 이득은 줄어들 수 있다.
- 기존 cache budget 자체가 부족하면 restore-only 정책만으로 장기 해결이 아닐 수 있다.
- 동일 cache key 저장 시 충돌 가능성이 있으므로 save 조건을 exact miss 중심으로 설계한다.
- 문제가 발생하면 restore/save 분리 조건만 되돌려 기존 `actions/cache@v5` 단일 step으로 롤백할 수 있다.
