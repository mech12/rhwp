- **브랜치**: `local/task1534` (base: `stream/devel`)

## 1. 배경 / 문제

HWPX 직렬화(저장) 시 폼 컨트롤(`<hp:checkBtn>`, `<hp:radioBtn>`, `<hp:btn>`,
`<hp:comboBox>`, `<hp:edit>`)의 **속성값**(`caption` 등)에 포함된 XML 특수문자
(`&`, `<`, `>`, `"`)가 **이중 이스케이프**되어, 저장할 때마다 손상이 누적된다.

검증 환경(`samples/hwpx/` 전수 53개)에서 `form-002.hwpx` 1건이 표면화:

| 단계 | XML caption 원문 | 디코딩된 실제값 |
|------|------------------|----------------|
| 원본 | `IP R&amp;&amp;D연계` | `IP R&&D연계` |
| 1회 저장 | `IP R&amp;amp;&amp;amp;D연계` | `IP R&amp;&amp;D연계` (손상) |
| 2회 저장 | `IP R&amp;amp;amp;&amp;amp;amp;D연계` | 한 겹 더 증가 |

본문 텍스트(`<hp:t>`)는 정상(`aift`, `[2027] 온새미로` 등 `&` 포함 본문 무손상).

## 2. 원인 분석 (코드 무수정 조사 완료)

| 측 | 위치 | 동작 |
|----|------|------|
| 파싱 | `src/parser/hwpx/utils.rs:21` `attr_str()` | `String::from_utf8_lossy(&attr.value)` — quick-xml 원문 바이트 그대로, **unescape 없음** → IR에 `R&amp;&amp;D` 저장 |
| 파싱(본문) | `src/parser/hwpx/section.rs` `read_text_content_with_tabs` | `Event::Text` + `Event::GeneralRef`(`decode_xml_general_ref`)로 **정상 디코딩** → 대칭 |
| 직렬화 | `src/serializer/hwpx/form.rs:98` `write_form` → `start_tag_attrs` | writer가 속성값을 **자동 escape** → `R&amp;&amp;D` 를 다시 escape → `R&amp;amp;&amp;amp;D` |

**근본 원인**: 속성값은 "파싱 시 unescape 안 함 / 직렬화 시 escape 함"의 비대칭.
IR에 저장되는 문자열이 **시맨틱 값이 아니라 부분 이스케이프된 원문**이라는 점이 SSOT 위반.

## 3. 영향 범위

- **직접 표면화**: 폼 컨트롤 `caption`(버튼류). `&`/`<`/`>`/`"` 포함 시 트리거.
- **잠재 동일 결함**: `attr_str`로 읽어 IR에 보관 후 다시 writer로 출력되는 모든
  **문자열 속성** — 예: `name`, `command`, `selectedValue`(comboBox), `groupName`,
  `radioGroupName`, `passwordChar` 등. 숫자/열거형 속성은 특수문자가 없어 무영향.
- 그 외 모듈(표/그림/도형 caption 등)도 동일 패턴 가능 — 조사 단계에서 전수 확인.

## 4. 해결 방향 (후보 + 권장)

| 안 | 내용 | 장점 | 단점/리스크 |
|----|------|------|------------|
| **A (권장)** | `attr_str`이 XML 엔티티를 **unescape**하도록 수정. IR은 항상 시맨틱 값, writer가 유일 escape 권위 | 근본·SSOT 정합. 모든 문자열 속성 일괄 해소 | 전역 영향 — verbatim 재출력 경로(예: `<hp:parameters>` `escape_xml_text(&attr_str())`) 및 golden/baseline IR 스냅샷 변동 점검 필수 |
| B | 폼 caption 등 **텍스트성 속성에 한정**해 파싱 시 unescape | 영향 국소화 | 누락 속성 재발 가능, 동일 결함 산발 |
| C | 직렬화 측에서 caption만 verbatim(escape 생략) | 변경 최소 | 진짜 특수문자 미이스케이프 위험 — 잘못된 방향, 비채택 |

**권장: A**. 단, 전역 변경이므로 ① verbatim 재출력 경로가 unescape된 값을 다시
escape하는지(대칭 유지) ② golden/baseline IR 스냅샷 변화가 "교정"인지 "회귀"인지를
구현 단계에서 건별 확인한다. 리스크가 크다고 판단되면 B로 폴백.

## 5. 검증 전략

본 결함은 **기존 회귀 게이트(`hwpx-roundtrip` ir-diff / baseline)가 잡지 못함**
(IR_DIFF 0으로 통과). 따라서 수정과 함께 **가드 보강**이 필수.

1. **단위/통합 테스트 신규**: 폼 caption에 `&`/`<`/`>`/`"` 포함 → parse→serialize→
   reparse 후 값 불변 + XML 원문 이중 이스케이프 없음 단언. (`form-002` 픽스처 또는 합성)
2. **누적 회귀**: 동일 파일 2회 roundtrip 후 caption 길이/값 불변 단언.
3. **전수 회귀**: `cargo test --test hwpx_roundtrip_baseline` 통과 유지.
4. **텍스트 일치 재검**: `samples/hwpx/` 전수 원본↔저장본 export-text 비교에서
   form-002 PASS로 전환, 기존 50건 무회귀(k-water-rfp는 별건).
5. **전체 테스트**: `cargo test`(--lib 아님) — 레이아웃/golden 무회귀 확인.
   (메모리 룰 `feedback_full_cargo_test_before_pr` 준수)

## 7. 리스크

- **전역 unescape 부작용**: 이미 escape된 값을 verbatim 재출력하는 경로가 있으면
  under-escape 발생 가능 → 사전 grep 감사로 차단.
- **golden/baseline 스냅샷 변동**: IR 텍스트가 바뀌므로 스냅샷 업데이트 필요 가능 —
  변경 1건씩 "교정 여부" 확인 후 반영.
- **HWP5 어댑터 경로**(#852 HWPX→HWP)도 동일 IR을 쓰므로 caption 변화가 HWP5
  직렬화에 영향 없는지 확인.
