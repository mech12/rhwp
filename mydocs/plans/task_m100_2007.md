## 2. 진단 (본 조사에서 확인·정정)

이슈 본문의 "자리차지 표 stacking·페이지 밖 overflow" 가설을 `dump-pages` 전수 계측으로 재검했다.

- 전 126쪽 중 **과적재(used>body_h) 10쪽뿐, 초과폭 +2~+7px** (미미). 대형 표가 페이지 밖으로 넘치는 stacking overflow는 **관측되지 않음**.
- 즉 −8쪽의 실제 기전은 "표가 페이지 밖으로 넘침"이 아니라 **RowBreak 표(+셀 콘텐츠) 높이가 한글 대비 근본적으로 짧게 측정**되어 문서 전체가 더 적은 페이지로 압축되는 것.
- 이는 이슈 본문이 병기한 **#1937/#1842 표 행높이 계열의 "과소" 축**과 동근. 독립 stacking 결함이 아니다.

## 5.5 RCA 확정 (오라클 실측 — 원 가설 반증)

시각 오라클 + render-tree + pi-page 국소화로 **원인을 확정**했다. **이슈 원 가설(표 행높이 과소)은 반증됨.**

**증거:**
1. **표 행높이 일치**: render-tree 표0(10행) = rhwp **852.5px** vs 한글 COM **852.4px** (일치). 한글 전체 표 행높이 합계(COM `get_row_height`, 136개) 확보(`com_tables_all.tsv`).
2. **텍스트 라인높이 일치**: page3 `used=576.3px, hwp_used≈576.3px, diff=+0.0px`.
3. **pi-page 국소화**(`verify_pi_page_vs_hangul.py`, 신뢰): 델타가 **분산 누적** — page3에서 이미 +1, 이후 ~14쪽마다 +1씩 −9. 발산점이 **큰 표가 아니라 heading/섹션 경계**(pi33-38=제1절 구역, pi62-72, pi129-130 …).

**결론**: −9는 표/라인 **높이 측정 오류가 아니라**, heading·섹션 구분 경계에서 **한글이 rhwp보다 한 요소 더 일찍 개행**하는 **페이지 분할 규칙 차이**. rhwp가 페이지를 약간 더 그리디하게 채운다(과적재 +2~7px 정황 일치). 기전 후보: **keep_with_next / keep_lines / 외톨이줄 보호 / heading 앞 분할** 규칙 미적용 — #1995 초기 관측("keep_with_next/keep_lines 페이지네이션 미준수", 커밋 d27a98b1)과 **동일 축**. pi36(제1절 heading, ps_id=17 heading 스타일)에 rhwp가 keep 규칙 미적용 정황.

**재분류**: #2007 = "표 스택/overflow" 아님 → **heading/keep 규칙 페이지 분할 갭**. 심각도 낮은 롱테일(−9/126, 분산), **수정은 전 문서 공통 페이지분할 경로 → 광역 회귀 위험 최상**(headings 있는 모든 문서에 쪽수 영향). 별도 정식 타스크 + 코퍼스 회귀게이트 필수.

## 6. 판단

−9는 심각도 낮은 롱테일(#1995 심각 −97은 #2002로 해소). ✅ 오라클 확보 + **RCA 확정(§5.5)**: 표 행높이 과소 **아님**(표0·텍스트 라인 일치), heading/섹션 경계 **페이지 분할 규칙 갭**(keep_with_next 계열). 원 가설 반증.

**다음 관문 = 별도 정식 타스크**: 본 이슈는 "표 스택"이 아니라 **keep/heading 페이지분할** 문제로 재분류되어야 하며, 수정은 전 문서 공통 페이지분할 경로(광역 회귀 위험 최상)라 코퍼스 pi-page 회귀게이트를 갖춘 독립 타스크가 필요하다. #1995 초기 "keep_with_next 미준수" 관측과 병합 조사 권장. 본 세션에서 무근거 hot-path 페이지분할 수정은 착수하지 않는다.
