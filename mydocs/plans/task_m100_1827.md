# Task #1827 수행계획서 — 쪽 상단 spacing_before 트림의 저장 vpos 증거 정합

## 배경

제출 PR 리뷰 visual sweep 산출물(pr_1751/1756 등)에서 페이지 전체가 한글 PDF 대비
수 pt 위로 이동해 overlay 이중상으로 보이는 후보 다수. 원인 분석 결과 줄 단위
누적이 아니라 **페이지 상수 오프셋**(쪽 상단 spacing_before 트림)이 지배 원인.

## 원인 (정량 — samples/task1750/split_guard_spacing_before.hwp)

- p2 전 줄 Δy ≈ −5.1~−4.4pt (누적 성분은 30줄 +0.7pt 뿐).
- p2 첫 문단 sb=700HU, **저장 첫줄 vpos=700HU** — 한글은 쪽 상단 앞 간격을 유지,
  typeset 회계도 반영(sb=9.3px)하는데 렌더러의 column-top 트림(#853,
  paragraph_layout.rs:1571~)만 지워 typeset↔render 불일치 + 페이지 전체 상향.
- 한컴의 유지/트림 결정은 저장 첫줄 vpos 에 남는다 (유지 시 vpos≈sb, 트림 시 0).

## 수정 방향

\#853 의 para_index==0 클램프(LINE_SEG.vpos 상한)를 쪽 상단 문단 전체로 일반화:
`0 < vpos0 ≤ spacing_before` 일 때만 vpos0 적용 (합성 seg 제외, 누적축 vpos 는
조건 미충족으로 자동 제외 → 종전 트림 유지).
