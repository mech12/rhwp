- 부모 이슈: #1668 `[Tracking/RFC] CI Build & Test 실행 시간 단계별 단축`
- 선행 완료: #1664, #1666, #1849
- 마일스톤: M100 / v1.0.0 조판 엔진 체계화
- 작업 브랜치: `task-1667-rust-cache-strategy`
- 작업 성격: CI cache 전략 감사, measurement 기준 정리, 후속 코드 변경 후보 분리

## 착수 판단

#1667은 바로 `Swatinem/rust-cache`를 도입하는 작업으로 시작하지 않는다.

선행 이슈에서 확인된 사실은 다음과 같다.

- #1664: `CI / Build & Test` cargo cache는 PR restore-only / trusted branch save 정책으로 안정화됐다.
- #1666: PR 경로의 `release` / `release-test` profile 불일치 비용은 줄었지만, exact cache hit 이후에도
  `Dirty rhwp` / `Compiling rhwp`가 남았다.
- #1849: `devel` push full `release --tests` 비용은 release smoke + `release-test` 전체 회귀로 줄었지만,
  `Dirty rhwp` 신호 자체는 남았다.
- #1667 코멘트에는 Build & Test 외에 CodeQL Rust cache, Render Diff cargo cache, PR ref cache cleanup
  운영 항목이 추가로 이관됐다.

따라서 #1667의 1차 목표는 "캐시 액션 교체"가 아니라, 다음 질문을 분리해 답하는 것이다.

1. 현행 `actions/cache`의 `target` 전체 저장이 `rhwp` local crate 재사용에 실제로 도움이 되는가?
2. `Dirty rhwp`는 cache miss가 아니라 Cargo fingerprint, checkout timestamp, feature/test target 조합 문제인가?
3. CodeQL Rust와 Render Diff cache도 #1664의 PR restore-only 원칙을 따라야 하는가?
4. Render Diff처럼 PR 중심 workflow에서 cache save를 끄면 cold build 비용이 커지는가, 아니면 quota 안정성이 더 중요한가?
5. `Swatinem/rust-cache`가 위 문제를 실제로 줄일 근거가 있는가?

## 이슈 / 코멘트에서 이관된 발견사항

### 1. 메인테이너 결정사항

#1668 메인테이너 코멘트 기준:

- `Swatinem/rust-cache` 검토는 허용된다.
- 단, 도입 시 commit SHA pinning이 필수다. tag-only pinning은 사용하지 않는다.
- #1664 안정화 측정 후에도 hit rate 또는 quota 문제가 남을 때 검토한다.
- third-party action 도입은 별도 PR로 분리한다.
- 각 sub-issue의 before/after 보고에는 PR checks 시간, `Build & Test` job 시간, step 시간,
  cache hit/miss/save, cache 크기, 실패 가시성, runner-minutes, required check 변경 여부,
  회귀 가드 1:1 추적성을 포함한다.

### 2. Build & Test cargo cache

현재 상태:

- workflow: `.github/workflows/ci.yml`
- restore: `actions/cache/restore@v5`
- save: `actions/cache/save@v5`
- save 조건: `push` + `refs/heads/devel` 또는 `refs/heads/main` + exact hit 아님
- path: `~/.cargo/registry`, `~/.cargo/git`, `target`
- key: `Linux-cargo-${Cargo.lock hash}`

선행 관측:

- #1664 이후 PR save skipped와 trusted branch save가 확인됐다.
- cache cleanup 후 exact key `Linux-cargo-6a1af...` 저장과 후속 exact-hit restore가 확인됐다.
- #1666/#1849에서도 exact hit는 계속 확인됐다.
- 그러나 exact hit 상태에서도 `Dirty rhwp` / `Compiling rhwp`는 남았다.

해석:

- Build & Test cargo cache 정책 자체는 실패로 보기 어렵다.
- 남은 문제는 cache hit/miss보다 Cargo fingerprint 실효성 문제일 가능성이 크다.
- `target` 전체 저장을 유지할지, registry/git 중심으로 줄일지, profile/feature별 key를 나눌지는
  `Dirty rhwp` 로그와 step 시간의 상관을 본 뒤 판단한다.

### 3. CodeQL Rust cache

현재 상태:

- workflow: `.github/workflows/codeql.yml`
- step: `Cache cargo registry & build (rust)`
- action: `actions/cache@v5`
- path: `~/.cargo/registry`, `~/.cargo/git`, `target`
- key: `Linux-codeql-rust-${Cargo.lock hash}`

이관 관측:

- #1702 최종 리뷰에서 CodeQL Rust analyze의 cache step은 PR run에서도 save post-step 표면이 남아 있다는
  open question이 있었다.
- #1702 merge 후 `devel` push run에서 CodeQL Rust cache는 fallback `Linux-codeql-rust-`에 hit했다.
- 당시 restore cache size는 317,394,514 B였고, `Build Rust (for CodeQL)`은 58.97s였다.
- cleanup 전에는 post-cache save가 cache budget read-only로 reservation 실패했다.
- 실패 위치는 Analyze (rust) log line 2262-2263으로 기록됐다.

판단:

- CodeQL Rust cache는 #1664의 Build & Test cache 정책 범위 밖이었다.
- 현재는 별도 key namespace를 쓰지만 단일 `actions/cache@v5`라 PR cache save가 발생할 수 있다.
- #1667의 1차 코드 변경은 #1664와 같은 restore/save 분리로 진행한다.
- 이때 구현 방식은 `actions/cache/restore`와 `actions/cache/save`의 명시적 분리이고, 정책 결과는
  PR restore-only / trusted branch save-only다.
- 단, Build & Test cache 변경과 한 PR에 섞지 않는 편이 좋다.

### 4. Render Diff cargo cache

현재 상태:

- workflow: `.github/workflows/render-diff.yml`
- trigger: `pull_request`, `workflow_dispatch`
- step: `Cache cargo registry & build`
- action: `actions/cache@v5`
- path: `~/.cargo/registry`, `~/.cargo/git`, `target`
- key: `Linux-render-diff-cargo-${Cargo.lock hash}`

이관 관측:

- `Linux-render-diff-cargo-*` cache가 `refs/pull/*`에 반복 생성된다.
- PR ref cache는 base branch나 다른 PR에서 재사용되지 않는다.
- closed PR ref cache는 merged PR과 closed-unmerged PR을 모두 cleanup 후보로 분류해야 한다.
- Render Diff는 PR 중심 workflow라 단순히 save를 끄면 warm source가 사라질 수 있다.

판단:

- cache quota 측면에서는 Render Diff cargo cache가 현재 가장 큰 누적 요인이다.
- 다만 PR 중심 workflow 특성상 restore-only 전환의 비용과 이득을 먼저 측정해야 한다.
- 후보는 다음처럼 나눈다.
  - 현행 유지 + closed PR ref cleanup 운영
  - PR save 차단 + cold build 비용 수용
  - `workflow_dispatch` on `devel` 등 trusted seed run을 명시적으로 두고 PR은 restore-only
  - path 축소: `target` 제외, registry/git 중심 cache

### 5. stale PR ref cleanup 운영

#1664 cleanup에서 확인한 원칙:

- 삭제 대상: closed `refs/pull/*` cache
  - merged PR 포함
  - closed-unmerged PR 포함
- 유지 대상: `refs/heads/devel`, `refs/heads/main`, OPEN PR cache
- 큰 cache부터 우선 검토: `Linux-render-diff-cargo-*`, `Linux-codeql-rust-*`, `Linux-cargo-*`, `node-cache-*`
- `devel` push CI 성공 확인은 closed PR ref cache 삭제의 필수 조건이 아니라, trusted branch cache
  save/restore가 정상인지 확인하는 measurement 단계로 분리한다.
- 실제 삭제 전 후보 목록과 위험도를 이슈 코멘트로 남긴다.

#1667에서는 cleanup을 즉시 자동화하지 않는다. 먼저 현재 누적 원인을 Build & Test / CodeQL / Render Diff /
node cache로 분리하고, cleanup은 필요한 경우 별도 운영 조치로 메인테이너 승인 후 진행한다. scheduled 또는
`pull_request.closed` 기반 자동화는 수동 cleanup 정책이 안정화된 뒤 별도 이슈/PR에서 검토한다.

## 진행 방향

### 원칙

1. `Swatinem/rust-cache`는 1차 도입하지 않는다.
2. Build & Test, CodeQL Rust, Render Diff cache를 같은 PR에 섞지 않는다.
3. cache policy 변경과 measurement 문서는 분리한다.
4. cleanup은 운영 조치이므로 삭제 전 후보 목록과 위험도 코멘트를 남기고 승인받는다.
5. 회귀 가드 구조, test 파일명, `tests/golden_svg/**` 구조는 변경하지 않는다.

### 1차 권장 작업

1차는 "감사 + 정책 계획 + 후보 분리"로 진행한다.

산출물:

- `mydocs/plans/task_m100_1667.md`: 수행계획서
- `mydocs/plans/task_m100_1667_impl.md`: 구현계획서
- 필요 시 `mydocs/report/task_m100_1667_measurement.md`: 원천 측정 문서

1차 코드 변경 후보는 구현계획서에서 승인받아 확정한다. 현재 판단으로는 다음 순서가 안전하다.

1. CodeQL Rust cache restore/save 분리
   - #1664와 유사한 trusted branch save-only 정책을 적용하기 쉽다.
   - `Analyze (rust)` check 표면을 유지한다.
   - PR cache save 표면을 줄일 수 있다.

2. Render Diff cargo cache 정책 재검토
   - quota 영향은 가장 크지만 PR 중심 workflow라 단순 restore-only 전환의 cold build 비용을 먼저 평가해야 한다.
   - 별도 PR로 다룬다.

3. Build & Test `target` cache 실효성 실험
   - exact hit 이후 `Dirty rhwp`가 남는 원인을 분석한다.
   - profile/feature/target별 key 분리 또는 `target` 제외 여부는 측정 후 결정한다.

4. `Swatinem/rust-cache` 도입 검토
   - 위 조정으로도 quota/hit/fingerprint 문제가 남을 때만 별도 PR에서 SHA-pinned로 검토한다.
