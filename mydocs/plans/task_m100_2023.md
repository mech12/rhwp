# 수행계획서 — Task M100 #2023: 프론트 웹 리팩터링 계획 수립

- 이슈: #2023 (umbrella: #2022)
- 브랜치: `local/task2023` (base: `upstream/devel` `a23b8ae1`)
- 작성일: 2026-07-07

## 1. 목표

**산출물은 "프론트 웹 리팩터링 계획"이다 — 리팩터링 실행이 아니다.** #1883에서 Rust
리팩터링 계획을 수립한 방식과 동일하게, 프론트 웹 계층을 SOLID와 복잡도 2축으로 재진단하고
실행 하위 이슈를 분리할 수 있는 계획서를 작성한다.

대상은 `@rhwp/editor` 하나가 아니라 `rhwp-studio`, `/web` 레거시, 브라우저 확장,
VS Code extension, npm 패키지 공개 계약을 포함한 프론트 웹 전체다. 계획서 작성 후
maintainer/collaborator 리뷰에 부치고, 승인 후 실행 이슈들을 분리한다.

## 2. 선행 완료 사항

- #2022 umbrella 이슈 생성: 프론트 웹 구조·계약·복잡도 리팩터링 추적.
- #2023 계획 수립 이슈 생성: SOLID·복잡도·`/web` 레거시·공개 계약 진단.
- 작업 브랜치 `local/task2023`를 최신 `upstream/devel` 기준으로 생성.
- 오늘 할일(`mydocs/orders/20260707.md`)에 #2023 항목 추가.
- 초기 조사에서 `/web`은 레거시 웹 앱과 현재 `web/fonts` 실사용 자산이 섞여 있음을 확인.
- 초기 조사에서 `rhwp-studio` 대형 모듈, 확장 CSP/security 계약, VS Code webview, npm package
  계약이 함께 검토되어야 함을 확인.

## 3. 수행 단계 (4단계)

### 1단계 — 공개 계약·금지 목록 인벤토리

프론트 리팩터링에서 보존해야 하는 public/distribution/security contract를 먼저 문서화한다.

- `@rhwp/editor`: iframe 기반 embed 계약, 런타임 dependency 없음, postMessage 경계.
- `@rhwp/core`: WASM 패키지 API와 README 예제 계약.
- `rhwp-studio`: public asset 경로, font loader/substitution, PWA/빌드 계약.
- 브라우저 확장: CSP, inline script 금지, `web_accessible_resources`, sender/URL/file 검증.
- VS Code extension: webview CSP, WASM/font asset 경로, CDN/font fallback 계약.
- `/web`: 레거시 JS/HTML/CSS와 `web/fonts` 실사용 자산의 경계.

산출: `mydocs/tech/investigations/issue-2023/task_m100_2023_frontend_contract_guardrails.md`

### 2단계 — 현황 재진단 (SOLID + 복잡도)

`mydocs/manual/solid_scoring_guide.md`를 프론트 구조에 맞게 재해석하고 정량 baseline을 남긴다.

- **SOLID**: `InputHandler`, dialog, `diff-engine`, `wasm-bridge`, `core/types.ts`, EventBus,
  extension shared boundary, `/web` asset ownership을 S/O/L/I/D 관점으로 평가.
- **복잡도**: 파일 LOC, 함수 LOC, CC>25 함수, `any`/`this: any`, export/public surface,
  브라우저별 확장 코드 중복, legacy `/web` 실사용 경계를 실측.
- **비교 기준**: `mydocs/feedback/rhwp-studio-code-review.md`는 2026-02 기준 baseline으로만
  참고하고, 현재 수치는 `upstream/devel` 기준으로 재측정한다.

산출: `mydocs/tech/investigations/issue-2023/task_m100_2023_frontend_diagnosis.md`

### 3단계 — 프론트 리팩터링 계획서 초안

1·2단계 결과를 바탕으로 실행 전용 계획을 작성한다.

- `/web` 정리 순서: `web/fonts` canonical 이전 → 빌드/문서 갱신 → 레거시 삭제 검토.
- `rhwp-studio` 해체 우선순위: InputHandler, dialog, diff-engine, WasmBridge 등.
- 확장 공통화 우선순위: Chrome/Firefox/Safari 차이와 `rhwp-shared` 확대 가능성.
- 금지 목록: runtime UI framework 도입, CSP 완화, `web_accessible_resources` 확대,
  font license 위반, 공개 API/iframe 계약 변경 등.
- 실행 하위 이슈 분리안과 각 이슈의 성공 기준.

산출: `mydocs/plans/task_m100_2023_frontend_refactoring_plan.md`

### 4단계 — 리뷰 게시 + 마무리

- 계획 요약을 #2023에 게시하고 maintainer/collaborator 리뷰를 요청한다.
- 리뷰 요청 항목:
  - #1883과 동일한 SOLID + 복잡도 2축 적용이 적절한지.
  - 프론트용 SOLID 재해석이 타당한지.
  - `/web` 레거시 정리와 `web/fonts` canonical 이전을 분리하는 방향이 맞는지.
  - `@rhwp/editor` iframe/무의존 계약을 public contract로 고정할지.
  - React/Vue/Svelte 등 런타임 UI 프레임워크 도입 금지를 명시할지.
  - 확장 CSP/manifest/security guardrail을 금지 목록에 포함할지.
  - 실행 하위 이슈 분리 순서가 적절한지.
- 최종 보고서 작성 후 다음 단계 승인을 요청한다.

## 4. 검증 계획

- 코드 무변경(문서·진단 전용) — 빌드/테스트 게이트는 수행하지 않는다.
- 진단 문서에는 재현 가능한 스캔 명령과 기준 커밋을 남긴다.
- 공개 이슈에 게시할 요약은 내부 문서 경로·계약·금지 목록 표현을 점검한 뒤 올린다.
- `git diff --stat`과 문서 링크 확인으로 산출물 범위를 검증한다.

## 5. 범위 제외

- 리팩터링 실행 코드 변경 일체.

- npm 패키지 API 변경.
