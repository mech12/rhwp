# 수행계획서 — #1950 HWP3→HWP5 변환 후 렌더 이탈 (char-offset 드리프트)

- 이슈: edwardkim/rhwp#1950 (M100)
- 브랜치: `local/task1950` (#1937·#1949 포함 상태 기준)
- 유형: HWP3→HWP5 변환(직렬화) 시 char 슬롯/offset 정보 왜곡 — 시각 충실도
- 작성일: 2026-07-06

## 1. 문제 정의

HWP3 문서를 HWP5 로 변환 저장(`render-diff --via hwp`) 후 렌더가 원본 렌더와 어긋남
(20k 서베이 7건, hwp3 표본 65건 중 10.8% 비-PASS). 최대 376px 변위, 1건 페이지 1→2 증가.

- HWP3→IR→렌더(원본)는 정상. **IR→HWP5 직렬화→재파스→렌더**에서 이탈.
- 대표: 2955331(OVER 376px, TextLine8/TextRun±1), 15047877(PAGE 1→2, node 106→104).

## 2. 착수 전 조사 (코드 무수정) — 근본 원인 확인

`convert` + `ir-diff`(2955331) 실측:

- 문단 0: **`cc A=16 vs B=33`**, `controls A=2 vs B=3`. B 가 **secd(SectionDef) 를 위치 0
  에 삽입**(ctrl[0] form→secd, form/pghd 뒤로 밀림) — **#1915(HWP3-origin page_def 보존)**
  에서 온 삽입.
- `char_offsets[1]: 1→8`, `char_offsets[17]: 17→24`, 다수 문단 cc 발산
  (A=1 vs B=10, A=31 vs B=88 …). 드리프트 문단(TextLine8)은 cc 31→88.

→ **원인 축 2개**:
1. **HWP3-origin IR 의 컨트롤 char 슬롯 폭이 HWP5 와 다름**(HWP3 유래는 좁고, HWP5 직렬화·
   재파스는 8유닛 확장) → char_shape/offset 경계가 밀려 run 분할·줄 배치가 어긋남.
2. **#1915 secd 삽입**(8유닛 슬롯)이 문단 0 이후 전 char offset 를 추가로 밀어 가중.

## 3. 미해결 질문 (Stage 1 대상)

- 두 축(기본 슬롯 폭 vs #1915 secd) 중 지배 요인, 각 기여도(secd 없는 문단도 드리프트?).
- 올바른 char 모델 기준: HWP3 원본 렌더가 정답(4유닛 전제)인가, HWP5 8유닛으로 정규화가
  정답인가. → **한글 HWP3→HWP5 변환 오라클**(한글 편집기 변환본)로 char_offset 정답 확인.
- 수정 위치: HWP3 파서(4→8 슬롯 정규화, IR 통일) vs HWP5 직렬화 경로(변환 시 offset 재계산).
  CLAUDE.md: **HWP3 전용 로직은 parser/hwp3 안에서만**.

## 4. 수행 접근

원본 HWP3 렌더는 정답이므로, **변환(직렬화) 후 char_offset/슬롯이 원본과 시각 정합**
하도록 만든다. 오라클로 정답 char 모델을 확정한 뒤 최소 수정.

## 5. 단계 구성 (구현계획서에서 상세화)

- **Stage 1 — 원인 분해 + 오라클 정답 확정**: secd 기여 vs 기본 슬롯 폭 기여 분리
  (예: #1915 secd 임시 비활성 렌더 비교). 한글 변환 오라클(가능 시)·원본 렌더 기준으로
  char_offset 정답 모델 확정. 지배 요인·수정 위치 결정.
- **Stage 2 — 수정 설계**: char 슬롯/offset 정합안(파서 정규화 또는 직렬화 재계산).
  #1915 secd 삽입과의 상호작용 재검토(슬롯 반영 or page_def 무슬롯 전달). 부작용 평가.
- **Stage 3 — 구현 + 게이트**: 7건 render-diff --via hwp PASS 목표(±1px, 페이지 정합),
  hwp3 표본 회귀(비-PASS 10.8%→목표), hwp5/hwpx 무회귀, #1915 page_def 보존 유지.
