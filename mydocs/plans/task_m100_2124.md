## 1. 목적

이 작업의 목적은 프론트 웹 리팩터링을 바로 시작하는 것이 아니라, 이후 Phase A/B 리팩터링을
비교할 공식 기준선을 먼저 고정하는 것이다.

고정 대상은 다음 네 가지다.

1. TypeScript/JavaScript 복잡도 측정 도구와 산식.
2. 공식 측정 모집단과 제외군.
3. 공개 계약, 보안 계약, 폰트 계약, smoke gate.
4. `solid_scoring_guide.md`를 구체 review unit에 적용하기 위한 SOLID evidence와 채점 절차.

Phase 0 산출물은 후속 리팩터링의 성공 여부를 판단하는 기준선이다. 따라서 이 이슈에서는 대형
모듈 해체나 폴더 이동을 수행하지 않는다.

## 3. 공식 측정 모집단과 제외군

위해 별도 그룹으로 측정하되, `rhwp-studio` runtime 개선 목표에는 섞지 않는다.

## 5. 확정 금지 목록

이 목록은 후보가 아니라 #2023 v2와 #2124 기준의 확정 guardrail이다.

- 이 이슈에서 `InputHandler`, dialog, `WasmBridge`, `diff-engine` 해체를 시작하지 않는다.
- Phase 0 기준선 승인 전 complexity fail gate로 PR을 막지 않는다.
- 리팩터링과 기능/보안 변경을 같은 PR에 혼합하지 않는다.
- `@rhwp/editor` iframe/postMessage/public embed 계약을 변경하지 않는다.
- `@rhwp/editor` runtime dependency를 추가하지 않는다.
- `@rhwp/editor` package에 runtime dependency를 추가하지 않는다.
- #2023 v2 승인대로 이번 프론트 리팩터링 전체에는 React/Vue/Svelte 등 runtime UI framework를
  도입하지 않는다(v1.0까지). 이후 필요성은 별도 RFC/umbrella에서 재론한다.
- 확장 CSP를 완화하지 않는다.
- inline script를 추가하지 않는다.
- `web_accessible_resources`를 확대하지 않는다.
- sender/URL/file signature/size 검증을 약화하지 않는다.
- `publicDir: false` 규칙을 우회하지 않는다.
- Chrome/Firefox `build.mjs`의 명시적 copy 계약을 암묵 `publicDir` 동작으로 대체하지 않는다.
- font fallback을 변경하지 않는다.
- `/web` legacy를 삭제하지 않는다.

`postMessage('*')`와 origin/source 미검증 같은 현재 구현은 호환 계약 자체가 아니라 알려진 보안 부채다.
#2124에서는 변경하지 않지만, 보안 강화를 영구 금지하지 않으며 별도 이슈와 migration 검토 대상으로 둔다.

## 6. 실행 원칙

1. **기준선 우선**: 예비 진단 수치는 참고만 하고, #2124에서 리뷰 승인된 snapshot만 공식 baseline으로 둔다.
2. **advisory 우선**: Phase 0에서는 complexity 결과를 fail gate로 쓰지 않는다.
3. **계약 불변**: public API와 message envelope, VS Code webview, extension manifest/CSP, font fallback은
   변경하지 않는다. 현재 `targetOrigin: '*'`는 영구 계약이 아닌 security debt로 기록한다.
4. **모집단 명시**: include/exclude, generated glue, dist, tests/e2e, font binaries, legacy `/web` 그룹의 처리 방식을 snapshot에 기록한다.
5. **표면 분리**: runtime source, legacy `/web`, extension, VS Code, npm wrapper를 한 숫자로 섞지 않고 그룹을 나눈다.
6. **3-browser sync**: Chrome/Firefox/Safari 중 하나의 extension 코드나 asset copy 계약을 건드리면
   나머지 두 브라우저와 `rhwp-shared` 계약 영향을 확인한다.
7. **후속 단계 분리**: Phase A `assets/fonts`는 #2124 승인·종료 뒤 #2125에서 진행하고, Phase B 구조
   리팩터링은 #2124/#2125 완료 뒤 별도 판단한다.
8. **총량 감소 판정**: hotspot 분포 개선과 총량 감소를 구분한다. 통이동은 준비 단계일 뿐 개선 성과로
   계상하지 않는다.
9. **과도기 예외 심사**: CC>25 +1~2는 자동 허용치가 아니다. hotspot Max/LOC 감소, 단일 책임,
   gate 통과, Total CC·CC>25 sum 공개, 다음 anchor와 해소 조건을 모두 갖춘 예외만 reviewer가 승인한다.

## 9. 예상 단계

구현 계획서는 다음 5단계 안에서 구체화한다.

1. **측정 도구·모집단 확정**: 프론트 metrics 실행 방식, include/exclude, schema v2와 diff를 고정한다.
2. **metrics snapshot 작성**: 공식 기준 커밋에서 분포·총량/LOC/type/export/font hash를 수집한다.
3. **계약 snapshot 작성**: public API, WASM JSON boundary, extension/VS Code message와 보안 표면을 문서화한다.
4. **font·smoke·SOLID evidence 작성**: `web/fonts` inventory, smoke manifest, frontend SOLID evidence 부록을 작성한다.
5. **종료 정리**: #2124 체크리스트 근거, 단계 보고서, 최종 보고서, 후속 Phase A/B 판단을 정리한다.
