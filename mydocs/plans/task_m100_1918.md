# 수행 계획서 - Task #1918

**이슈**: #1918 표 입력/삭제 시 워터마크·정적 이미지 레이어 포함 페이지가 매 입력마다 고비용 재렌더링됨
**브랜치**: `local/task1918`
**마일스톤**: M100 / v1.0.0 editing foundation stage
**작성일**: 2026-07-05

---

## 1. 배경

`rhwp-studio`에서 표 내부 셀에 텍스트를 입력하거나 삭제할 때, 워터마크나 큰 이미지성 배경이 포함된
문서에서 입력 반응이 현저히 느려진다. 최초 재현은 `samples/복학원서.hwp`의 표 입력 지연이었으나,
추가 분석 결과 특정 샘플의 표 조판 문제가 아니라 **표 편집 후 페이지 전체 재렌더링이 발생하면서
변하지 않은 정적 이미지 레이어까지 매 입력마다 다시 그리는 성능 문제**로 일반화된다.

문제는 "워터마크 플래그가 있는 그림 컨트롤"에만 한정되지 않는다. 실제 샘플에는 표/셀 `BorderFill`
이미지 채움으로 워터마크성 배경을 구성한 문서와, PNG/JPG/OLE 같은 정적 객체가 같은 페이지에 있는
문서가 포함된다.

## 2. 현상과 근거

### 2.1 샘플별 구조

| 샘플 | 구조 | 렌더 특성 |
|------|------|-----------|
| `samples/복학원서.hwp` | 표와 같은 페이지에 큰 뒤쪽 그림 워터마크. JPG 약 85KB, 약 131mm x 131mm, GrayScale/brightness/contrast 효과 | render 약 681.8ms |
| `samples/253E164F57A1BC6934-empty.hwp` | 2쪽 문서. 각 페이지가 1x1 표 셀의 이미지 채움으로 구성. JPG 약 107KB/253KB, `FitToSize`, brightness/contrast 효과 | render 약 1124.3ms |
| `samples/143E433F503322BD33.hwp` | 2단 문서. 표, JPG 그림, OLE 객체, PNG 이미지 채움이 함께 존재 | render 약 197.7ms |
| `samples/table-001.hwp` | 일반 표 중심, 큰 이미지성 정적 레이어 없음 | render 약 3.2ms |

### 2.2 기준 측정

```text
target/debug/rhwp bench \
  samples/253E164F57A1BC6934-empty.hwp \
  samples/143E433F503322BD33.hwp \
  samples/복학원서.hwp \
  samples/table-001.hwp \
  -n 5
```

```text

복학원서.hwp:                   size 112.0 KB, pages 1, parse 4.3ms, layout 1.2ms, render 681.8ms,  serialize 33.9ms, total 721.2ms
```

`samples/복학원서.hwp`의 문제 셀을 대상으로 별도 편집 벤치를 수행했을 때, 셀 텍스트 편집 명령은
약 `0.61ms`였고 편집 후 렌더링은 약 `681ms`였다. 따라서 1차 병목은 표 내부 문자열 편집이 아니라
편집 후 페이지 갱신 경로에 있다.

## 3. 원인 후보

현재 편집 경로는 표 내부 텍스트만 바뀌어도 페이지 단위 invalidation과 전체 페이지 렌더링을 수행한다.
그 결과 변하지 않은 워터마크/배경 이미지/정적 객체 레이어의 이미지 변환, 필터 적용, base64 직렬화,
JSON 파싱 비용이 반복될 수 있다.

확인한 주요 경로:

- `rhwp-studio/src/engine/input-handler.ts`
  - `afterPageLocalEdit` 이후 `document-page-invalidated` 발생
- `rhwp-studio/src/view/canvas-view.ts`
  - `refreshInvalidatedPage`에서 페이지 캔버스 갱신
- `rhwp-studio/src/view/page-renderer.ts`
  - flow 렌더 후 background/behind/front overlay 레이어 적용
  - 레이어별 `renderPageToCanvasFiltered` 호출
  - `getLayerPlaneSummary`에서 page layer tree 조회 및 JSON parse
- `src/wasm_api.rs`
  - `render_page_to_canvas_filtered`
  - `build_page_layer_tree`
- `src/paint/json.rs`
  - 이미지 op 직렬화에서 base64 payload 포함 가능
- `src/renderer/web_canvas.rs`
  - 이미지 변환, 필터, opacity 처리

## 4. 목표

1. 표 텍스트 입력/삭제 시, 문서 구조와 정적 이미지 레이어가 변하지 않는 경우 정적 레이어를 재사용하여
   체감 지연을 줄인다.
2. 그림 컨트롤 워터마크뿐 아니라 표/셀 `BorderFill` 이미지 채움, 배경 이미지, OLE/PNG 기반 정적 객체까지
   포함하는 일반화된 개선으로 설계한다.
3. 렌더링 결과의 시각 출력과 편집 정확성을 유지한다.
4. 문제 샘플 3종과 비교 샘플 1종으로 성능 전후를 재측정하고, 회귀 방지 기준을 남긴다.

## 6. 수정 방향

구현계획서에서 확정할 후보는 다음과 같다.

1. **정적 레이어 캔버스 재사용**
   - 페이지 크기, zoom, layer kind, layer summary, 이미지 효과, BinData 참조 등이 변하지 않으면
     background/behind/front 계열 레이어를 재사용한다.
   - 텍스트 flow만 바뀌는 편집에서는 정적 레이어 재렌더링을 피한다.

2. **레이어 시그니처 도입**
   - `Picture` 컨트롤뿐 아니라 `BorderFill` 이미지 채움, OLE, PNG/JPG 정적 객체를 포함하는
     시그니처를 정의한다.
   - 시그니처가 변하지 않는 동안에는 고비용 렌더/직렬화 경로를 생략한다.

3. **이미지 변환/직렬화 캐시**
   - BinData ID, 크기, 채움 모드, crop, effect, brightness, contrast, opacity 등을 캐시 키 후보로 검토한다.
   - base64 payload와 필터 적용 결과가 매 입력마다 새로 생성되지 않도록 한다.

4. **요약 API 비용 축소**
   - `getPageLayerTree`/summary 조회가 큰 JSON 또는 이미지 payload를 동반하는 경우, summary 전용 경량 API
     또는 payload 제외 구조를 검토한다.

## 7. 단계 개요

구현계획서에서 최소 3단계, 최대 6단계로 확정한다. 수행계획 기준의 예상 단계는 다음과 같다.

1. **재현 계측 고정**
   - 문제 샘플 3종과 비교 샘플 1종의 현재 렌더/편집 비용을 재측정한다.
   - `rhwp-studio` 편집 경로에서 어느 함수가 실제 시간을 쓰는지 브라우저/콘솔 계측 지점을 확정한다.

2. **정적 레이어 식별과 캐시 설계**
   - 페이지/레이어 단위로 재사용 가능한 조건을 정의한다.
   - `Picture` 워터마크와 `BorderFill` 이미지 채움이 모두 같은 정책에 들어오는지 확인한다.

3. **최소 구현**
   - 텍스트 편집 후 page-local invalidation에서 정적 레이어를 보존하는 경로를 추가한다.
   - 캐시 무효화 조건을 보수적으로 둔다.

4. **성능 검증**
   - `samples/복학원서.hwp`, `samples/253E164F57A1BC6934-empty.hwp`,
     `samples/143E433F503322BD33.hwp`에서 입력/삭제 전후 시간을 비교한다.
   - `samples/table-001.hwp`에서 일반 표 편집 무회귀를 확인한다.

5. **회귀 검증과 보고**
   - 필요한 Rust/TypeScript 테스트를 추가 또는 갱신한다.
   - `cargo test`와 관련 `rhwp-studio` 검증을 수행한다.
   - 단계별 보고서와 최종 보고서에 측정 결과를 기록한다.

## 8. 리스크와 가드레일

- 정적 레이어를 잘못 재사용하면 이미지 위치, z-order, 선택/커서 표시, 편집 후 페이지 이동이 stale 상태로
  남을 수 있다. 캐시 조건은 처음에는 보수적으로 잡는다.
- 표 편집이 페이지 수나 레이아웃을 바꾸는 경우에는 정적 레이어를 재사용하지 않아야 한다.
- zoom 변경, 페이지 크기 변경, 스크롤 가상화, 레이어 표시 옵션 변경 시 캐시를 무효화해야 한다.
- `BorderFill` 이미지 채움은 그림 컨트롤과 다른 경로로 렌더링될 수 있으므로, 그림 컨트롤만 기준으로
  최적화하지 않는다.
- 이미지 base64 payload를 캐시할 경우 메모리 증가가 발생할 수 있으므로 LRU 또는 페이지 생명주기 기반 해제를
  함께 검토한다.

## 9. 검증 기준

- 기능:
  - 세 문제 샘플에서 표 입력/삭제가 정상 동작한다.
  - 워터마크/배경 이미지/정적 객체의 시각 출력이 편집 전후 유지된다.
  - 일반 표 편집과 기존 페이지 렌더링 경로가 회귀하지 않는다.

- 성능:
  - 문제 샘플에서 입력/삭제 후 재렌더 비용이 기준 대비 유의미하게 감소한다.
  - 정적 이미지 레이어가 없는 `samples/table-001.hwp`는 성능 저하가 없어야 한다.

- 자동 검증:
  - 관련 Rust 테스트 또는 벤치 명령
  - 관련 `rhwp-studio` 테스트 또는 수동 계측 절차
  - `cargo test` 기본 회귀
