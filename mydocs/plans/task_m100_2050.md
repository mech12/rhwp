# 수행계획서 — task_m100_2050

- **이슈**: #2050 [vscode] 뷰어 좌측 사이드바 — 페이지 썸네일 + 목차(개요) + 북마크 바로가기
- **브랜치**: `local/task2050` (분기 기준: `upstream/devel`)
- **마일스톤**: M100 (v1.0.0)

## 1. 목표

PDF 뷰어 확장처럼 rhwp-vscode 뷰어 **좌측에 네비게이션 사이드바**를 추가한다. 세 개 탭/섹션으로 구성:

1. **썸네일** — 페이지 축소 렌더 목록, 클릭 시 해당 페이지 이동, 현재 페이지 하이라이트
2. **목차(개요)** — 문서 개요/조문 계층 트리, 클릭 시 해당 위치 페이지 이동
3. **북마크** — 사용자 북마크 목록, 클릭 시 이동

### 레이아웃(안)
```
┌──────────┬─────────────────────────────┐
│ 사이드바  │  편집/뷰 영역 (#scroll-container)│
│ (280px)  │                             │
│ [썸네일] │   ┌───────────┐             │
│ [목차]   │   │  page 1   │             │
│ [북마크] │   └───────────┘             │
│ ┌──────┐ │   ┌───────────┐             │
│ │ p1 ▓ │ │   │  page 2   │             │
│ │ p2   │ │   └───────────┘             │
│ │ p3   │ │                             │
│ └──────┘ │                             │
│  ◀ 토글  │                             │
└──────────┴─────────────────────────────┘
```
- 사이드바 상단 탭 전환(썸네일/목차/북마크), 접기 토글 버튼.
- 폭 기본 280px, 접기 가능.

## 2. 기술 현황 (조사 완료)

| 요소 | 데이터 소스 | 상태 |
|------|------------|------|
| 썸네일 | `renderPageToCanvas(page, canvas, scale)` | 기존 API, TS만 |
| 목차 | `build_structure(doc, mode)` (`src/document_core/queries/structure.rs`) — 개요/조문 계층 트리, `section`·`paragraph` 포함 | **WASM 미노출** → 바인딩 추가 필요 |
| 북마크 | `getBookmarks()` | 기존 노출 |
| 위치→페이지 | `getPageOfPosition(section, para)` | 기존 노출 |

## 3. 작업 단계 (Phase)

### Phase A — Rust: `getStructure` WASM 바인딩 + WASM 재빌드
- `src/wasm_api.rs`에 `getStructure(mode: &str) -> String` 추가. 내부에서 `core`의 `build_structure_native(mode)` 호출(신규 얇은 래퍼) → `build_structure` 재사용, JSON 반환.
- 네이티브 `cargo build`/`cargo test`로 바인딩 컴파일 검증.
- Docker로 WASM 재빌드 → `pkg/` 갱신.
- **범위 한정**: 파서/렌더/레이아웃 무변경. 읽기 전용 질의 노출만.

### Phase B — vscode: 사이드바 레이아웃 + 썸네일
- `hwp-editor-provider.ts` HTML/CSS에 사이드바 컨테이너(`#nav-sidebar`) + 탭 구조 추가.
- `viewer.ts`에 썸네일 생성(작은 scale로 `renderPageToCanvas`) + 클릭 스크롤 + 현재 페이지 하이라이트.
- 편집 영역과 사이드바 flex 레이아웃, 접기 토글.

### Phase C — vscode: 목차 + 북마크 패널 + 네비게이션
- 로드 시 `getStructure("auto")` 호출 → 트리 렌더, 클릭 → `getPageOfPosition` → 해당 페이지 스크롤.
- `getBookmarks()` → 목록 렌더, 클릭 → 이동.
- 빈 문서(개요/북마크 없음) 시 안내 문구.

## 4. 의존성 / 순서

- Phase A(WASM 바인딩+재빌드)가 **선행**되어야 Phase C(목차)가 pkg의 새 API를 쓸 수 있다.
- 썸네일/북마크(Phase B, 북마크)는 기존 API라 A 없이도 가능.
- Rust 변경(Phase A)과 vscode 변경(B/C)은 서로 다른 크레이트 → **PR 분리 고려** 필요(아래).

## 5. PR 전략 (검토 필요)

두 가지 안:
- **안 1 (단일 PR)**: Phase A~C 한 브랜치·한 PR. pkg 재빌드 산출물 포함. 리뷰 범위 큼.
- **안 2 (분리 PR)**: (a) Rust `getStructure` 바인딩 PR, (b) vscode 사이드바 PR. 의존 순서상 (a) 먼저 merge 후 (b). 리뷰 명확하나 2회.

→ **구현계획서 승인 시점에 확정** (기본 제안: 안 2, Rust 코어 변경과 UI 변경 분리).

## 6. 리스크

- **썸네일 성능**: 페이지 많은 문서에서 전체 썸네일 즉시 렌더 시 비용 큼 → **지연 렌더(IntersectionObserver)** 필요.
- **목차 정확도**: `build_structure`의 `auto` 모드는 개요 스타일 우선, 없으면 조문 패턴. 일반 문서에서 개요 미사용 시 목차가 비거나 조문 패턴 오검출 가능 → 빈 목차 처리 및 mode 선택 옵션 고려.
- **WASM 재빌드**: Docker 환경 필요(CLAUDE.md 규칙: WASM은 Docker 전용).
- 회귀: 편집 영역 레이아웃 flex 전환이 기존 스크롤/줌 좌표 계산에 영향 줄 수 있음 → 사이드바를 편집 영역 바깥 래퍼에 배치해 `#scroll-container` 내부 좌표계 불변 유지.

## 7. 검증

- Rust: `cargo build`/`cargo test` 통과, `getStructure` JSON 스키마 확인.
- WASM: Docker 재빌드 성공, pkg 갱신.
- vscode: webpack 빌드 통과, 실기기에서 썸네일/목차/북마크 클릭 이동 동작(작업지시자 시각 판정).
