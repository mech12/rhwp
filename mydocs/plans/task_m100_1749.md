# Task #1749 수행계획서 — 단일줄 fit 의 저장 LINE_SEG 신뢰가 누적높이 초과를 우회 (overfill)

## 배경
hwpdocs 페이지·PI 대조 S1(overfill) 원인 규명(#1745 후속 조사)에서 확정. 대표:
`36371084_결재문서본문_2026년 상반기 중대산업재해 발생 대비 비상조치 훈련계획 알림.hwpx` —
1쪽 used 1011.8px > 본문 990.2px, 꼬리 공백 문단 pi18(" ")을 1쪽에 수용. 한글(OLE)은 2쪽.

## 원인 (fit 트레이스 실증 — 이슈 #1749)
`typeset_paragraph` 의 `saved_single_line_bottom_fits`(fe6de3ef 도입, 각주 예약 계열):
- pi18: `cur_h=981.4 h_fit=17.3 avail=986.2 normal_fits=false` — 누적높이로는 탈락.
- 저장 LINE_SEG bounds(top 968.3, bottom 985.7px ≤ avail+0.5)로 통과 → 배치 → overfill.
- 이 문서는 **누적좌표**(다음 pi19 저장 vpos=74902 가 리셋 없이 본문높이 74265HU 를 초과)라
  저장 vpos 가 페이지 배정을 인코딩하지 않음 — bounds 신뢰의 전제가 성립 안 함.

## 도입 커밋의 보호 케이스 분석
fe6de3ef 합성 테스트(`page_bottom_text_box_fit_keeps_line_even_when_advance_overflows`,
Paginator 엔진 경로): **문서 마지막 문단**의 줄이 저장 좌표상 페이지 하단에 정확히 닿고
트레일링 line_spacing 만 advance 를 초과 — 이때 줄 유지가 정답. 이 케이스는 "다음 실줄이
없다"는 특성으로 결함 케이스와 구분 가능.

## 수정 방향
`saved_single_line_bottom_fits` 에 **저장 flow 의 페이지-마지막 증거** 조건 추가:
다음 문단의 첫 실줄(synthetic 제외)이 (a) 없거나, (b) 현재 줄보다 작은 vpos 로 리셋되는
경우에만 bounds 신뢰. 누적좌표 문서(다음 vpos 증가 지속)는 신뢰 비활성 → 누적높이 판정.
- typeset(메인 파이프라인)만 수정. Paginator(engine.rs) 합성 테스트 보호 케이스는 무변경.
