# Task #2164 수행 계획서

## 1. 작업 개요

- 이슈: [#2164](https://github.com/edwardkim/rhwp/issues/2164)
- 제목: Chrome 확장 v0.2.8에서 문단 부호 표시 후 표 셀 입력 시 Enter 문단 겹침
- 작업 브랜치: `codex/task_m100_2164`
- 기준 브랜치: `upstream/devel`
- 재현 원본: `samples/issue2164/의견제출서(양식).hwp`
- 원본 SHA-256: `a8f1874850d9970ba2f9cff868ada599081c349d71fe3c4a3e7a1023e399c98c`

## 2. 제보 환경과 증상

- Windows 11 25H2 빌드 26200.8737
- Chrome 150.0.7871.115 64비트
- rhwp Chrome 확장 v0.2.8
- `의견제출 요지` 표 셀에 내용을 입력하고 Enter를 누르면 새 문단이 아래로 전진하지 않고 기존 내용과 겹친다.
- 문단 부호를 표시한 상태에서 제보되었으며, 한컴오피스 한글 2024에서는 같은 동작이 정상이다.

## 3. 사전 코드 대조

현재 셀 Enter 경로는 다음과 같다.

1. `rhwp-studio/src/engine/command.ts`의 `SplitParagraphInCellCommand`
2. `src/wasm_api.rs`의 `splitParagraphInCell`
3. `src/document_core/commands/text_editing.rs`의 `split_paragraph_in_cell_native`
4. 분할 전·후 셀 문단 `reflow_cell_paragraph`
5. 표 dirty 처리와 section 재페이지네이션

짧은 셀 텍스트 입력은 page-local 렌더와 지연 페이지네이션 경로를 사용한다. Enter는 구조 변경이므로 full refresh 경로를 타며, full refresh 직전에 지연 페이지네이션을 flush한다. 따라서 다음 세 축을 분리해서 확인한다.

- Enter 직전 지연 페이지네이션 상태가 문단 분할과 충돌하는지
- 분할 후 다문단 셀의 행 높이와 각 문단 y advance가 다시 측정되는지
- 문단부호와 캐럿이 갱신된 cell paragraph path 및 bbox를 사용하는지

Issue #671은 파일에 저장된 `LINE_SEG`가 없는 셀 문단의 정적 렌더 fallback을 고친 작업이다. 이번 문제는 편집 중 새 문단 생성 경로이므로, #671과 같은 시각 증상만으로 같은 원인이라고 단정하지 않는다.

## 4. 수행 단계

### Stage 1. 재현과 원인 계층 분리

1. 현재 `upstream/devel` 기반 WASM을 `wasm-pack build --target web --out-dir pkg`로 빌드한다.
2. 이미 실행 중인 `http://localhost:7700`에서 원본을 열어 제보 절차를 재현한다. 별도 개발 포트는 열지 않는다.
3. 문단 부호 ON/OFF, 짧은 입력 후 즉시 Enter, 지연 flush 뒤 Enter를 각각 비교한다.
4. Enter 전후 셀 문단 수, `LINE_SEG`, 행 높이, render tree의 문단 y, 캐럿 rect를 기록한다.
5. Chrome 확장 v0.2.8과 현재 rhwp-studio에서 재현 범위가 같은지 구분한다.
6. 결과를 `mydocs/working/task_m100_2164_stage1.md`에 기록한다.

### Stage 2. 구현 계획 확정과 회귀 테스트 추가

1. Stage 1에서 확인된 실제 결함 계층만 수정 대상으로 선택한다.
2. `mydocs/plans/task_m100_2164_impl.md`에 수정 함수, 데이터 계약, 회귀 위험과 검증 항목을 기록한다.
3. 원본 HWP 또는 최소 합성 문서로 다음 계약을 검증하는 회귀 테스트를 먼저 추가한다.
   - Enter 뒤 셀 문단 수가 1 증가한다.
   - 새 문단의 y와 캐럿 y가 이전 문단보다 정상적인 줄 간격만큼 증가한다.
   - 문단 부호 ON/OFF가 레이아웃 좌표를 바꾸지 않는다.
   - Enter 전 지연 페이지네이션 상태 유무에 따라 결과가 달라지지 않는다.
4. 구현 계획과 실패하는 회귀 테스트를 작업지시자에게 보고하고 수정 승인을 받는다.

### Stage 3. 최소 범위 정정

1. Stage 2에서 확정한 계층만 수정한다.
2. 샘플명, 특정 셀 번호, 입력 문자열에 의존하는 하드코딩은 사용하지 않는다.
3. 일반 본문 Enter, 기존 표 셀 편집, 중첩 표, Backspace 병합 경로가 변하지 않도록 범위를 제한한다.
4. focused Rust/TypeScript 테스트와 WASM 빌드를 통과시킨다.
5. 결과를 `mydocs/working/task_m100_2164_stage2.md`에 기록한다.

### Stage 4. 실제 브라우저·확장 검증과 전체 회귀

1. `wasm-pack build --target web --out-dir pkg` 후 `localhost:7700`에서 동일 절차를 다시 수행한다.
2. 문단 부호 ON/OFF, 연속 Enter, Backspace 후 재입력, 저장 후 재열기를 확인한다.
3. Chrome 확장 개발 빌드에서도 같은 원본으로 검증한다.
4. Windows 교차 검증이 필요하면 `win10-ted`의 저장소에서 SSH 기본 셸, `cmd`, PowerShell 환경을 구분해 기록한다.
5. PR 전 검증은 `cargo fmt --check`, focused tests, `CARGO_INCREMENTAL=0 cargo test --profile release-test --tests`, 필요한 경우 `cargo clippy --all-targets -- -D warnings`, 프론트 빌드를 수행한다.
6. 시각 증적과 결과를 `mydocs/report/task_m100_2164_report.md`에 정리하고 작업지시자 판정을 받는다.

## 5. 완료 조건

- 제보 원본에서 Enter 후 텍스트, 새 문단, 문단 부호가 겹치지 않는다.
- 한컴 2024처럼 새 문단과 캐럿이 다음 줄 위치로 이동한다.
- 문단 부호 ON/OFF가 실제 문단 배치에 영향을 주지 않는다.
- Chrome 확장과 rhwp-studio에서 같은 결과를 확인한다.
- 원본이 `samples/issue2164/`에 보존되고 자동 회귀 테스트가 추가된다.
- 전체 검증과 작업지시자 시각 판정을 통과한다.

## 6. 승인 요청

Stage 1 재현과 원인 계층 분리를 위 범위로 진행하고, 결과를 보고한 뒤 구현 계획 승인 전에는 소스 코드를 수정하지 않는다.
