3. `Swatinem/rust-cache`는 위 두 단계 이후에도 hit rate 또는 quota 문제가 남을 때만 SHA-pinned 별도 PR로 검토

1번은 #1865와 수동 cleanup으로 핵심 리스크가 줄었다.

- PR run에서 Render Diff cargo save가 skipped 되는 것을 확인했다.

- stale `refs/pull/*` cache cleanup으로 총량이 10GB 아래로 내려갔다.

후속 PR 표본은 더 모아야 하지만, 이는 measurement 누적 작업이다. 현재 바로 착수할 수 있는 다음 본류는
Build & Test exact-hit 이후에도 남는 local crate compile의 성격을 분리하는 것이다.

## 문제 정의

현재 Build & Test cache는 다음 구조다.

- restore: `actions/cache/restore@v5`
- save: `actions/cache/save@v5`
- path: `~/.cargo/registry`, `~/.cargo/git`, `target`
- key: `Linux-cargo-${Cargo.lock hash}`
- PR run: restore-only
- `devel/main` push: exact hit이 아니면 save 허용

선행 관측에서는 exact hit 상태에서도 다음 compile 신호가 남았다.

| step | 관측 | 현재 해석 |
|------|------|-----------|
| Build | `Dirty rhwp` + `Compiling rhwp` | `devel` push release smoke라 정상 비용일 수 있음 |
| Native Skia tests | `Compiling rhwp` | `native-skia skia` feature 조합 때문에 별도 산출물일 수 있음 |
| Run lib tests | `Dirty rhwp` + `Compiling rhwp` | lib test harness / fingerprint 영향 분석 필요 |
| Run integration tests | `Dirty rhwp` + `Compiling rhwp` | integration test target / fingerprint 영향 분석 필요 |
| Clippy | `Checking rhwp` | check 계열. link compile과 분리 |

따라서 이번 단계의 핵심은 모든 `Compiling rhwp`를 제거 대상으로 보는 것이 아니라, 정상 compile과 개선 가능 compile을
분리하는 것이다.

## 분석 질문

1. `Dirty rhwp`는 어떤 step에서 반복적으로 발생하는가?
2. 각 `Dirty rhwp` 로그의 원인은 `dependency info changed`, 소스 파일 timestamp 변경, feature 변경, test harness 변경 중 무엇인가?
3. exact cache hit, fallback hit, miss에서 `Dirty rhwp` 패턴이 달라지는가?
4. `target` 전체 cache가 Build & Test local crate 재사용에 실제로 기여하는가?
5. `Run lib tests`와 `Run integration tests`를 분리 유지할 때 반복 compile이 구조적으로 unavoidable 한가?
6. step 통합, profile/feature별 cache key 분리, cache path 축소, `Swatinem/rust-cache` 중 어느 후보가 실제 개선 가능성이 있는가?

## 데이터 수집 기준

### run 선정

- `devel` push `CI / Build & Test` success run 3개 이상
- full CI가 실행된 PR `CI / Build & Test` success run 3개 이상
- fast-pass run은 제외
- 가능하면 cache exact hit / fallback hit / miss 사례를 분리

### step별 기록 항목

| 항목 | 기록 내용 |
|------|-----------|
| run URL | GitHub Actions run URL |
| event/ref | `pull_request`, `push`, `refs/heads/devel`, `refs/pull/*/merge` |
| head SHA | run 대상 SHA |
| cache restore | exact hit / fallback hit / miss |
| cache size | restore된 `Linux-cargo-*` 크기 |
| save | skipped / success / failure |
| Build | profile, `Dirty rhwp`, `Compiling rhwp`, `Finished` 시간 |
| Native Skia tests | feature, profile, compile 여부, finished 시간 |
| Run lib tests | profile, test harness 여부, dirty 원인, finished 시간 |
| Run integration tests | profile, dirty 원인, finished 시간 |
| Clippy | check 계열 여부 |

## 판단 기준

### 정상 비용으로 분류할 수 있는 경우

- `devel` push Build step의 `release` profile compile
- PR/devel test step과 다른 feature set을 사용하는 Native Skia compile
- test harness 또는 integration test target 생성 때문에 한 번 필요한 compile

### 개선 후보로 분류할 수 있는 경우

- 같은 profile / 같은 feature / 같은 target에서 같은 run 안에 반복 `Dirty rhwp`가 발생하는 경우
- exact cache hit 뒤에도 source timestamp 또는 dependency info 변경으로 local crate가 매번 dirty 되는 경우
- `target` cache가 restore되어도 유의미한 reuse가 없고 cache 크기만 크게 유지되는 경우
- step 분리 때문에 같은 산출물을 재사용하지 못하는 것이 로그로 확인되는 경우

## 개선 후보

이번 계획서는 후보를 확정하지 않고, 로그 분석 후 아래 중 하나를 선택한다.

| 후보 | 내용 | 장점 | 리스크 |
|------|------|------|--------|
| A. 현행 유지 + 문서화 | 반복 compile이 Cargo 구조상 정상 비용이면 코드 변경 없음 | 리스크 없음 | 시간 단축 없음 |
| B. step 통합 실험 | lib/integration test cargo invocation 일부 통합 검토 | 중복 compile 감소 가능 | step별 가시성 및 1:1 추적성 약화 가능 |
| C. cache key 분리 | profile/feature/target별 key 분리 | hit 정확도 개선 가능 | cache 총량 증가 가능 |
| D. target 제외 | registry/git 중심 cache로 축소 | quota 안정성 향상 | compile 시간 증가 가능 |
| E. `Swatinem/rust-cache` 검토 | pruning/shared-key/save-if 등 검토 | target cache 운영 개선 가능 | third-party action SHA pinning 및 별도 PR 필요 |
