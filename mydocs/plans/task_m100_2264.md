# 수행계획서 — M100 #2264: PDF 경로 시스템 폰트 전량 등록 제거

## 1. 배경

`src/renderer/pdf.rs:110` 의 `fontdb.load_system_fonts()` 가 시스템 폰트를 전량 등록한다. 이후 usvg 가 텍스트를 path 로 변환하며 매칭·폴백 후보 face 를 실제로 메모리에 실체화하는데, 시스템 CJK 폰트는 개당 수십 MB 라 RSS 가 크게 뛴다.

## 2. 측정 근거

`load_system_fonts()` 를 임시 제거하고 비교 (native release, `/usr/bin/time -l`, 1페이지):

| 문서 특성 | 현재 | 제거 시 | 차이 |
|---|---|---|---|
| 이미지 다수, 8쪽 | 185 MB | 78 MB | **−107 MB** |
| 이미지 중간, 74쪽 | 122 MB | 45 MB | **−77 MB** |
| 텍스트 거의 없는 최소 문서 | 5 MB | — | 거의 0 |

**비용은 일정하지 않고 텍스트·폰트 사용량에 비례한다.** 등록 자체(디렉터리 스캔 + face 메타데이터 파싱)는 싸다. 실제 비용은 usvg 가 face 데이터를 메모리에 올릴 때 발생한다.

## 3. 핵심 제약 — 단순 제거는 답이 아니다

usvg 는 텍스트→path 변환에 fontdb 를 필요로 한다. 시스템 폰트를 그냥 빼면 **폰트 폴백 결과가 바뀌어 시각 회귀가 발생**한다. 위 측정에서 제거 후에도 렌더가 된 것은 `ttfs/` 프로젝트 폰트가 대신 매칭됐기 때문이며, **시각 동등성은 검증되지 않았다.**

따라서 목표는 "로드하지 않는다" 가 아니라 **"필요한 폰트만 남긴다"** 이다.

## 4. 접근

`fontdb` 는 `faces()` 로 등록된 face 메타데이터를 열람하고 `remove_face(id)` 로 개별 제거할 수 있다 (fontdb 0.23).

1. `load_system_fonts()` — 종전대로 등록 (메타데이터만 파싱, 저렴)
2. **문서가 실제로 참조하는 폰트 패밀리 집합**을 SVG 문자열에서 수집 (`font-family` 속성/스타일)
3. 참조 집합 ∪ 폴백 패밀리(serif/sans/mono, 수식 폰트 체인) 에 속하지 않는 **시스템 face 를 제거**
4. 프로젝트 폰트(`font_paths`, `ttfs/`)는 제거 대상에서 제외 — 의도적으로 번들된 것이며 크기도 작다

usvg 의 글리프 폴백은 DB 를 순회하며 face 데이터를 열어보므로, **DB 를 줄이면 폴백 스캔이 대형 CJK face 를 실체화하지 않는다.**

참조 집합이 비면 가지치기를 건너뛴다 (기존 동작 유지).

## 5. 위험

- **글리프 폴백 변화**: 문서가 참조하는 폰트에 없는 글리프를, 종전에는 usvg 가 시스템 폰트 전체를 뒤져 찾아냈다. 가지치기 후 후보가 줄면 해당 글리프가 tofu 가 될 수 있다. → **PDF 시각 회귀 게이트가 1차 방어선.** 회귀가 나오면 유지 집합을 넓힌다(예: 시스템 CJK 폰트 소수를 항상 유지).
- 폰트 패밀리 수집 누락 시 필요한 폰트가 제거될 수 있다 — 수집은 usvg 에 넘기기 직전의 SVG 기준이어야 한다.
