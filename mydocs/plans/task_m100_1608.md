**제목**: `is_hwp3_origin` 오탐지 — 네이티브 한글2022 HWPX(xmlVersion 1.4)에 부당한 HWP3 보정(tolerance) 적용

**전사(前史)**: Task #1600(−1쪽 갭) 조사 중 분리된 독립 버그. 상세 `mydocs/tech/investigations/issue-1600/render_minus1_page_gap.md`

## 1. 배경

`src/parser/hwpx/mod.rs:181`:
```rust
let is_hwp3_origin = hwpml_version.as_deref() == Some("1.4");
```
`is_hwp3_origin`은 HWP3→HWPX 변환본을 식별해 한글97 "마지막 줄 tolerance"(1600 HU ≈ 21px)를
각 섹션 `page_def.pagination_bottom_tolerance`에 부여한다(Task #554, hwpx/mod.rs:264-268).

## 2. 문제 (확정)

판별 기준 `head version == "1.4"` 는 **HWPML 스키마 버전**일 뿐 HWP3 변환 지표가 아니다.
네이티브 한글2022 HWPX도 `head version 1.4`라 **거의 모든 모던 HWPX가 HWP3-origin으로 오탐지**
된다(저장소 `samples/hwpx` 전수 포함). 증거(version.xml, 서울시 opengov 36387725):
`application="Hancom Office Hangul" appVersion="11"`(한글2022), `major=5 minor=1`(네이티브 HWP5.1).

### 메커니즘 (코드 경로 확정)

- **파싱 시점**(hwpx/mod.rs:264): `is_hwp3_origin` → `page_def.pagination_bottom_tolerance` 설정.
- **렌더(기본 TypesetEngine)**: `page_layout.rs:88` `pagination_tolerance_px` →
  `available_body_height() = body − footnote + tolerance`(page_layout.rs:148) → 본문 fit 판정에
  +21px 추가 공간 → 경계 문서가 1쪽 적게 렌더.
- 별개 메커니즘인 렌더 시점 bool `hwp3_origin_page_tolerance`(rendering.rs:2242, 콘텐츠
  휴리스틱 `uses_hwp3_origin_page_tolerance`)는 near-top vpos reset / title-filler 전용으로
  본 버그(본문 높이 tolerance)와 무관 — **이번 수정 대상 아님**.

## 4. 판별자 부재 (정밀 탐지 난점 — 조사 확정)

진짜 HWP3 변환본(`samples/hwp3-sample-hwpx.hwpx`)과 네이티브가 **메타데이터 동일**:
head version 1.4 / application "Hancom Office Hangul" / secCnt 1 / isEmbeded 1 모두 일치.
→ version·application·embedding 단일 메타데이터로 **구별 불가**.

### 권고: 방향 3

근거:
- 메타 판별자 부재(§4)로 방향 1·2의 깔끔한 정밀 수정 불가가 조사로 확정됨.
- 방향 3은 **유일하게 측정된 net 개선(+6)** 이고 통제 게이트(개선−회귀>0)를 충족.
- "거의 모든 네이티브 HWPX에 부당한 tolerance" 라는 **원 버그를 직접 제거**.
- 잔여 비용: ① 네이티브 2건 +회귀, ② 진짜 HWP3 변환본의 tolerance 상실 → `samples/hwp3-*`
  baseline·시각 회귀로 안전성 검증(아래 게이트).

> tolerance를 **완전 삭제**할지, 진짜 HWP3 변환본에는 살아있도록 더 보수적인 가드(예: 렌더
> 시점 콘텐츠 휴리스틱 재사용)를 둘지는 구현계획서에서 RED 측정 후 확정한다. 1차 권고는
> **완전 제거**(가장 단순, 측정상 net 개선)이며, HWP3 변환본 회귀가 발견되면 가드 추가로 보강.
