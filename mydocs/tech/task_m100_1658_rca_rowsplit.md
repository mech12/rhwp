# RCA — #1658 법령 별표 표 행분할 over-pagination (낭비 페이지)

- 일자: 2026-06-29 / 브랜치: local/task1658 / 도구: dump-pages, RHWP_TABLE_DRIFT
- 대상: 대형 오라클 over 지배항 = `law 별표 .hwp` (rhwp 가 표를 더 잘게 나눠 페이지↑).

## 1. 대표 케이스
`국토교통부/[별표 1] 부설주차장…(주차장법 시행령).hwp` — 한글 4쪽 / rhwp **5쪽**(Δ+1).
(스케일 대표: `산업통상부/[별표 4] 액화석유가스…` 한글25 / rhwp33 Δ+8 도 동류.)

## 2. 구조 (dump-pages + RHWP_TABLE_DRIFT)
- 표 = **15행×2열**, body h=1009.1px.
- 행높이(cut=mt, 측정 정확): `[27,50,31,31,228,126,104,104,104,52,25,23,23,42, **2557.9**]`.
  → **행 14(마지막)가 2557.9px = 약 2.5 페이지 분량의 거대 셀**(나머지 행 합 ≈ 972px = 1쪽).
- 페이지별 셀 라인 cut: p1 end=[1], p2 [1..35], p3 [35..36], p4 [36..70], p5 [70..].
  → **페이지별 셀 라인 수 = 1 / 34 / 1 / 34 / 나머지.**

## 3. 근인 (확정 방향)
- **페이지 3 이 단 1줄**(라인 35→36)만 담는다. 페이지 2·4 는 각 34줄. 라인 높이는 균일(~28px)이라
  페이지 3 도 34줄 들어가야 정상. → **이 1줄 낭비 페이지가 Δ+1 의 원인**(한글엔 없음).
- **측정 버그 아님**: `RHWP_TABLE_DRIFT` 결과 cut_sum==mt_sum(3530.6, diff 0). 행높이 정확.
- ∴ 원인은 **intra-row 셀 라인 cut 루프(`typeset_block_table` → `advance_row_cut`)의 상태/경계 버그**:
  연속(continuation) fragment 에서 page 3 의 가용/시작 상태가 잘못 계산되어 1줄만 자르고 페이지를 끊음
  (두 정상 34줄 페이지 사이에 1줄 페이지가 끼는 1/34/1/34 비정상 패턴).

## 4. 의의 — fixpoint 아님, 국소 버그
- #1653/#1658 의 하단 표 over 는 px-level fixpoint(난제)였으나, **본 행분할 over 는 측정·산술이
  정확한데 cut 루프가 낭비 페이지를 만드는 국소 버그** → **수정 가능성 높음, 스케일 ROI 큼**
  (대형 오라클 over 지배항).

## 5. 다음 단계 (수정 착수안)
1. `advance_row_cut`(LayoutEngine) 연속 fragment 의 라인 cut 계산 계측:
   page2→page3 경계에서 page3 의 시작 available/누적이 왜 1줄만 허용하는지 추적
   (current_height 리셋 누락 / first_frag_overhead 오적용 / start_cut 비어있지 않을 때의 off-by-one 의심).
2. 1줄 낭비 제거 → rhwp 5→4쪽(한글 일치) 확인. 양 게이트(75/441) 무회귀 검증.
3. 산업통상부 Δ+8 등 다중 낭비 페이지 케이스로 일반성 확인.
