# 최종 결과 보고서 — M100 #2264: PDF 경로 시스템 폰트 메모리

## 결론

**"참조하지 않는 시스템 face 가지치기" 접근은 실패했다. 코드 변경 없이 종료한다.**

PDF 경로가 쓰는 100MB 이상의 메모리는 **문서가 실제로 사용하는 한글 폰트 데이터**다. 출력을 바꾸지 않고는 걷어낼 수 없다.

## 경과

### 초기 관측과 그 함정

`load_system_fonts()` 를 제거하고 측정하면 RSS 가 크게 떨어진다:

| 문서 | 현재 | `load_system_fonts()` 제거 | 차이 |
|---|---|---|---|
| 이미지 다수, 8쪽 | 185 MB | 78 MB | −107 MB |
| 이미지 중간, 74쪽 | 122 MB | 45 MB | −77 MB |

이 수치만 보면 "시스템 폰트 전량 등록이 낭비" 라는 결론이 나온다. **이것이 함정이었다.** 제거 후에도 렌더가 되긴 하지만, 그건 `ttfs/` 프로젝트 폰트가 대신 매칭된 결과이고 **출력이 달라진다.**

### 시도한 접근

fontdb 는 `faces()` 로 face 메타데이터를 열람하고 `remove_face(id)` 로 개별 제거할 수 있다. 등록 자체는 메타데이터만 파싱하므로 싸다(텍스트 없는 최소 문서의 PDF 경로 RSS 는 5MB). 실제 비용은 usvg 가 face 데이터를 메모리에 올릴 때 발생한다.

→ **시스템 폰트를 등록하되, 문서가 참조하지 않는 face 만 제거**하면 출력을 유지한 채 메모리를 줄일 수 있다고 판단했다.

### 구현 중 발견한 두 결함

1. **SVG 의 `font-family` 값은 XML 엔티티로 escape 되어 있다** — `font-family="HY견명조,&apos;Batang&apos;,serif"`. 이를 풀지 않으면 패밀리 이름이 `&apos;batang&apos;` 로 수집되어 실제 face 와 매칭되지 않는다.
2. **generic family 의 실제 폰트명은 SVG 에 절대 등장하지 않는다** — `serif`/`sans-serif` 키워드만 나오고, 실제 폰트명(macOS 의 `AppleMyungjo` 등)은 `set_serif_family()` 로 fontdb 에 따로 설정된다. SVG 만 훑어서는 폴백 폰트를 알 수 없다.

두 결함 때문에 1차 시도는 필요한 face 를 몽땅 제거했고, PDF 시각 회귀 게이트가 이를 잡아냈다 (픽셀 차이 4.5~10.7%, 한 문서는 PDF 생성 실패).

### 결함 수정 후 — 그리고 진짜 결론

XML 엔티티를 풀고 폴백 패밀리를 명시적으로 보존하도록 고친 뒤 재측정:

| 문서 | before | after | 메모리 절감 | 픽셀 차이 |
|---|---|---|---|---|
| 텍스트 거의 없는 최소 문서 | 5 MB | 5 MB | 0 | **동일** |
| 이미지 다수, 8쪽 | 287 MB | 266 MB | −21 MB | 0.0144% |
| 이미지 중간, 74쪽 | 322 MB | 301 MB | −21 MB | 0.0378% |
| 비트맵 다수, 20쪽 | 517 MB | 512 MB | −5 MB | 0.3738% |
| 이미지 다수, 41쪽 | 251 MB | 252 MB | **+1 MB** | 0.0240% |

**폰트를 올바로 보존하니 메모리 절감이 사실상 사라졌다.** 그러면서도 픽셀 차이는 0 이 아니다 — 여전히 일부 face 가 제거되어 희귀 글리프의 폴백이 달라진다.

즉 앞선 107MB 절감은 **렌더링을 망가뜨려서 얻은 것**이었다. 시스템 폰트가 점유하는 메모리는 낭비가 아니라, 문서의 폰트 체인(한 문단이 14개 패밀리를 나열한다)을 따라가며 usvg 가 **실제로 사용하는** 한글 폰트의 데이터다. 한글 폰트는 개당 수십 MB 다.

메모리 이득 없이 시각 회귀만 남는 변경이므로 **채택하지 않는다.**

## 확인된 사실 (다음 시도를 위한 기록)

- `ttfs/` 프로젝트 폰트는 5.1 MB(2개 파일)로 원인이 아니다.
- fontdb 등록(디렉터리 스캔 + face 메타데이터 파싱)은 저렴하다. 텍스트가 없는 문서의 PDF 경로 RSS 는 5 MB 다.
- 비용은 **usvg 가 텍스트를 shaping 하며 매칭된 face 데이터를 실체화할 때** 발생하고, 문서의 텍스트·폰트 사용량에 비례한다.
- SVG 렌더러가 만드는 폰트 체인은 한 항목당 14개 안팎의 패밀리를 나열한다. 앞쪽이 미설치면 뒤로 밀리며, 결국 시스템 CJK 폰트가 선택된다.

## 남은 선택지 (미검증, 재조사 필요)

1. **번들 폰트 우선 매칭** — 시스템 CJK 폰트 대신 `ttfs/opensource` 의 경량 폰트를 우선 쓰게 하면 메모리는 줄지만 **출력이 바뀐다.** 시각 충실도를 포기하는 트레이드오프이므로 작업지시자 판단이 필요하다.
2. **usvg/svg2pdf 단계의 메모리 프로파일링** — 폰트 외에 이미지 디코딩·PDF chunk 생성이 차지하는 몫을 분리 측정해야 한다. 본 이슈에서는 폰트 항만 다뤘다.
3. **PDF 경로 자체를 우회** — Quick Look 이 목적이라면 PDF 대신 다른 표현(예: 페이지 이미지)을 검토한다.

## Quick Look 에 대한 함의

macOS Quick Look 확장은 약 80MB 경고 / 120MB 강제 종료다. **PDF 경로로는 이 한도를 맞출 수 없다는 것이 본 조사의 결론이다.** #2263 으로 파싱·조판 경로는 한도 안에 들어왔으나, PDF 변환 단계가 여전히 100MB 이상을 쓴다.

Quick Look 을 계속 추진한다면 위 3번(PDF 우회)을 먼저 검토해야 한다.
