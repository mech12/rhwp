# 최종 결과 보고서 — M100 #2264: PDF 경로 메모리

## 결론

PDF 변환 메모리의 지배항은 **폰트 등록도, 이미지 디코딩도 아니라 `svg2pdf` 의 텍스트 임베드(폰트 서브셋)** 였다.

`PdfExportOptions.embed_text` (기본 `true` = 종전 동작) 와 CLI `--text-as-paths` 를 추가했다. 끄면 글리프를 path 로 변환해 서브셋 경로를 통째로 건너뛴다.

**#2263 과 결합하면 1페이지 PDF 최대 RSS 가 41~99 MB 로 떨어져, macOS Quick Look 의 120MB 강제 종료선 아래로 전부 들어온다.**

## 경과 — 두 번의 오진

### 1차 오진: "문서와 무관하게 일정한 비용"

이슈 최초 등록 시 "시스템 폰트 전량 등록이 문서와 무관하게 +85~140MB 를 붙인다"고 적었다. **틀렸다.** 텍스트가 거의 없는 최소 문서의 PDF 경로 RSS 는 5MB 다. 일정 비용이면 나올 수 없는 값이다. 이슈 본문을 정정했다.

### 2차 오진: "시스템 face 가지치기로 해결"

`load_system_fonts()` 를 제거하면 RSS 가 107MB 떨어진다. 이 수치에 이끌려 "참조하지 않는 시스템 face 를 제거" 하는 접근을 구현했으나 **실패했다.** 폰트를 올바로 보존하니 절감이 사실상 사라졌다 (287→266, 517→512, 251→252). 107MB 절감은 **폰트 폴백이 깨진 결과**였다. 이 접근은 폐기했다.

교훈: **비용의 위치를 추측하지 말고 단계별로 분해해 측정한다.**

## 단계별 프로파일링

`examples/task2264_profile.rs` — SVG 를 입력받아 단계마다 별도 프로세스로 실행하고 `/usr/bin/time -l` 로 최대 RSS 를 측정한다. 텍스트/이미지/시스템폰트를 개별 제거(ablation) 할 수 있다.

대상: 텍스트 요소 1639개, 이미지 2개, 4.3 MB SVG (1페이지)

| 단계 | 최대 RSS |
|---|---|
| SVG 읽기 | 5 MB |
| fontdb 구성 (시스템 폰트 등록 + `ttfs/`) | **6 MB** |
| usvg 파싱 (텍스트 shaping + 이미지 디코딩) | **22 MB** |
| **`svg2pdf::to_chunk`** | **164 MB** |

폭발 지점은 `svg2pdf::to_chunk` 다. 폰트 *등록* 은 6MB 로 싸고 usvg 파싱도 22MB 로 싸다.

ablation 으로 원인 특정:

| chunk 단계 구성 | 최대 RSS | 해석 |
|---|---|---|
| 기본 | 164 MB | — |
| 텍스트 제거 | 51 MB | **−113 MB** ← 지배항 |
| 이미지 제거 | 150 MB | −14 MB (미미) |
| 시스템 폰트 미등록 | 55 MB | −109 MB (단, 출력이 바뀜) |
| **`embed_text = false`** | **69 MB** | **−95 MB (출력 유지)** |

이미지가 아니라 **텍스트**가 범인이고, 정확히는 svg2pdf 가 텍스트를 PDF 폰트로 임베드하며 수행하는 **폰트 서브셋** 이다. 한글 폰트는 개당 수십 MB 이므로 서브셋 과정에서 전체 폰트를 메모리에 올린다.

## 구현

```rust
pub struct PdfExportOptions {
    ...
    /// 텍스트를 PDF 폰트로 임베드할지 여부. false 면 글리프를 path 로 변환한다.
    pub embed_text: bool,   // 기본 true = 종전 동작
}
```

- `svg2pdf::ConversionOptions.embed_text` 로 전달.
- CLI `--text-as-paths` 로 노출.
- **기본값은 종전 동작(`true`)** 이므로 기존 사용자에게 회귀가 없다. 저메모리가 필요한 호출자(Quick Look 등)만 끈다.

## 트레이드오프 — 반드시 인지할 것

`embed_text = false` 면 **PDF 의 텍스트 선택·검색 기능을 잃는다.** 글자는 벡터 도형이 된다.

시각적으로는 동일한 내용·레이아웃·폰트로 렌더된다. 래스터화 비교에서 픽셀 차이가 잡히지만(80dpi 기준 9~11%), 이는 **글리프 외곽선의 안티앨리어싱·힌팅 차이**다. 차이 픽셀의 71% 가 글자 내부가 아니라 **가장자리**에 몰려 있고, 육안 비교에서 내용 차이는 없다(path 쪽이 미세하게 굵게 보인다). 구조적 회귀가 아니다.

따라서 **본문 검색이 필요한 PDF 내보내기에는 기본값(임베드)을 유지**하고, 미리보기·썸네일처럼 메모리 제약이 지배적인 용도에만 끈다.

## Quick Look 에 대한 함의

macOS Quick Look 확장은 약 80MB 경고 / **120MB 강제 종료** 다.

**#2263 + #2264 로 모든 측정 문서가 120MB 강제 종료선 아래로 들어왔다** (41~99 MB). 일부는 80MB 경고선을 넘지만 경고는 종료 사유가 아니다.

앞서 "PDF 경로로는 Quick Look 한도를 맞출 수 없다" 고 낸 결론은 **번복된다.** 초기에 세운 "HWP→PDF→`QLPreviewProvider`" 아키텍처가 다시 유효하다.

남은 과제:
- 80MB 경고선 아래로 더 내리려면 usvg 파싱(22MB)·이미지 디코딩 항을 추가로 다뤄야 한다.
- 샌드박스 폰트 경로(CWD 상대경로 `ttfs/`) 정리는 Quick Look 착수 시 함께 처리한다.

## 남은 관측 (미해결)

전체 문서 내보내기에서 비트맵이 매우 많은 한 문서는 `--text-as-paths` 가 오히려 메모리를 늘렸다 (514→546 MB). 글리프를 path 로 펼치면 경로 지오메트리가 커지므로, 텍스트가 매우 조밀한 문서에서는 서브셋 비용을 상회할 수 있다. **1페이지 렌더(미리보기 용도)에서는 관측되지 않았다.** 전체 문서 내보내기에 이 옵션을 기본으로 켜면 안 되는 또 하나의 이유다.
