## 2. 수정한 결함 — InFrontOfText treat_as_char 표 흐름 높이 미예약 (+3쪽)

## 3. 주 드라이버(−94) 최종 확정 — 전면 매뉴얼 이미지 96장이 한 페이지에 스택

수정 후에도 rhwp 174 vs 한글 268 (−94). 다음을 배제/확인:

- **순수 본문**: 한글과 완벽 정렬 (p14 문단 간격·줄 pitch 동일). 배제.
- **자리차지(TopAndBottom) 단일셀 박스**: stored line_seg 높이를 정확히 예약(예: pi=158 콘텐츠 14줄=194.7px 정확, pi 최대 673.8px). 배제.
- **over-full 페이지**: 10/174 뿐, 최대 126%. 미미.

남은 −94는 **한글이 콘텐츠를 더 많은 페이지에 분산**하는 데서 온다. 오라클 Q&A/사례 구간(예: 오라클 p186~209)은 대형 테두리 박스가 페이지당 1~2개 + 페이지 절반이 공백인데, rhwp 는 같은 콘텐츠를 더 조밀하게 담는다. 개별 박스/표 높이는 정확하므로 원인은 **페이지 분할 규칙 차이**다.

**반증된 가설 (기록)**:
- `keep_with_next`/`keep_lines` 미준수 가설 → **반증**. typeset.rs 가 이들을 미참조하는 건 사실이나, 청년주거·편람 문서 모두 해당 플래그가 **전부 0**(설정된 문단 없음). page_break_before 도 0. 페이지 분할 규칙 플래그는 원인이 아니다.

**콘텐츠 양은 동일, 밀도가 다름 (정확한 텍스트 줄 카운트)**:
- fitz 텍스트 줄: **오라클 5923 vs rhwp 6426** — rhwp 가 오히려 약간 많음. **콘텐츠 과소측정/드롭 아님**.
  - (앞서 관측한 "2.6배"는 밴드측정이 표 테두리선을 줄로 오계수한 아티팩트였음 — 정정.)
- 이미지도 정상 렌더(rhwp 128개, 최대 994px). 드롭 아님.
- **밀도 차이가 원인**: rhwp 평균 36.9줄/쪽(≈92% fill) vs 오라클 22.1줄/쪽(중앙값 9, 137/268쪽이 <12줄). 같은 콘텐츠를 오라클은 성기게(공백 많이), rhwp 는 조밀하게 담아 174 vs 268.

**★ 근본 원인 최종 확정 (결정적) — 임베드 매뉴얼 이미지 96장이 한 페이지에 스택**:

페이지별 정렬 대조로 divergence 위치를 특정했다:
- Q&A/본문 구간 정렬: rhwp p118 ≈ 오라클 p117 (동일 Q77~Q80).
- 부록(서식1~14) 정렬: rhwp p157~174 = 오라클 p251~268 (각 18쪽).
- **divergence 는 그 사이 제4장 구간**: rhwp p118→157 = 39쪽 vs 오라클 p117→251 = **134쪽**. 오라클 p120~244 중 **101쪽이 전면 이미지**.

원인: **문단 0.1004 에 전면 플로팅 이미지 96개**(각 130×182mm ≈ 전면 A4, tac=false, 제4장 "청년월세 한시 특별지원사업 매뉴얼" 본문을 이미지로 삽입)가 있는데, rhwp 는 이 96개를 **전부 한 페이지(p141)의 단일 앵커 y=−509(페이지 밖)에 완전히 스택**한다(render-tree 고유 y=1개). 한글은 각 전면 이미지를 흐름 배치하여 **~96쪽**으로 펼친다.

**정량 정합**: 96 이미지 → +95쪽 → rhwp 174 + 95 = **269 ≈ 오라클 268**. 즉 이 단일 결함이 −94 를 거의 전부 설명한다.

**#1995 진단 정정**: RowBreak 표 문제가 **아니다**. 서베이의 "RowBreak 표 다수" 상관관계는 우연(정부 연구보고서가 RowBreak 콜아웃 박스와 임베드 매뉴얼 이미지를 둘 다 많이 보유). 실제 축은 **한 문단의 다수 전면 플로팅 이미지가 페이지 흐름으로 펼쳐지지 않는 결함**.

이는 앞서 확정한 InFrontOfText treat_as_char 박스 결함(§2)과 같은 계열("플로팅 객체가 흐름 높이를 예약/전진하지 않음")의 **이미지판·다중판**이다.
