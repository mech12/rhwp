## 2. 최종 UI

```
[−]  [ 100% ▾ ]  [+]
          ├─ 폭 맞춤                 → 1쪽 + fitWidth
          ├─ 쪽 맞춤 (전체 보기)      → 1쪽 + fitPage
          ├─ 두 쪽 맞춤              → 2쪽 + fitPage
          ├────────────────────
          └─ 50 / 75 / 100 / 150 / 200%  → 배치 유지 + 수동 배율
```

- 현재 선택 항목에 `✓` 표시. 바깥 클릭 / `Esc` 로 닫힘.
- 맞춤 3항목이 쪽 배치까지 결정하는 유일한 진입점이다. 2쪽 → 1쪽 복귀는 `폭 맞춤` / `쪽 맞춤` 으로 이루어진다.
- `−`/`+`, Ctrl+휠은 수동 배율로 전환한다(맞춤 해제).

### 설계 요점

- `currentZoom`(숫자 하나) → `zoomMode`(`manual`/`fitWidth`/`fitPage`) + **실제 적용 배율**로 분리. `currentZoom` 이 계속 실제 배율을 담으므로 렌더·썸네일 등 하위 경로는 수정하지 않았다.
- 맞춤 기준 쪽 크기는 **문서 전체의 최대 폭·최대 높이**. 현재 쪽 기준으로 하면 쪽 크기가 섞인 문서에서 스크롤 중 배율이 요동친다.
- 스크롤바 진동은 두 겹으로 차단: `ResizeObserver` 의 `contentRect`(스크롤바 제외) 기준 계산 + 1% 히스테리시스.

### 구현 중 잡은 결함 2건

1. **레이아웃 재구성 누락** — 기존 `applyZoom()` 의 `if (newZoom === currentZoom) return;` 조기 반환 탓에, 배율이 우연히 같은 채로 1쪽↔2쪽 배치만 바뀌면 레이아웃이 재구성되지 않는다. `setZoom(relayoutAnyway)` 인자로 우회.
2. **세로 padding 이중 차감** — `ResizeObserver` 의 `contentRect.height` 는 이미 padding 이 제외된 값인데 `computeFitZoom()` 이 이를 또 뺐다. 리사이즈 경로에서 쪽 맞춤 배율이 실제보다 작게 나온다. 파라미터 의미를 content-box 로 통일하여 수정.

## 4. 검증 상태 — **시각 확인 미완**

수행한 검증:

- `npx tsc --noEmit` (webview / extension 양쪽) → 통과
- `npm run compile` (webpack production) → 에러·경고 없이 성공

**수행하지 못한 검증**: 수행계획서 4장의 수동 시나리오 1~7 (실제 렌더 결과 육안 확인).

rhwp-vscode 에는 자동화 테스트 하네스가 없어 시각 검증은 확장 개발 호스트를 띄워 눈으로 보는 수밖에 없다. 확장 개발 호스트 실행을 시도했으나 창을 띄우지 못하고 종료되었고, 전체 화면 캡처 방식은 작업지시자의 다른 창까지 담기므로 중단했다(캡처본 삭제 완료).

**따라서 아래 시나리오는 작업지시자 확인이 필요하다.**

1. `두 쪽 맞춤` → 가로 스크롤 없이 두 쪽 전체가 보이는가
2. `폭 맞춤` → 쪽 폭이 뷰포트를 채우고 세로 스크롤만 생기는가
3. `쪽 맞춤` → 한 쪽 전체가 뷰포트 안에 들어오는가
4. 각 맞춤 모드에서 **에디터 패널 폭/높이를 바꾸면** 배율 표시(%)가 따라 바뀌는가
5. 사이드바 접기/펼치기에서도 재계산되는가
6. `−`/`+`, Ctrl+휠 사용 시 맞춤이 해제되고 수동 배율이 되는가
7. 세로/가로 문서, 쪽 크기가 섞인 문서에서 배율 진동이 없는가

확인 방법: `rhwp-vscode` 를 VS Code 로 열고 F5 (확장 개발 호스트) → `.hwp` 파일 열기.
