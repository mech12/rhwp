# #1949 Stage 1 — 핫스팟 특정 (프로파일링)

- 브랜치: `local/task1949` / 대표 파일: `수면비행선박 안전시설요건.hwpx`(0.3MB, >400s)
- 방법: 단계별 타이밍 + 구조 분석 + 렌더 재호출 카운트(코드 무수정/임시 계측 후 원복)

## 1. 결론 — 렌더가 거대 셀을 **페이지마다 통째로 재구성**한다 (O(pages × cell))

병목은 파싱·페이지네이션이 아니라 **렌더(레이아웃) 단계**이며, 원인은 한 표의 거대 셀을
그 표가 걸치는 **모든 페이지에서 재구성**하는 것이다.

## 2. 실측 근거

### 2-1. 단계 분해 — 병목은 렌더
- `info`(파싱): 즉시. `dump`(IR): 19.8s. `dump-pages`(페이지네이션): **19.5s, 112쪽 완료**.
- `export-svg`(페이지네이션+렌더): **>400s**. → 렌더가 ~380s 지배.

### 2-2. 문서 구조 — 3×1 RowBreak 표의 셀[2]가 거대
- 바깥 표: **3행×1열 RowBreak**(제목반복). 셀[0]/셀[1] 은 1문단, **셀[2] 문단수 = 2507**.
- 셀[2] 안에 **중첩 표 수십 개**(9×9=74셀, 14×2=28셀, 8×2, 6×2, 4×3 …)가 산재.

### 2-3. 재호출 카운트 — 바깥 표가 112쪽 전부에 렌더
`dump-pages` 집계: 바깥 표(pi=0)가 **112쪽 전부에 PartialTable 로 렌더**(범위 1~112).
→ `layout_partial_table` 이 이 표에 대해 **112회 호출**된다.

### 2-4. 핫스팟 코드 — 매 호출마다 셀 문단 전체 재구성
`src/renderer/layout/table_partial.rs:581-615`:
```rust
let mut composed_paras: Vec<_> = cell.paragraphs.iter()
    .map(|p| compose_paragraph(p)).collect();        // 셀[2] 2507문단 전체 compose
// ... 이어서 2507문단 전체 recompose_for_cell_width 루프(606-615)
```
- `layout_partial_table` 은 한 페이지에 보일 슬라이스(`start_cut`/`end_cut`)만 그리면서도
  **셀 문단 전체(2507)를 매 호출 재구성**한다. 그 안의 중첩 표들도 페이지마다 다시 배치.
- 총 비용 ≈ **112쪽 × (2507 compose + ~30 중첩표 배치)** ≈ 28만 compose + 수천 중첩표 배치.
  compose_paragraph 는 env-var-per-call 계측만으로도 단일 페이지가 타임아웃될 만큼 hot 임을
  확인(계측 후 원복).

## 3. 근본 원인 요약

RowBreak 표의 한 셀 콘텐츠가 여러 페이지에 걸쳐 분할될 때, 렌더는 각 페이지에서 그 셀의
**전체 콘텐츠(문단 compose + 중첩표 배치)를 처음부터 재계산**한다. 셀 폭(`inner_width`)은
페이지 간 불변이므로 재구성 결과는 매번 동일 → **순수 중복 계산**. 페이지 수 × 셀 크기로
비용이 폭증(112 × 2507 → 병리적).

## 4. Stage 2 제안 (승인 요청)

- **셀 콘텐츠 구성 메모이제이션**: `(para_index, control_index, cell_index)`(+inner_width) 키로
  `composed_paras`(compose+recompose 결과)를 표 단위로 1회 계산 후 연속 페이지 재사용.
  → O(pages × cell) → O(cell + pages).
- 무효화: 문서 단위 캐시(렌더 1회 라이프사이클), inner_width 변동 시 별도 키.
- 중첩표 배치 비용까지 캐시할지(레이아웃 서브트리 재사용) Stage 2 에서 범위 확정.
- **불변 보장**: render-tree 좌표 before/after 완전 동일(순수 성능 최적화) + 시각 회귀 게이트.

## 5. 리스크

- 캐시 키 오류 → 다른 셀/폭 결과 오재사용(시각 회귀). render-tree 동일성으로 엄격 검증.
- 렌더 캐시는 광역 → 표 다수 문서 회귀 주의. `cargo test --lib` + 표/중첩표 통합 테스트.

→ Stage 2(메모이제이션 설계) 진행 승인 요청.
