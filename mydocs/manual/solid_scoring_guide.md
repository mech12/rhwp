# SOLID 리팩토링 관점 + 100점 채점 기준 가이드

> 대상: 컬래버레이터·컨트리뷰터. 리팩토링 거버넌스 2원칙(SOLID·복잡도) 중 **SOLID 축의
> 공유 기준**이다. #1883 계획 리뷰와 Phase 2 실행, 그리고 향후 6차 코드 리뷰의 채점 기준.
> 복잡도 축의 정량 기준은 코드 품질 대시보드([dashboard.md](dashboard.md)) 참조.

## 1. 왜 SOLID 인가 — rhwp 의 역사가 증명

이 프로젝트의 품질 도약은 SOLID 위반의 해소로 만들어졌다 (r-code-review 1~5차 기록):

- 2차 리뷰 5.4점의 최대 원인 = `wasm_api.rs` 2.4만 줄 **God Object** (S 위반)
- Task 149 에서 `document_core/` 분리 + CQRS 도입 → 3차 리뷰 **9.0점** (S 3→9, D 5→9)
- 반대로 4차 리뷰(8.9)의 하락 요인 = 대형 메서드 재축적 (S 재위반)

즉 SOLID 점수는 추상적 이론이 아니라 **"국소 수정의 회귀 범위"를 예측하는 실용 지표**다.
#1582 감사가 지적한 문제(수정 하나에 전 레이어 암묵 의존 확인 필요)가 정확히 SOLID 위반의
비용이다.

## 2. 5원칙 — rhwp 실사례로 읽기

각 원칙마다: 정의 → rhwp 위반 사례(실제) → 준수 사례(실제) → 리뷰 시 체크 질문.

### S — 단일 책임 (Single Responsibility)
한 모듈/타입/함수는 한 가지 변경 이유만 가진다.
- **위반 사례**: 구 `wasm_api.rs`(로딩+브릿지+직렬화+편집 전부) / 현 `Document` 에 canonical
  IR + package 보존(`extra_streams`) + provenance(`is_hwp3_variant`) 혼재(#1582) /
  `typeset_section_with_variant` 7,059줄(분할·앵커·flow·소스분기 집적)
- **준수 사례**: `document_core/commands/` vs `queries/` (CQRS), `paint`(backend 중립 visual IR)
- **체크**: "이 파일(함수)을 고치는 이유가 두 가지 이상인가?" / "PR 하나가 서로 다른 도메인
  축 두 개를 동시에 건드리는가?"

### O — 개방-폐쇄 (Open-Closed)
확장에 열려 있고 수정에 닫혀 있다 — 새 케이스 추가가 기존 코드 수정 없이 가능해야 한다.
- **위반 사례**: 렌더/레이아웃 경로에 소스-포맷 분기(`is_hwp3_variant` 류) 산재 — 새 포맷
  변형마다 각 지점을 손대야 함 (#1582 → Phase 1 `LayoutCompatibilityProfile` 로 policy 화)
- **준수 사례**: `Renderer` trait — SVG/Canvas/Skia/CanvasKit 백엔드가 코어 수정 없이 추가됨
- **체크**: "비슷한 다음 요구가 오면 기존 함수를 또 고쳐야 하는가, 새 구현을 꽂으면 되는가?"

### L — 리스코프 치환 (Liskov Substitution)
같은 계약(trait)의 구현들은 서로 바꿔 써도 의미가 유지된다.
- **위반 신호**: 특정 백엔드에서만 다른 의미로 동작하는 trait 메서드, "이 구현에서는 이 인자를
  무시함" 류의 예외
- **준수 사례**: 렌더러 백엔드 간 `RenderNode` 계약 일관성 (render-diff 가 이를 게이트로 검증)
- **체크**: "구현 A 를 B 로 갈아끼우면 호출부 가정이 깨지는가?"

### I — 인터페이스 분리 (Interface Segregation)
사용자는 쓰지 않는 것에 의존하지 않는다 — 표면은 목적별로 얇게.
- **위반 사례**: `DocumentCore` pub fn 277개 — 문서 상태·layout·undo·validation·view 가 한
  표면(#1582: session aggregate). 렌더만 쓰는 쪽도 편집 표면 전체에 노출됨
- **준수 사례**: Task 149 이후의 얇은 WASM 바인딩(HwpDocument = thin wrapper)
- **체크**: "이 호출자가 실제로 쓰는 메서드가 표면의 몇 %인가?"

### D — 의존성 역전 (Dependency Inversion)
고수준 정책은 저수준 세부에 의존하지 않는다 — 코어는 어댑터를 모른다.
- **위반 신호**: 코어(model/document_core)가 특정 출력 포맷·플랫폼(브라우저/WASM) 세부를 참조
- **준수 사례**: `DocumentCore` 가 WASM 을 모름 → CLI/서버/FFI 재사용 가능(3차 리뷰 D 9/10 근거)
- **체크**: "이 의존의 화살표가 안쪽(코어)을 향하는가, 바깥(어댑터)을 향하는가?"

## 3. 100점 채점 기준

**5원칙 × 각 20점 = 100점.** 원칙별 앵커:

| 점수 | 앵커 (해당 원칙 기준) |
|---|---|
| **20** | 위반 사례 없음 + **구조가 원칙을 강제** (타입/모듈 경계·trait 계약·게이트로 위반이 컴파일/CI 에서 걸림) |
| **16** | 경미한 국소 위반 몇 곳, 신규 코드는 일관되게 준수, 위반이 문서화·추적됨 |
| **12** | 핫스팟 여러 곳에 위반 누적 — 국소 수정에 회귀 위험이 생기기 시작 |
| **8** | 광범위 위반 — 대형 God Object/거대 함수급, 수정 시 전 레이어 확인 필요 |
| **4 이하** | 원칙이 사실상 부재 |

- 중간 점수(예: 14, 18)는 앵커 사이 보간으로 부여한다.
- **정량 신호와 결합**: S/I 는 대시보드 지표(1,200줄 초과 파일 수, CC>25 함수 수)가 하한
  근거가 된다 — 예: CC>25 가 수십 개면 S 는 16 을 넘기 어렵다. O/L/D 는 구조 검토(분기 산재,
  trait 계약, 의존 방향)로 채점한다.
- **과거 점수 환산**: r-code-review 1~5차는 원칙별 /10 표기 — ×2 하면 본 기준과 연속된다
  (예: 3차 S 9/10 = 18/20, 총점 9.0/10 ≈ 90/100). 4차 8.9 ≈ 89.

## 4. 언제 어떻게 채점하나

- **정식 채점**: 코드 리뷰 사이클(6차 예정)에서 5원칙 전부 — 근거(파일:함수, 대시보드 수치)를
  반드시 병기한다. 점수만 있는 채점은 무효.
- **PR 리뷰 미니 채점**: 전 원칙 채점이 아니라 **해당 PR 이 건드리는 원칙만** 짚는다 —
  예: "이 PR 은 단일 함수를 300줄 키움(S 하락 방향)" / "분기 추가 대신 profile 조회로 처리(O 준수)".
- 채점은 비난이 아니라 **회귀 위험의 조기 신호 공유**다 — 낮은 점수 지점 = 다음 리팩토링 후보.
