## Critic

- [Blocker] `REQ-HML-EQ-001/004`은 모델에 열거된 일부 속성만 담고 나머지 `raw_ctrl_data`/`unknown`은 기본값으로 버리는데, `REQ-HML-EQ-006/007`은 보존 불가능한 미지원 의미를 감지해 export를 차단한다고 한다. 읽을 때 버린 속성은 preflight가 감지할 수 없으므로 조용한 데이터 손실 경로다. 알 수 없는 EQUATION 속성/자식의 raw 보존 모델과 blocker 판정을 명시하거나, 지원 가능한 입력 스키마를 먼저 증거로 폐쇄해야 한다.
- [Blocker] A1~A3와 R4~R5에서 미검증이라고 인정한 `8 UTF-16 units`, `BaseUnit -> font_size`, `TextColor` endian, zero-size `width/height`를 `REQ-HML-EQ-001/002/006`은 이미 구현 계약으로 고정했다. 이는 근거 없는 핵심 포맷 사실이며 offset·렌더링·색상 손상을 만들 수 있다. 구현 전에 fixture/한컴 재열기/기존 parser 대조로 각 매핑을 확정하는 validation gate와 실패 시 범위 축소 규칙이 필요하다.
- [Blocker] `getHmlSaveState`가 SHOULD이자 Open Decision인 상태는 UC-04, M8, NFR-07의 “부모가 blocker path/code를 표시” 계약을 충족하지 못한다. 현재 필수 `exportHml` 실패 예시는 `RPC_ERROR` 문자열뿐이고 부모가 이를 파싱하지 말라고 했으므로, 선택 API를 제외하면 구조화 blocker를 전달할 필수 경로가 없다. 같은 PR의 MUST로 확정하거나 `exportHml` 오류 envelope 자체에 typed blockers를 넣어야 한다.
- [Blocker] embed wire contract가 구현 가능한 수준으로 닫히지 않았다. JSON 예시의 `result: "<transferable Uint8Array>"`는 실제 structured-clone shape, transfer list에 넣는 `ArrayBuffer`, sender buffer detach/ownership, zero-copy 여부를 정의하지 않으며 T-RPC-02의 “copied transferable buffer” 표현도 서로 충돌한다. 기존 runtime 계약을 정확히 인용하고 success envelope와 transfer semantics를 고정해야 M7과 하위 호환을 검증할 수 있다.
- [Blocker] Detailed Test Plan이 Requirements를 추적하지 못한다. `BaseLine/BaseUnit/TextColor/Version/Font`, CDATA/entity, unknown equation semantics blocker, invalid XML character, offset mismatch, blocker 전수 수집, `hml-export` capability 협상에는 독립 RED/acceptance case가 없다. 현재 표만 green이어도 주요 요구사항과 손실 차단 계약은 미검증이므로 requirement-to-test 매트릭스와 각 expected code/path가 필요하다.
- [Blocker] 완료 기준이 repo 밖 절대경로의 ExamBank fixture에 의존하면서 fixture 복제와 사용 허가는 Open Decision/Dependency로 남아 있다. 개발자가 “이 문서만으로 RED fixture를 고정”한다는 Completion bar와 재현 가능한 CI가 성립하지 않는다. 최소화·익명화한 repo-owned fixture, 출처/허가, checksum을 이번 범위의 선행조건으로 확정해야 한다.
- [Gap] PRD 8개 섹션은 형식상 모두 있으나 Rollout + measurement는 stage checklist와 수동 canary뿐이다. NFR-05의 “유의미한 병목”에는 baseline/threshold가 없고, 배포 후 equation import/export 실패·blocker율·회귀를 누가 어떤 기간 관찰해 rollback할지도 없다. 측정하지 않기로 한 성능은 acceptance에서 제거하거나 기준을 정하고, canary owner·관찰창·rollback trigger를 명시해야 한다.
